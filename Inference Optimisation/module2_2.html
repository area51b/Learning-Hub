KV Cache - What It Is and Why It Matters

The single most important concept in inference serving. It determines how many users you can serve, at what speed, and at what cost.

You have encountered the KV Cache in every module so far - it was introduced in Module 1.3, grew in Module 1.4, became the Decode bottleneck in Module 1.7, and appeared as the memory manager's job in Module 2.1. Now it gets its own module, because it deserves one.

The KV Cache is the reason inference is hard to scale. It is also the reason most of the major inference optimisations - Paged Attention, Flash Attention, prefix caching, KV quantisation, disaggregation - exist. Understanding it deeply means you can explain why each of those techniques matters, and when to recommend them.

1 What the KV Cache Stores - and Why We Need It
▾

Recall from Module 1.3: during Attention, every token computes three vectors - a Query (Q), a Key (K), and a Value (V). The Query asks "what am I looking for?" The Key says "what do I contain?" The Value says "what do I send if selected?"

During Decode, the model generates one new token at a time. Each new token needs to run Attention against every previous token - including all the original prompt tokens. Without a cache, those K and V vectors would be recomputed from scratch every single step, for every single layer, for every token already in the sequence. This is spectacularly wasteful.

With KV Cache

2
Compute K,V only for new token 10
3
Write token 10's K,V to cache
4
Run Attention, generate next token
Only compute what is new. Memory read instead of recomputation. Scales cleanly.
What exactly is stored: For each token, at each layer, for each attention head - one Key vector and one Value vector. That is it. The Query vectors are not stored - they are computed fresh for the current token only. The cache grows by exactly one K,V pair per layer per head every decode step.

3 How KV Cache Competes for GPU Memory
▾

GPU memory (HBM) is shared between three competing consumers. Understanding this three-way split is essential for any conversation about inference capacity planning:

GPU memory split - A100 80GB running Llama 3 70B FP16
Model weights (~140GB - needs 2 GPUs)
KV Cache (dynamic - grows with requests)
Activations + overhead (relatively small)
Single A100 80GB - after model weights loaded (using 2-GPU tensor parallel, each GPU holds ~70GB weights):
70GB weights | 8GB KV | 2GB overhead
0 GB80 GB
With only ~8GB left for KV Cache per GPU, a 70B model can only serve a handful of concurrent requests before memory is exhausted. This is why techniques like Paged Attention (which manages KV Cache memory efficiently) and KV quantisation (which reduces KV Cache size) are critical for production workloads - and why model selection (70B vs 13B) has such a large impact on concurrency.
The concurrency math: If 8GB of KV Cache space is available and each concurrent request at 4k tokens on a 70B GQA model uses roughly 2GB of KV Cache, you can serve 4 concurrent requests per GPU pair. Add KV INT8 quantisation and that doubles to 8. Switch to a 13B model and the KV Cache shrinks dramatically - suddenly you can serve 30–40 concurrent users. Model and precision choice drive concurrency more than almost anything else.
4 What Happens When the Cache Fills Up
▾

When GPU memory runs out of space for new KV Cache blocks, the inference server has to make a choice. Different systems handle this differently - and the choice has real consequences for latency and throughput:

Q
Queue the new request
The incoming request waits until existing requests finish and free their KV Cache blocks. Simple and safe - no quality impact. The cost is increased TTFT for the waiting request.
Trade-off: Higher TTFT under load. Can queue up significantly during traffic spikes.
E
Evict and recompute (swap to CPU)
The KV Cache for a low-priority or paused request is swapped out to CPU RAM or disk. When that request resumes, its KV Cache is swapped back in and Prefill re-runs for the evicted tokens.
Trade-off: High latency penalty for evicted requests. CPU-GPU memory transfer is slow.
P
Paged Attention - fine-grained block allocation
Instead of allocating one large contiguous block per request, KV Cache is divided into small fixed-size pages. Pages from different requests are interleaved in GPU memory. This eliminates memory fragmentation - the main cause of wasted KV Cache space in naive implementations.
Result: Near-zero fragmentation, higher concurrency, the same GPU serves significantly more requests. This is the core innovation in vLLM. You will learn it in detail in Module 3.2.
Why fragmentation matters: In a naive system, each request's KV Cache is pre-allocated as one large contiguous block - sized for the maximum possible output length. A request that generates 50 tokens wastes the space reserved for 500. With many concurrent requests, this internal fragmentation can waste 60–80% of available KV Cache memory. Paged Attention treats GPU memory like an OS treats RAM - allocating pages on demand, sharing pages across requests where possible.
5 Prefix Caching - Reusing KV Cache Across Requests
▾

Many real-world applications send the same prefix to the model on every request. A customer service chatbot always prepends a 2,000-token system prompt. A RAG application always includes the same retrieved context. A multi-turn conversation always includes previous turns.

Prefix caching (also called prompt caching or KV Cache reuse) stores the KV Cache for shared prefixes and reuses it across multiple requests - avoiding the Prefill compute cost entirely for those tokens.

Without prefix caching: each request pays full Prefill cost for the entire prompt. With prefix caching: only the unique portion of each request goes through Prefill. TTFT drops dramatically for prefix-heavy workloads.
Real-world impact: For a RAG application with a 4,000-token system prompt + retrieved context sent on every request, prefix caching can reduce Prefill compute by 80–95% and cut TTFT from 800ms to under 100ms. Anthropic's API offers prompt caching as a first-class feature. vLLM implements it automatically. It is one of the highest-impact, lowest-effort optimisations for prefix-heavy workloads.
6 4 Key Ideas to Remember
▾
01
The KV Cache stores K and V vectors - not Q - to avoid redundant compute during Decode
Every decode step only computes Q for the new token. K and V for all previous tokens are read from cache. Without it, compute cost per token grows linearly with sequence length. With it, only memory bandwidth grows.
02
KV Cache size = tokens × KV heads × layers × head_dim × bytes - and it compounds fast
All four factors multiply together. A 70B model with 8 KV heads, 80 layers, at 128-dim heads, FP16 uses ~10MB per 1,000 tokens. A 32k context costs ~320MB per request - which is why high concurrency requires careful memory management.
03
KV Cache competes with model weights for GPU memory - and concurrency is the casualty
After model weights are loaded, whatever GPU memory remains is split between active request KV Caches. Fewer remaining bytes means fewer concurrent requests. Quantisation, GQA, and Paged Attention all exist to maximise the concurrent requests that fit in that remaining space.
04
Prefix caching reuses KV Cache across requests - eliminating Prefill for shared content
When multiple requests share a common prefix (system prompt, RAG context, few-shot examples), the KV Cache for that prefix can be computed once and reused. This is one of the highest-ROI optimisations for applications with consistent prompt structure.
MODULE 2.2 - CHEAT SHEET
KV Cache
Stores Key and Value vectors for all tokens seen so far. Avoids recomputing them on every decode step. Lives in GPU HBM.
KV Cache Size Formula
Tokens × KV heads × layers × head_dim × bytes_per_value × 2 (K and V). Grows with every token generated.
Memory Fragmentation
Wasted GPU memory from pre-allocating maximum-length blocks per request. Can waste 60–80% of KV Cache space in naive systems.
Paged Attention
Manages KV Cache in small fixed pages rather than large contiguous blocks. Eliminates fragmentation. Core innovation in vLLM. Covered in Module 3.2.
Prefix Caching
Stores and reuses KV Cache for shared prompt prefixes across requests. Eliminates Prefill cost for repeated content. High-ROI for RAG and system-prompt-heavy apps.
KV Quantisation
Reducing KV Cache precision from FP16 to INT8 or INT4. Halves or quarters memory use per request. Enables higher concurrency at cost of slight accuracy trade-off.
Concurrency Limit
Maximum simultaneous requests a GPU can serve. Determined by: (GPU memory − model weights) ÷ KV Cache per request.
KV Cache Eviction
When memory is full: queue new requests, swap cache to CPU RAM, or drop and recompute. All options have latency costs - Paged Attention minimises the need for eviction.
QUICK CHECK
A customer runs a legal research assistant. Every request includes the same 6,000-token legal knowledge base as a system prompt, followed by a short user question (~50 tokens). They are seeing high TTFT (over 2 seconds) even when the server is lightly loaded. What is the most impactful single optimisation, and why?
A. Switch to a smaller model - the 6,000-token prompt is too long for most models to handle efficiently
B. Enable prefix caching - the 6,000-token system prompt is identical across all requests, so its KV Cache can be computed once and reused. Each request then only needs to prefill ~50 tokens instead of 6,050, reducing TTFT from ~2 seconds to under 100ms
C. Add more GPUs - the TTFT issue is caused by insufficient compute capacity
D. Apply KV Cache quantisation - reducing KV Cache precision will speed up Prefill
