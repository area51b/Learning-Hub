Throughput vs Latency

The central trade-off in inference serving - what each means, why they conflict, and how to advise customers on the right balance for their workload.

~10 min read Concept level

Every inference system lives somewhere on a spectrum between two extremes: respond to each request as fast as possible, or process as many requests as cheaply as possible. These two goals are in direct tension - and every configuration decision in inference serving is, at its core, a choice about where to sit on that spectrum.

This module gives you the mental model, the vocabulary, and the decision framework to have this conversation confidently with customers - from engineers asking about max_batch_size to CFOs asking why their AI bill is high.

1 Latency and Throughput - Precise Definitions
▾

These terms are used loosely in many conversations. Using them precisely signals technical credibility.

Latency
Time for one request
How long it takes to get a response for a single request. Usually decomposed into TTFT (time to first token) and TPOT (time per output token).
Measured: milliseconds per request, or ms per token
Felt by: the individual user waiting for a response
Driven by: model size, KV Cache reads, GPU speed
Degraded by: large batch sizes, full queues, long contexts
Throughput
Requests processed per unit time
How many tokens or requests the system processes per second across all concurrent users. Determines cost efficiency - more throughput per GPU means lower cost per token.
Measured: tokens per second, requests per second
Felt by: the business paying the infrastructure bill
Driven by: batch size, GPU utilisation, model efficiency
Degraded by: small batches, idle GPU slots, fragmented KV Cache
The precise relationship: Latency and throughput are not always in conflict - up to a point, adding more requests to a batch improves throughput with minimal latency impact. But past a certain batch size, each new request added increases queue wait time and TTFT without proportional throughput gains. Finding that inflection point for a specific workload is the core task of inference capacity planning.
2 The Latency-Throughput Curve - Interactive
▾

As load increases on an inference server, throughput rises - until it hits a ceiling set by GPU compute and memory capacity. Latency starts low but climbs sharply as the system saturates. The sweet spot for most production workloads is just below the saturation point.

Why the curve goes vertical: As the system approaches maximum GPU capacity, new requests queue longer and longer. TTFT climbs steeply - not because the GPU slowed down, but because requests are waiting behind a full pipeline. Adding one more request to an already-saturated system can disproportionately increase latency for everyone. This is the queueing theory effect: utilisation above ~80% causes latency to rise non-linearly.
3 Workload Profiles - Where Each Type Sits on the Curve
▾

Different applications have fundamentally different positions on the throughput-latency curve. Recognising the workload type quickly is the first step to any inference architecture conversation.

CH
Interactive Chat / Copilot
Priority: TTFT < 500ms, TPOT < 30ms
Batch size: Small (2–16)
Queue tolerance: Near zero
Cost sensitivity: Medium - users pay per message
Curve position: Left of sweet spot - intentionally under-saturated
Examples: Claude.ai, GitHub Copilot, ChatGPT
API
Developer API (mixed workloads)
Priority: Balanced - SLA on P95 latency
Batch size: Medium (16–64)
Queue tolerance: Low - seconds acceptable
Cost sensitivity: High - charged per token
Curve position: At the sweet spot
Examples: Anthropic API, OpenAI API
RAG
RAG / Agentic Pipelines
Priority: Moderate TTFT, high quality
Batch size: Medium (8–32)
Queue tolerance: Seconds - user is reading retrieved results
Cost sensitivity: High - many LLM calls per pipeline
Curve position: Near sweet spot, slight throughput bias
Examples: Document Q&A, code review agents
BCH
Batch / Offline Processing
Priority: Maximum throughput, minimum cost
Batch size: Large (64–512)
Queue tolerance: Minutes to hours
Cost sensitivity: Very high - millions of documents
Curve position: Right of sweet spot - near saturation
Examples: Document classification, data extraction, evals
The mixed-workload challenge: Most production systems serve multiple workload types simultaneously - interactive users and batch jobs on the same cluster. The common solution is priority queuing: interactive requests jump the queue and get served before batch jobs, keeping TTFT low for users while batch jobs fill remaining capacity. vLLM and TensorRT-LLM both support request priority levels for exactly this pattern.
4 How to Measure Performance - Percentiles and SLAs
▾

Averages lie. A system with a 200ms average TTFT can still have 10% of users waiting over 2 seconds - and those users leave. Production inference systems are measured with percentiles, and SLAs (Service Level Agreements) are written against them.

Tail latency is where production systems fail users: The P99.9 is typically caused by infrequent but severe events - KV Cache full causing an eviction, a long request getting inserted into a batch, a garbage collection pause, or a thermal throttle on the GPU. Monitoring P99 and P99.9 separately from the mean is non-negotiable in production. A system that looks healthy on average can be silently failing its most demanding users.
5 The Levers - What Actually Moves the Needle
▾

When a customer needs to improve their throughput-latency position, there are a small number of high-leverage interventions. Understanding what each lever moves - and what it costs - is the core of inference architecture advice:

LeverLatencyThroughputNotes
Continuous batching Neutral +++ Free - config change only. Most impactful single step from static batching.
Smaller model +++ +++ Reduces both. Quality trade-off - must validate before deploying.
Quantisation (INT8) + ++ Reduces model memory → more KV Cache space → higher concurrency.
Prefix caching +++ ++ Reduces TTFT dramatically for prefix-heavy workloads. Near-zero cost.
More GPUs + +++ Linear throughput scaling. High cost. Right answer only after software optimisations exhausted.
Increase batch size - ++ Improves throughput but increases TTFT. Choose based on workload type.
Disaggregation ++ ++ Separates Prefill/Decode onto dedicated hardware. Complex but powerful at scale.
The right sequence for a customer struggling with performance: Start with continuous batching (free, immediate). Then prefix caching if workload has repeated prefixes. Then quantisation to increase concurrency. Then model selection review. Only after those are exhausted consider more hardware. Adding GPUs before optimising software is the most expensive mistake in inference architecture.
6 4 Key Ideas to Remember
▾
01
Latency is felt by users; throughput is felt by the finance team
Latency determines user experience - will they wait, or will they leave? Throughput determines cost efficiency - how many users can you serve per dollar of GPU spend? Both matter, but they speak to different stakeholders and are optimised with different levers.
02
The latency-throughput curve has a sweet spot - and a cliff
Up to roughly 70–80% GPU utilisation, throughput rises with minimal latency impact. Past that point, queue depth grows and latency rises non-linearly. Operating near saturation is efficient for batch workloads but disastrous for interactive ones.
03
Measure P95 and P99 - never just the mean
Average latency hides tail events. A system with 210ms average TTFT and 2,400ms P99.9 is failing its highest-traffic moments. SLAs should be written against P95 or P99. Monitoring only the mean gives false confidence.
04
Optimise software before buying hardware
The most common and expensive mistake in inference architecture is adding GPUs before exhausting software optimisations. Continuous batching, prefix caching, and quantisation together can often double or triple effective throughput with zero additional hardware cost.
MODULE 2.4 - CHEAT SHEET
Latency
Time for one request. Decomposed into TTFT and TPOT. Felt by the individual user. Degraded by large batches and full queues.
Throughput
Tokens or requests processed per second across all users. Determines cost per token. Improved by larger batches and higher GPU utilisation.
Sweet Spot
The GPU utilisation level (typically 70–80%) where throughput is near-maximum and latency is still acceptable. The target for balanced workloads.
Saturation
When the system is near or at maximum GPU capacity. Throughput plateaus. Latency rises steeply and non-linearly. Queue depth grows.
P95 / P99 Latency
The latency experienced by the 95th or 99th percentile of requests. The correct metric for SLAs. Much more informative than the mean.
Tail Latency
P99.9+ latency - the worst-case outliers. Usually caused by KV Cache evictions, batch queue spikes, or hardware events. The source of user complaints.
Priority Queuing
Serving interactive requests before batch jobs on the same cluster. Keeps TTFT low for users while batch fills remaining capacity.
Optimisation Order
Continuous batching → prefix caching → quantisation → model selection → hardware. Never buy GPUs before exhausting software options.
QUICK CHECK - PART 2 FINAL
A customer's CTO says: "Our P50 latency looks fine but our users are complaining. We're thinking of doubling our GPU fleet to fix it." What is your response, and what would you investigate first?
A. Doubling GPUs is the right call - more hardware always reduces latency
B. P50 looking fine while users complain is a tail latency problem. Check P95 and P99 - they are almost certainly high. Before buying more GPUs, investigate: (1) whether continuous batching is enabled, (2) KV Cache eviction events causing latency spikes, (3) whether a small number of long requests are creating queue delays for short ones. Hardware is the last resort, not the first.
C. The model is too large - switch to a smaller model to reduce per-token latency
D. The complaints are probably about response quality, not speed - this is a model fine-tuning issue
