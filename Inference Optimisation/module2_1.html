What Is Inference?

Training built the model. Inference is everything that happens after - serving it to users at speed, scale, and cost that actually works in production.

~10 min read Concept level

You have spent Part 1 learning how a Transformer works internally - tokens, embeddings, attention, FFN, layers, Prefill, Decode. All of that is the model. Now we zoom out to the system that runs the model.

Inference is the process of taking a trained model and using it to generate responses to real user requests. It is where the rubber meets the road - and where the vast majority of real-world AI cost, latency, and engineering challenge lives. Understanding inference as a system is what separates an SA who can explain architecture from one who can advise on production deployments.

1 Training vs Inference - Two Completely Different Problems
▾

Most AI coverage focuses on training - the phase where the model learns from data. But training happens once. Inference happens millions of times a day. They are fundamentally different problems:

Training
Building the model
Runs once (or a few times during development)
Optimises model weights from billions of examples
Takes days to months on large GPU clusters
Cost measured in millions of dollars for large models
Throughput matters most - process as much data as possible
Result: a frozen set of model weights saved to disk
Inference
Using the model
Runs continuously - every user request is an inference call
Applies fixed weights to generate a response
Must respond in milliseconds to seconds
Cost measured in dollars per million tokens served
Latency and cost per request matter most
Result: a response streamed back to the user
Why this distinction matters for customer conversations: When a customer says "our AI is expensive", they almost always mean inference cost - not training cost. When they say "our AI is slow", they mean inference latency. Training is a one-time capital expense. Inference is the ongoing operational cost that scales with usage. Every optimisation technique in Part 3 reduces inference cost or latency - not training cost.
2 The Inference Serving Stack
▾

Inference is not just "run the model." A production inference system has multiple layers, each doing a specific job. Understanding this stack is essential for talking to customers about where bottlenecks occur and which tools solve which problems.

Where vLLM and Triton fit: vLLM is primarily an inference server and engine - it handles both the KV Cache management layer and the execution layer. Triton Inference Server sits above vLLM and TensorRT-LLM as the orchestration layer - it handles request routing, batching coordination, and multi-model serving. TensorRT-LLM is a pure execution engine - optimised CUDA kernels that run the model as fast as possible on NVIDIA hardware. Knowing which layer each tool operates in helps you answer "which tool do I need?" questions clearly.
3 A Request's Journey Through the System
▾

From the moment a user hits send to the moment the response completes, here is every step that happens inside a production inference system:

1
Request arrives at API Gateway
The user's prompt hits the load balancer. Authentication is checked, rate limits applied, and the request is forwarded to an available inference server instance.
Wall time: <5ms
2
Tokenisation
The raw text prompt is converted into token IDs using the model's tokeniser. A 500-word prompt becomes approximately 650 tokens. The token count determines how much GPU memory the KV Cache will need.
Wall time: <2ms
3
Queue and KV Cache allocation
The request enters the inference server's scheduling queue. The KV Cache manager checks available GPU memory and reserves blocks for this request's KV Cache. If memory is full, the request waits or an existing request's cache is evicted.
Wall time: 0–500ms (depends on queue depth)
4
Prefill phase
All input tokens are processed in one forward pass through all model layers. KV Cache is populated. Compute-intensive. TTFT clock starts ticking from the user's send - everything up to this point contributes to TTFT.
Wall time: 50–500ms for typical prompts
5
Decode phase (token by token)
The first output token is returned to the user (TTFT achieved). Each subsequent token is generated by one forward pass reading the full KV Cache. Tokens are streamed as they are produced - the user sees the response build up in real time.
Wall time: 20–100ms per token (TPOT)
6
Stop token and cleanup
The model generates a special end-of-sequence token when it is done. The KV Cache blocks for this request are freed, the GPU memory is returned to the pool, and the slot is available for the next request.
Wall time: <5ms
What determines total response time: Queue wait + Prefill + (TPOT x output tokens). For a 200-token prompt, 500-token output on a moderately loaded system, a realistic breakdown might be: 50ms queue + 120ms Prefill + (25ms x 500 tokens) = 12.67 seconds total. The Decode phase dominates for anything beyond short responses - which is why TPOT optimisation matters so much.
4 The 5 Metrics That Define Inference Performance
▾

Every customer conversation about inference performance eventually comes down to one or more of these five metrics. Knowing what each measures and what drives it makes you a much more credible technical advisor:

Inference performance metrics - know all five
TTFT
< 500ms
Time to First Token - target for chat apps
Time from sending prompt to first token appearing. Driven by queue wait + Prefill. Long prompts and overloaded servers increase TTFT.
TPOT
< 30ms
Time per Output Token - target for real-time feel
Time between successive output tokens. Driven by Decode memory bandwidth. KV Cache size and GPU memory speed are the main factors.
Throughput
tok/s
Tokens per second across all concurrent users
How many tokens the system generates per second in total across all requests. Drives cost efficiency. Higher throughput = lower cost per token.
Concurrency
req/GPU
Simultaneous requests per GPU
How many requests a single GPU can handle at the same time. Driven by available KV Cache memory. More concurrent requests = better GPU utilisation.
Cost per Token
$/1M tok
The business metric everything else feeds into
GPU cost divided by tokens served. Improving throughput, concurrency, and GPU utilisation all reduce cost per token. This is the metric CEOs and CFOs care about - the others are the levers that move it.
The fundamental tension: Optimising for low latency (TTFT, TPOT) often conflicts with optimising for high throughput and low cost. Serving each request as fast as possible means not waiting to batch it with others - which wastes GPU capacity. Batching many requests together maximises throughput but adds queue wait time. This tension is the central challenge of inference serving - and the subject of Module 2.4.
5 Online Inference vs Batch Inference
▾

Not all inference workloads are the same. The two major categories have completely different requirements and are optimised differently:

Online Inference
Batch Inference
Priority
Low latency - user is waiting
High throughput - cost efficiency
Use cases
Chat, copilots, voice assistants, search
Document processing, data extraction, evaluation pipelines
Timing
Real-time, requests arrive unpredictably
Scheduled, large volumes processed overnight or off-peak
Key metric
TTFT and TPOT
Throughput and cost per token
GPU strategy
Reserve capacity, optimise for latency, avoid over-batching
Maximise GPU utilisation, large batch sizes, spot/preemptible GPUs acceptable
Customer example: A fintech company processing 500,000 loan documents overnight is a batch workload - they care about cost per document and total job time, not individual response latency. A legal tech company with lawyers using a document chat assistant is an online workload - each lawyer is waiting, so TTFT and TPOT matter. Both customers need inference optimisation, but the right solutions are very different.
6 4 Key Ideas to Remember
▾
01
Inference is the operational cost - training is the capital cost
A model trained once costs millions. Serving it to users costs money every single request - at scale, inference cost dwarfs training cost. This is why inference optimisation has become one of the most important engineering disciplines in AI.
02
The serving stack has multiple layers - bottlenecks can be at any layer
A slow inference system might be bottlenecked at the API layer (too many requests), the scheduler (poor batching), the KV Cache manager (memory full), or the execution engine (model too large for GPU). Knowing the stack helps you diagnose which layer is the problem.
03
There are 5 key metrics - and the business cares about cost per token
TTFT, TPOT, throughput, concurrency, and cost per token. The first four are technical levers. Cost per token is the business outcome. Being able to explain how a technical optimisation moves cost per token makes you credible in both technical and business conversations.
04
Online and batch inference need fundamentally different strategies
Before recommending any optimisation, ask: is this a latency-sensitive online workload or a throughput-focused batch workload? The answer changes everything - from GPU selection to batching strategy to which software stack to use.
MODULE 2.1 - CHEAT SHEET
Inference
Using a trained model to generate responses. Happens continuously in production. The main source of ongoing AI operational cost.
Training
Building the model from data. Happens once. Results in a fixed set of weights. Not the same problem as inference.
Inference Server
Software that manages request queuing, batching, and scheduling for GPU execution. Examples: vLLM, Triton Inference Server.
Inference Engine
Software that executes the model forward pass on GPU. Examples: TensorRT-LLM, vLLM engine. Runs the actual transformer operations.
TTFT
Time to First Token. Latency from prompt submission to first output token. Driven by queue wait + Prefill time.
TPOT
Time per Output Token. Interval between successive output tokens during Decode. Driven by memory bandwidth and KV Cache size.
Online Inference
Real-time serving where users are waiting. Optimise for TTFT and TPOT. Examples: chat, copilots, voice assistants.
Batch Inference
Scheduled, high-volume processing with no live users. Optimise for throughput and cost per token. Examples: document processing, data extraction.
QUICK CHECK
A customer's engineering team reports that their chatbot has acceptable TTFT (under 1 second) but users are complaining that responses "feel slow" even after the first word appears. The CTO asks if they need a bigger model or better hardware. What is your assessment?
A. A bigger model would improve response quality and speed simultaneously - they should upgrade to a 70B model
B. The TTFT is fine so Prefill is not the issue. The complaint about feeling slow after the first word points to high TPOT - the Decode phase is the bottleneck. The fix is not a bigger model (that makes it worse) but Decode optimisation: KV Cache management, memory bandwidth improvements, or GQA if the model supports it.
C. The problem is likely network latency between the server and users - they need a CDN
D. This is a training issue - the model was not trained on enough conversational data
