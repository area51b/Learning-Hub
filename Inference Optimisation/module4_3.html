TensorRT-LLM - How It Differs from vLLM, When It Wins

NVIDIA's compiled inference engine. What it actually does under the hood, why compilation produces faster engines than runtime execution, and the exact conditions where its performance advantage justifies the operational cost.

~10 min read Concept level

TensorRT-LLM is NVIDIA's production inference engine for large language models. Where vLLM is a Python server that runs models at inference time using PyTorch, TensorRT-LLM compiles models ahead of time into GPU-specific optimised engines - fusing operations, selecting the fastest kernel for each layer on each GPU, and eliminating all Python overhead from the serving path. The result is consistently 20–40% more throughput than vLLM at the same hardware count, at the cost of a more complex build-and-deploy workflow.

This module covers what TensorRT-LLM actually does (compilation, kernel fusion, quantisation), how it differs architecturally from vLLM, when that difference matters enough to justify the investment, and how the deployment workflow looks in practice. After this module you should be able to explain TRT-LLM's value proposition accurately and know which customer situations call for it over vLLM.

1 What TensorRT-LLM Is - Compiled Engines vs Runtime Execution
▾

The core distinction between TensorRT-LLM and vLLM is compile-time optimisation versus runtime execution. vLLM loads model weights and runs inference through PyTorch, making decisions at runtime about how to execute each operation. TensorRT-LLM takes a model, analyses it, and produces a compiled TensorRT engine - a binary optimised specifically for the target GPU, with operations fused, memory pre-planned, and kernel implementations selected ahead of time.

TensorRT-LLM build-then-serve workflow
1
Model weights (HuggingFace / checkpoint)
Starting point: standard model weights in HuggingFace format or NVIDIA checkpoint format. Same weights vLLM would use.
▼
2
Quantisation (optional - Model Optimizer / AMMO)
Apply INT8, FP8, or INT4 AWQ quantisation to weights using NVIDIA's calibration tools. Produces quantised checkpoint. This step requires a calibration dataset and takes 10–60 minutes depending on model size.
▼
3
trtllm-build - compilation to TensorRT engine
Core step. TRT-LLM analyses the model graph, fuses operations (LayerNorm + attention + activation into single kernels), selects optimal kernel implementations per GPU architecture, pre-plans memory layout. Output: .engine files for each GPU. Takes 5–120 minutes for large models.
▼
4
Engine validation
Run sample prompts through the compiled engine and compare outputs to the original model. Verify quality metrics meet requirements before deploying to production.
▼
5
Triton Inference Server serves the engine
The compiled .engine file is served via Triton Inference Server (Module 4.4), which handles HTTP/gRPC endpoints, batching orchestration, and health checks. TRT-LLM handles the GPU execution; Triton handles the serving layer.
Key constraint: a compiled engine is GPU-specific. An engine built for H100 will not run on A100, and vice versa. If you change GPU type, you must recompile. Engines are also batch-size-specific - you specify the maximum batch size at compile time and the engine is optimised for that profile.
The engine is a binary artifact, not a model: A TRT-LLM engine is analogous to a compiled executable - it contains fused CUDA kernels for a specific GPU, not portable model weights. This is why TRT-LLM deployments have a CI/CD build step that vLLM deployments do not. Teams that manage TRT-LLM well treat engine compilation like a build pipeline: version-controlled build configs, automated compilation on model updates, validation gates before production promotion.
2 Kernel Fusion - Why Compiled Engines Are Faster
▾

A transformer forward pass consists of dozens of sequential operations per layer - LayerNorm, Q/K/V projection, attention, output projection, FFN expand, activation, FFN contract, residual add, and more. In an unoptimised runtime (like naive PyTorch), each of these is a separate GPU kernel call: the result is written to HBM, the next kernel reads it back, does its operation, writes again, and so on. Each HBM round-trip costs bandwidth and latency.

Kernel fusion combines multiple sequential operations into a single kernel that keeps data in SRAM (fast on-chip memory) between operations, never writing intermediate results to HBM. TensorRT automatically identifies fusible operation sequences during compilation and combines them.

Per-layer operations - unfused vs fused kernel execution
Kernel selection is also GPU-specific: For a given operation like matrix multiplication, NVIDIA provides dozens of kernel implementations optimised for different matrix shapes, batch sizes, and GPU generations. At compile time, TensorRT benchmarks candidate kernels against the actual target GPU and selects the fastest one for each layer's specific shape. vLLM uses a fixed set of kernels that must work acceptably across many GPU types. This per-GPU selection is worth an additional 10–20% on top of fusion gains.
3 Quantisation in TensorRT-LLM - Deeper Hardware Integration
▾

Both vLLM and TRT-LLM support quantisation (INT8, FP8, INT4), but TRT-LLM's implementation is more deeply integrated with the hardware. Where vLLM applies quantisation as a layer on top of PyTorch, TRT-LLM compiles quantisation directly into the engine - fused dequantise-multiply-quantise operations run in a single kernel pass without ever materialising full-precision intermediate tensors.

vLLM quantisation
Applied via HuggingFace quantised weights (AWQ, GPTQ) or runtime quantisation
PyTorch handles dequantise at runtime
Supports AWQ, GPTQ, FP8, GGUF
Setup: load pre-quantised model
Best for: quick deployment, broad model support
TRT-LLM quantisation
Applied via NVIDIA Model Optimizer (AMMO) with calibration dataset
Compiled into engine - no runtime overhead
Supports INT8 (SmoothQuant), FP8, INT4-AWQ, INT4-GPTQ
Setup: calibrate → quantise → compile
Best for: maximum throughput at fixed model
FP8 on H100 - TRT-LLM's strongest advantage: The H100's Transformer Engine can execute FP8 matrix multiplications natively in hardware. TRT-LLM's compiled FP8 engine uses these hardware instructions directly with no abstraction overhead. In benchmarks, TRT-LLM FP8 on H100 consistently outperforms vLLM FP8 on H100 by 25–45% in tokens/second - because the TRT-LLM engine has been compiled to use H100-specific FP8 Tensor Core instructions explicitly, while vLLM goes through a PyTorch abstraction layer that adds overhead.

The calibration requirement is a genuine operational consideration. NVIDIA Model Optimizer needs a representative sample of the model's input distribution (typically 512–1024 examples from your actual use case) to determine optimal quantisation scales. Using a generic calibration dataset for a domain-specific model (e.g. legal documents) produces worse quality than calibrating on domain-appropriate text. Teams deploying TRT-LLM in production should invest in a domain-relevant calibration set and re-run calibration when the use-case distribution shifts.

6 What a TRT-LLM Deployment Actually Looks Like
▾

Many customers hear "TensorRT-LLM" and picture a simple pip install. The actual deployment workflow is closer to a software build pipeline. Knowing this lets you set expectations correctly in customer conversations and scope the implementation effort accurately.

TRT-LLM production deployment workflow
1
Prepare environment
Pull NVIDIA TRT-LLM Docker container (NGC registry). Verify CUDA driver version compatibility. Allocate GPU node(s) for compilation - this is compute-intensive.
docker pull nvcr.io/nvidia/tensorrt-llm:latest
2
Download and convert model weights
Download from HuggingFace. Run TRT-LLM's convert_checkpoint.py to transform weights into TRT-LLM's intermediate checkpoint format. This is lossless - no quantisation yet.
python convert_checkpoint.py --model-dir ./llama3-70b --output-dir ./trtllm-ckpt --tp 4
3
Quantise (if required)
Run NVIDIA Model Optimizer with a calibration dataset representative of your production traffic. Produces a quantised checkpoint. Skip this step if serving at BF16/FP16. Duration: 15–60 minutes for 70B.
python quantize.py --model-dir ./trtllm-ckpt --qformat fp8 --calib-dataset ./calib-data.json
4
Compile engine with trtllm-build
The core compilation step. Specify target GPU, max batch size, max sequence length, parallelism config. Duration: 10 minutes (small models) to 2 hours (405B, multi-GPU). Output: .engine files, one per GPU.
trtllm-build --checkpoint-dir ./trtllm-ckpt --output-dir ./engines --tp 4 --max-batch-size 64 --max-seq-len 8192
5
Validate engine quality
Run sample prompts. Compare outputs to reference model. Run automated quality benchmarks (perplexity, task accuracy). Gate on quality before promoting to production.
python run.py --engine-dir ./engines --max-output-len 512
6
Serve via Triton Inference Server
Configure Triton model repository pointing to the compiled engine. Deploy Triton container. Triton handles HTTP/gRPC endpoints, batching policy, health checks. (Module 4.4 covers this in detail.)
tritonserver --model-repository ./triton-model-repo
Engine recompilation triggers: A team running TRT-LLM in production must recompile whenever: (1) the model weights change (fine-tune update), (2) the target GPU type changes, (3) the required max batch size or sequence length changes, (4) TRT-LLM version is updated (engine format is versioned). This means build automation is not optional - it is a prerequisite for sustainable TRT-LLM operations. Teams without CI/CD infrastructure for engine builds often revert to vLLM after a painful manual recompilation cycle.
7 4 Key Ideas to Remember
▾
01
Compilation eliminates Python overhead and fuses kernels - that is where the 20–40% throughput gain comes from
The compiled TRT engine runs pure C++/CUDA with no Python in the hot path. Kernel fusion keeps intermediate results in SRAM instead of round-tripping through HBM. GPU-specific kernel selection squeezes the last performance from the target hardware. These three factors compound to produce consistent throughput advantages over runtime execution.
02
The engine is GPU-specific and must be recompiled when GPU type, batch size, or model changes
A TRT-LLM engine built for H100 will not run on A100. An engine compiled for max-batch-size 64 cannot serve 128 simultaneous requests. A model fine-tune requires a new compilation run. These constraints are non-negotiable - they are inherent to how TensorRT compilation works. Teams must treat engine builds as first-class CI/CD artifacts.
03
TRT-LLM's quantisation is calibrated and hardware-fused - the throughput gains from FP8/INT8 are larger than with vLLM
NVIDIA Model Optimizer calibrates quantisation scales against actual data, and TRT-LLM compiles those scales directly into the engine. On H100, this produces FP8 inference that uses native Tensor Core instructions with no abstraction overhead - routinely 25–45% faster than vLLM FP8 on the same hardware.
04
TRT-LLM wins at: fixed model, high scale, tight SLAs, experienced team. vLLM wins everywhere else.
The TRT-LLM advantage is real but only materialises under specific conditions. Recommending it to a team that is still evaluating models, lacks MLOps infrastructure, or serves at modest scale is setting them up for operational pain without proportional benefit. The right recommendation requires asking about model stability, concurrency, SLAs, and team capability - not just GPU type.
MODULE 4.3 - CHEAT SHEET
TensorRT-LLM
NVIDIA's compiled inference engine. Build-time optimisation: kernel fusion, GPU-specific kernel selection, quantisation compiled into engine. 20–40% throughput advantage over vLLM at peak load.
Kernel fusion
Combining multiple sequential GPU operations (LayerNorm, attention, activation) into a single kernel that keeps data in SRAM. Eliminates HBM round-trips between ops. Primary source of TRT-LLM's latency advantage.
trtllm-build
The compilation command. Takes converted checkpoint + config, outputs .engine files per GPU. GPU-specific - must recompile for different GPU types or batch sizes. Duration: 10 min to 2 hours.
NVIDIA Model Optimizer (AMMO)
Quantisation tool for TRT-LLM. Calibrates INT8/FP8/INT4 scales against a representative dataset. Produces quantised checkpoint for trtllm-build. Requires domain-appropriate calibration data for best quality.
Engine constraints
Compiled engine is GPU-specific + batch-size-specific. Model update, GPU change, or batch size change = recompile. Build automation (CI/CD for engines) is mandatory for sustainable TRT-LLM ops.
In-flight batching
TRT-LLM's term for continuous batching. Functionally equivalent to vLLM's iteration-level scheduling - requests join and leave the batch at each decode step without waiting for the full batch to complete.
When TRT-LLM wins
Fixed model in production + high concurrency (>50 users) + tight TPOT SLA (<30ms) + experienced MLOps team + H100/H200 hardware. All four conditions should be present.
Serving layer
TRT-LLM does not serve HTTP on its own - it requires Triton Inference Server on top. Triton handles endpoints, batching policy, and health checks. TRT-LLM handles GPU execution. (Module 4.4.)
QUICK CHECK
A customer has been running vLLM with Llama 3 70B on a 4× H100 SXM cluster for six months. The model is stable (no fine-tuning planned), they serve 150 concurrent users at peak, their TPOT SLA is 25ms, and they have a dedicated ML platform team. They are hitting their vLLM throughput ceiling and asking what to do. What is your recommendation, and what is the most important operational requirement you need to communicate upfront?
A. Add more GPUs - scaling horizontally with vLLM is simpler than switching to TensorRT-LLM and will solve the throughput ceiling.
B. This customer is the ideal TensorRT-LLM candidate: stable model, high concurrency, tight SLA, experienced team, H100 SXM with NVLink. Recommend migrating to TRT-LLM with FP8 quantisation - expect 25–40% throughput improvement on the same 4× H100 hardware, likely resolving the ceiling without adding GPUs. The critical operational requirement to communicate upfront: the compiled engine is GPU-specific and must be rebuilt whenever the model, GPU type, max batch size, or TRT-LLM version changes. They need a CI/CD pipeline for engine builds - not a one-time compilation. If they do not have automated build infrastructure, set up that pipeline before the first production deployment or they will face a painful manual recompilation cycle on the first model update.
C. Recommend switching to a smaller, faster model - if they are hitting a throughput ceiling, the model is too large for their hardware.
D. Enable speculative decoding in vLLM - this will improve throughput at high concurrency without requiring a migration to TensorRT-LLM.
