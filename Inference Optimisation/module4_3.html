TensorRT-LLM - How It Differs from vLLM, When It Wins

NVIDIA's compiled inference engine. What it actually does under the hood, why compilation can produce faster engines than a pure runtime stack, and the conditions where its performance advantage typically justifies the additional build-and-operate complexity.

~10 min read Concept level

TensorRT-LLM is NVIDIA's production inference engine for large language models. Where vLLM is a Python server that runs models at inference time using PyTorch plus its own CUDA kernels, TensorRT-LLM compiles models ahead of time into GPU-specific optimised engines - fusing operations, selecting kernels based on the actual GPU architecture and model shapes, and eliminating Python from the hot path. In many high-throughput benchmarks this yields roughly 20-40% more throughput than a well-tuned vLLM stack on the same hardware, at the cost of a more complex build-and-deploy workflow.

This module covers what TensorRT-LLM actually does (compilation, kernel fusion, quantisation), how it differs architecturally from vLLM, when that difference is likely to matter enough to justify the investment, and how the deployment workflow looks in practice. After this module you should be able to explain TRT-LLM's value proposition accurately and know which customer situations are strong candidates for it over vLLM.

1 What TensorRT-LLM Is - Compiled Engines vs Runtime Execution
▾

The core distinction between TensorRT-LLM and vLLM is compile-time optimisation versus primarily runtime execution. vLLM loads model weights and runs inference through PyTorch plus its own CUDA kernels, with execution decisions resolved at runtime and heavily delegated to libraries like cuBLAS and cuDNN. TensorRT-LLM takes a model, analyses it, and produces a compiled TensorRT engine - a binary optimised specifically for the target GPU, with operations fused, memory pre-planned, and concrete kernel implementations selected ahead of time for the model's actual shapes.

TensorRT-LLM build-then-serve workflow
1
Model weights (HuggingFace / checkpoint)
Starting point: standard model weights in HuggingFace format or NVIDIA checkpoint format. Same weights vLLM would use.
▼
2
Quantisation (optional - Model Optimizer / AMMO)
Optionally apply INT8, FP8, or selected INT4 schemes to weights using NVIDIA's calibration tools. Produces a quantised checkpoint. This step requires a calibration dataset and typically takes 10-60 minutes depending on model size and hardware.
▼
3
trtllm-build - compilation to TensorRT engine
Core step. TRT-LLM analyses the model graph, fuses operations (for example, combining LayerNorm + attention + activation into fewer kernels), selects kernel implementations tuned for the specific GPU architecture and tensor shapes, and pre-plans memory layout. Output: .engine files for each GPU. For large models this can take 5-120 minutes.
▼
4
Engine validation
Run sample prompts through the compiled engine and compare outputs to the original model. Verify quality metrics meet requirements before deploying to production.
▼
5
Triton Inference Server serves the engine
The compiled .engine file is served via Triton Inference Server (Module 4.4), which handles HTTP/gRPC endpoints, batching orchestration, and health checks. TRT-LLM handles the GPU execution; Triton handles the serving layer.
Key constraint: a compiled engine is tightly coupled to a GPU generation/architecture and build profile. Practically, an engine built for H100 should be treated as non-portable to A100, and vice versa. If you change GPU type or significantly change engine parameters, you should plan to recompile. Engines are also batch-size-specific - you specify the maximum batch size at compile time and the engine is optimised for that profile.
The engine is a binary artifact, not a model: A TRT-LLM engine is analogous to a compiled executable - it contains fused CUDA kernels for a specific GPU profile, not portable model weights. This is why TRT-LLM deployments have a CI/CD build step that vLLM deployments do not. Teams that manage TRT-LLM well treat engine compilation like a build pipeline: version-controlled build configs, automated compilation on model or config updates, and validation gates before production promotion.
2 Kernel Fusion - Why Compiled Engines Are Faster
▾

A transformer forward pass consists of dozens of sequential operations per layer - LayerNorm, Q/K/V projection, attention, output projection, FFN expand, activation, FFN contract, residual add, and more. In an unoptimised runtime (like naive PyTorch), each of these is a separate GPU kernel call: the result is written to HBM, the next kernel reads it back, does its operation, writes again, and so on. Each HBM round-trip costs bandwidth and latency.

Kernel fusion combines multiple sequential operations into a single kernel that keeps data in SRAM (fast on-chip memory) between operations, never writing intermediate results to HBM. TensorRT automatically identifies fusible operation sequences during compilation and combines them. vLLM also uses fused kernels in several hotspots, but TRT-LLM pushes fusion further because it can specialise the engine for a specific model and GPU at build time.

Per-layer operations - unfused vs fused kernel execution
Kernel selection is also GPU and shape-specific: For a given operation like matrix multiplication, NVIDIA provides many kernel implementations optimised for different matrix shapes, batch sizes, and GPU generations. At compile time, TensorRT can benchmark or analytically choose candidate kernels against the actual target GPU and layer shape, and bake those choices into the engine. vLLM benefits from architecture-aware kernels in NVIDIA libraries and its own CUDA code, but it cannot specialise the full end-to-end graph per deployment in quite the same way. This extra level of per-GPU, per-shape tuning is typically worth another ~10-20% on top of fusion gains in high-load scenarios.
3 Quantisation in TensorRT-LLM - Deeper Hardware Integration
▾

Both vLLM and TRT-LLM support quantisation (INT8, lower-bit schemes, FP8 on appropriate hardware), but TRT-LLM's implementation is more deeply integrated with the hardware and engine build. Where vLLM often applies quantisation as a layer on top of PyTorch (using HuggingFace integrations, third-party libraries, or runtime dequantisation), TRT-LLM compiles quantisation directly into the engine - fused dequantise-multiply-quantise paths can run in a single kernel pass without ever materialising full-precision intermediate tensors.

vLLM quantisation
Applied via HuggingFace/third-party quantised weights (AWQ, GPTQ, etc.) or runtime quantisation flows
PyTorch and underlying libraries handle dequantise at runtime
Supports a range of formats depending on model and integration (for example AWQ, GPTQ; FP8 support depends on PyTorch and vLLM versions)
Setup: typically load a pre-quantised model or use a supported quantisation workflow
Best for: quick deployment and broad model coverage, where flexibility and iteration speed matter more than absolute peak tokens/sec
TRT-LLM quantisation
Applied via NVIDIA Model Optimizer (AMMO) with a calibration dataset
Compiled into the engine graph - minimal extra runtime overhead
Supports INT8 (e.g. SmoothQuant-style flows), FP8 on supported GPUs, and selected INT4 schemes (such as INT4-AWQ / INT4-GPTQ for supported architectures and configs)
Setup: calibrate → quantise → compile, with validation after
Best for: maximum throughput and efficiency when the model and hardware profile are stable
FP8 on H100 - TRT-LLM's strongest advantage in practice: The H100's Transformer Engine can execute FP8 matrix multiplications natively in hardware. TRT-LLM's compiled FP8 engines use these hardware instructions directly with very little abstraction overhead. In NVIDIA and community benchmarks, TRT-LLM FP8 on H100 often outperforms comparable vLLM FP8 stacks by roughly 25-45% in tokens/second under high-load conditions - because the TRT-LLM engine is built specifically to exploit H100 FP8 Tensor Core instructions, while vLLM typically routes through a more generic PyTorch abstraction that adds some overhead.

The calibration requirement is a genuine operational consideration. NVIDIA Model Optimizer needs a representative sample of the model's input distribution (typically 512-1024 examples from your actual use case) to determine reasonable quantisation scales. Using a generic calibration dataset for a domain-specific model (e.g. legal documents) will usually produce worse quality than calibrating on domain-appropriate text. Teams deploying TRT-LLM in production should invest in a domain-relevant calibration set and re-run calibration when the use-case distribution shifts meaningfully.

They are increasingly converging: Since around 2023, the performance and feature gap between vLLM and TRT-LLM has narrowed. vLLM has added more kernel optimisations, better batching, and improving support for lower-precision formats; TRT-LLM has expanded model coverage and simplified parts of its tooling. The decision is now less about a fundamental capability gap and more about where each tool sits on the complexity-versus-performance curve for your specific workload, hardware, and team.

6 What a TRT-LLM Deployment Actually Looks Like
▾

Many customers hear "TensorRT-LLM" and picture a simple pip install. The actual deployment workflow is closer to a software build pipeline. Knowing this lets you set expectations correctly in customer conversations and scope the implementation effort accurately.

TRT-LLM production deployment workflow
1
Prepare environment
Pull NVIDIA TRT-LLM Docker container (NGC registry). Verify CUDA driver version compatibility. Allocate GPU node(s) for compilation - this is compute-intensive, especially for large models.
docker pull nvcr.io/nvidia/tensorrt-llm:latest
2
Download and convert model weights
Download from HuggingFace or another source. Run TRT-LLM's convert_checkpoint.py to transform weights into TRT-LLM's intermediate checkpoint format. This is lossless - no quantisation yet.
python convert_checkpoint.py --model-dir ./llama3-70b --output-dir ./trtllm-ckpt --tp 4
3
Quantise (if required)
Run NVIDIA Model Optimizer with a calibration dataset representative of your production traffic. Produces a quantised checkpoint. Skip this step if serving at BF16/FP16. Duration: roughly 15-60 minutes for a 70B-class model on suitable hardware.
python quantize.py --model-dir ./trtllm-ckpt --qformat fp8 --calib-dataset ./calib-data.json
4
Compile engine with trtllm-build
The core compilation step. Specify target GPU, max batch size, max sequence length, and parallelism configuration. Duration: from around 10 minutes for small models to 1-2 hours for very large models (e.g. 70B+ with tensor/pipeline parallelism). Output: .engine files, one per GPU or partition.
trtllm-build --checkpoint-dir ./trtllm-ckpt --output-dir ./engines --tp 4 --max-batch-size 64 --max-seq-len 8192
5
Validate engine quality
Run sample prompts. Compare outputs to a reference model. Run automated quality benchmarks (for example perplexity or task-specific metrics). Gate promotion on quality; if quality is unacceptable, revisit quantisation and/or build settings.
python run.py --engine-dir ./engines --max-output-len 512
6
Serve via Triton Inference Server
Configure a Triton model repository pointing to the compiled engines. Deploy the Triton container. Triton handles HTTP/gRPC endpoints, batching policy, health checks, and metrics. TRT-LLM handles the GPU execution. (Module 4.4 covers Triton in detail.)
tritonserver --model-repository ./triton-model-repo
Engine recompilation triggers: A team running TRT-LLM in production should plan to recompile whenever: (1) model weights change (for example, new fine-tune), (2) the target GPU generation changes (for example A100 → H100), (3) the required max batch size or sequence length changes materially, or (4) the TRT-LLM/TensorRT major version is updated in a way that affects engine compatibility. This means build automation is not optional - it is a prerequisite for sustainable TRT-LLM operations. Teams without CI/CD infrastructure for engine builds often end up doing painful, manual rebuilds and may revert to vLLM after the first few cycles.
7 4 Key Ideas to Remember
▾
01
Compilation removes Python from the hot path and lets you fuse kernels - this is where most of the throughput gain comes from
The compiled TRT engine runs C++/CUDA with no Python interpreter in the hot path. Kernel fusion keeps intermediate results in SRAM instead of round-tripping through HBM. GPU- and shape-specific kernel selection squeezes more performance out of the target hardware. These factors compound to produce consistent throughput advantages over a purely runtime-based stack in high-load scenarios.
02
The engine is tightly coupled to GPU generation and build profile - expect to recompile when hardware or key parameters change
A TRT-LLM engine built for H100 should be treated as non-portable to A100, and vice versa. An engine compiled for max-batch-size 64 is not ideal for serving 128 simultaneous requests. A model fine-tune or a major TRT-LLM upgrade requires a new compilation run. These constraints are inherent to how TensorRT compilation works, not policy. Teams must treat engine builds as first-class CI/CD artifacts.
03
TRT-LLM's quantisation is calibrated and hardware-fused - FP8/INT8 gains on H100-class hardware are often bigger than with vLLM
NVIDIA Model Optimizer calibrates quantisation scales against actual data, and TRT-LLM compiles those scales directly into the engine graph. On H100, this produces FP8 inference that uses native Tensor Core instructions with very little abstraction overhead - often delivering ~25-45% more tokens/sec than comparable vLLM FP8 stacks on the same hardware when heavily loaded.
04
TRT-LLM is best for: fixed model, high scale, strict SLAs, experienced team. vLLM is often best everywhere else.
The TRT-LLM advantage is real but tends to materialise under specific conditions: the model is relatively stable, concurrency is high, TPOT SLAs are tight, and the team can own build pipelines. Recommending it to a team that is still evaluating models, lacks MLOps infrastructure, or serves at modest scale risks adding operational complexity without proportional benefit. The right recommendation starts with questions about model stability, concurrency, SLAs, hardware, and team capacity - not just GPU type.
MODULE 4.3 - CHEAT SHEET
TensorRT-LLM
NVIDIA's compiled inference engine. Build-time optimisation: kernel fusion, GPU/shape-specific kernel selection, quantisation compiled into the engine. In many high-load benchmarks, delivers roughly 20-40% higher throughput than a tuned vLLM stack on the same GPUs.
Kernel fusion
Combining multiple sequential GPU operations (LayerNorm, attention, activation, etc.) into a single or smaller number of kernels that keep data in SRAM. Eliminates unnecessary HBM round-trips between ops. A primary source of TRT-LLM's latency and throughput advantage.
trtllm-build
The compilation command. Takes a converted (and optionally quantised) checkpoint + config and outputs .engine files per GPU/profile. Engines are tied to GPU generation and build parameters. Compilation time ranges from minutes to ~1-2 hours for very large, multi-GPU models.
NVIDIA Model Optimizer (AMMO)
Quantisation tool for TRT-LLM. Calibrates INT8/FP8/selected INT4 scales against a representative dataset. Produces a quantised checkpoint for trtllm-build. Requires domain-appropriate calibration data for best quality.
Engine constraints
Compiled engine is tied to GPU generation and build profile (max batch size, max sequence length, key TRT-LLM/TensorRT version). Model update, GPU change, or major profile change = recompile. CI/CD for engine builds is effectively mandatory for sustainable TRT-LLM ops.
In-flight batching
TRT-LLM's term for continuous batching. Functionally similar to vLLM's scheduling: requests join and leave the batch at each decode step without waiting for the full batch to complete.
When TRT-LLM wins
Model is relatively fixed, concurrency is high (for example tens to hundreds of users), TPOT SLA is strict, the stack is NVIDIA-heavy (H100/H200, etc.), and the team can own build pipelines. Under those conditions, TRT-LLM's extra throughput can be worth the added complexity.
Serving layer
TRT-LLM does not expose HTTP/gRPC endpoints on its own - it runs under Triton Inference Server. Triton handles endpoints, batching policy, routing, and health checks; TRT-LLM handles GPU execution. (Module 4.4.)
QUICK CHECK
A customer has been running vLLM with Llama 3 70B on a 4× H100 SXM cluster for six months. The model is stable (no fine-tuning planned), they serve 150 concurrent users at peak, their TPOT SLA is 25ms, and they have a dedicated ML platform team. They are hitting their vLLM throughput ceiling and asking what to do. What is your recommendation, and what is the most important operational requirement you need to communicate upfront?
A. Add more GPUs - scaling horizontally with vLLM is simpler than switching to TensorRT-LLM and will solve the throughput ceiling.
B. This customer is a strong TensorRT-LLM candidate: stable model, high concurrency, tight-ish 25ms TPOT SLA, experienced team, H100 SXM with NVLink. Recommend evaluating a migration to TRT-LLM with FP8 quantisation - under these conditions, you can reasonably expect a material throughput improvement (often on the order of 25-40% in benchmarks) on the same 4× H100 hardware, which may resolve the ceiling without adding GPUs. The critical operational requirement to communicate upfront: the compiled engine is tied to GPU generation and build profile, and must be rebuilt whenever the model weights, GPU type, key engine parameters (like max batch size / sequence length), or TRT-LLM/TensorRT version change in incompatible ways. They need a CI/CD pipeline for engine builds and validation - not a one-off compilation. If they do not have automated build infrastructure, invest in that pipeline before the first production deployment to avoid painful manual rebuild cycles.
C. Recommend switching to a smaller, faster model - if they are hitting a throughput ceiling, the model is too large for their hardware.
D. Enable speculative decoding in vLLM - this will improve throughput at high concurrency without requiring a migration to TensorRT-LLM.
