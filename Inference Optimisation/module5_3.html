LLM-d - Kubernetes-Native Distributed LLM Inference

Where Dynamo is NVIDIA's fleet coordination layer, llm-d is the open-source, hardware-agnostic, Kubernetes-native answer - built by Red Hat, Google, and IBM on top of vLLM and the Kubernetes Inference Gateway.

5.4 - Dynamo vs LLM-d vs DIY

llm-d was announced in mid-2025 by Red Hat, Google, and IBM as an open-source project for high-performance distributed LLM inference on Kubernetes. It targets the same fleet-scale problems Dynamo addresses - routing blindness, disaggregation coordination, heterogeneous workloads - but from a fundamentally different starting point: instead of building a new coordination layer on top of existing tools, llm-d extends the tools and standards that Kubernetes teams already operate.

The key distinction is the deployment model. Dynamo introduces a new fleet-level process (the Dynamo router, planner, and NIXL transfer system) that teams adopt alongside their existing stack. llm-d routes through Kubernetes-native infrastructure - the Inference Gateway, Envoy proxy, and standard Kubernetes APIs - so teams already running LLM workloads on Kubernetes can adopt it incrementally without introducing a new system to operate. By February 2026, llm-d had reached v0.5, validating roughly 3,100 tokens/sec per B200 Decode GPU and order-of-magnitude TTFT reduction versus round-robin baseline.

1 The Core Philosophy - Extend Kubernetes, Don't Replace It
▾

The name "well-lit paths" is llm-d's own description of its goal: tested, benchmarked deployment patterns that teams can follow without building bespoke infrastructure. The phrase reflects the project's philosophy - rather than providing a novel system that teams must learn and operate, llm-d provides Kubernetes-native implementations of proven inference optimisations that plug into tooling teams already know.

This matters practically. A team running LLM workloads on Kubernetes already has Helm charts, Prometheus monitoring, Grafana dashboards, Kubernetes RBAC, and standard GitOps workflows. Adding Dynamo means adopting a new process with its own operational surface. Adding llm-d means installing a Helm chart that deploys the Inference Gateway scheduler and a configured vLLM deployment - things that look and behave like the Kubernetes resources the team already manages.

The foundational choice that shapes everything: llm-d chose vLLM as its primary inference engine, not TRT-LLM. This is a deliberate stance for hardware-agnosticism - vLLM supports NVIDIA A100+, AMD MI250, Google TPU v5e+, and Intel GPU Max without recompilation. TRT-LLM produces GPU-specific compiled binaries. For teams running mixed-accelerator infrastructure or non-NVIDIA hardware, llm-d is the only viable option among these two frameworks.
2 The Inference Gateway and Scheduler - Smart Routing the Kubernetes Way
▾

The Kubernetes Inference Gateway (IGW) is an official Kubernetes project - not just a llm-d component - that extends the standard Gateway API with LLM-specific concepts. llm-d builds on IGW rather than building its own routing layer, which means teams get a routing system that integrates with standard Kubernetes tooling and will be maintained upstream as the Kubernetes ecosystem evolves.

IGW and scheduler components - click to explore

The Endpoint Picker Protocol (EPP) is where llm-d's intelligence lives. Rather than reimplementing a full-featured load balancer, the EPP plugs into Envoy's extensible balancing policies as a custom filter. This design means llm-d inherits Envoy's production-grade flow control, connection management, circuit breaking, and observability - while adding the LLM-specific scoring logic on top.

Why Envoy matters here: Envoy is the data plane used by Istio, Linkerd-compatible configurations, and virtually every service mesh deployed in production Kubernetes environments. By building on Envoy, llm-d integrates naturally with existing service mesh deployments. A team already running Istio gets llm-d's inference-aware routing without replacing any networking infrastructure - they add the EPP filter to the existing Envoy configuration.
3 How the Inference Scheduler Scores Pods - Filter, Score, Select
▾

When a request arrives at the Inference Gateway, the scheduler does not pick a pod arbitrarily. It runs a two-phase process: first it filters out pods that cannot serve the request (overloaded, wrong model version, unhealthy), then it scores the remaining candidates across multiple dimensions and routes to the highest-scoring pod.

Scheduler scoring - select a request type to see how pods are evaluated

Teams can customise the scoring algorithm by implementing their own scorer plugins - the EPP defines an interface that any scorer must implement, allowing organisations to add business-specific logic (e.g., prefer pods in a specific availability zone, penalise pods with recent errors) without forking the core scheduler. The default scorers cover KV Cache-awareness, load balancing, SLA priority, and P/D disaggregation routing.

Predicted latency balancing (v0.3+): From the v0.3 release onwards, llm-d's scheduler gained a predicted latency scorer - rather than just looking at current load and cache state, it estimates the expected completion time for each pod given the request's input/output length and the pod's current queue. Benchmarks showed up to 3x P90 latency improvement in long-Prefill workloads compared to load-only balancing. This is a meaningful capability gap versus simple round-robin: a pod with low connection count but a large in-flight long-context request may have worse predicted latency than a busier pod with shorter requests.
4 Disaggregated Serving in llm-d - The Sidecar Model
▾

llm-d implements disaggregation differently from Dynamo. Rather than a dedicated fleet-level coordination process, disaggregation in llm-d is implemented via a sidecar container that runs alongside each Decode pod. The scheduler decides which Prefill pod should handle the Prefill phase and encodes that decision in an HTTP header (x-prefiller-host-port). The sidecar on the Decode pod reads this header and proxies the Prefill portion to the designated Prefill worker, then coordinates the KV Cache transfer back.

For KV Cache transfer, llm-d supports both its own vLLM KV connector interface and NIXL - so teams using NVIDIA hardware get the same high-bandwidth RDMA transfer that Dynamo uses for its inter-node handoff. The v0.4 release extended disaggregation support beyond NVIDIA to Intel XPU and Google TPU, reflecting the hardware-agnostic design principle.

The sidecar trade-off: The sidecar model has a smaller operational surface than a dedicated fleet coordinator - there is no new central process to deploy, monitor, or scale. But it also means disaggregation decisions are made per-request by the scheduler rather than by a planner with persistent state about worker health and queue depth. For very dynamic traffic patterns where Prefill pool capacity needs to be rapidly rebalanced, Dynamo's Planner has more flexibility. For teams who prioritise Kubernetes-native operations over maximum dynamic flexibility, the sidecar model is simpler to manage.
5 Well-Lit Paths - llm-d's Tested Deployment Blueprints
▾

llm-d organises its capabilities around "well-lit paths" - pre-tested, benchmarked deployment configurations for specific goals. Each path comes with Helm charts, a deployment guide, and benchmark results so teams can validate expected performance before committing. As of v0.5 (February 2026), three primary well-lit paths are supported:

Well-lit path 1
Intelligent Inference Scheduling
Deploy vLLM with the IGW Inference Scheduler - KV Cache-aware, load-aware, SLA-aware routing - without any disaggregation. The simplest adoption path: existing vLLM deployments add the scheduler and immediately get smarter routing. Recommended starting point for any Kubernetes team with multiple vLLM replicas.
Benchmark: up to 3x TTFT improvement vs round-robin; doubles throughput under SLO constraints.
Well-lit path 2
Disaggregated Prefill / Decode
Deploy dedicated Prefill and Decode vLLM pods with the disaggregated sidecar and IGW scheduler routing between them. Requires InfiniBand or sufficient network bandwidth for KV transfer. Validated on NVIDIA H100/H200, Intel XPU, and Google TPU. The v0.4 release showed 40% reduction in TPOT for DeepSeek V3.1 on H200 GPUs with this configuration.
Benchmark: 40% TPOT reduction (DeepSeek V3.1, H200); order-of-magnitude TTFT reduction vs round-robin at scale.
Well-lit path 3
MoE / Large Model Serving (Expert Parallelism)
Native vLLM support for very large Mixture of Experts models (DeepSeek-R1, DeepSeek V3) with wide expert parallelism (EP) scaling. v0.3 validated 2,200 tokens/sec/GPU on H200. v0.5 reached 3,100 tokens/sec per B200 Decode GPU with wide-EP and 50,000 output tokens/sec on a 16x16 B200 Prefill/Decode topology.
Benchmark: 3,100 tok/s per B200 Decode GPU; 50k output tok/s on 16x16 B200 topology (v0.5).

6 llm-d vs Dynamo - The Meaningful Differences
▾

These two projects solve the same set of problems with different design philosophies. Neither is universally superior - the right choice depends on the customer's infrastructure context, hardware, and operational preferences.

DimensionNVIDIA Dynamollm-d
Primary backer NVIDIA Red Hat, Google, IBM - community governed
Inference engine TRT-LLM, vLLM, SGLang (backend-agnostic) vLLM primary; SGLang roadmap

They are converging, not diverging: llm-d v0.5 adopted NIXL for KV transfer, the same library Dynamo uses. Dynamo supports vLLM as a backend, the same engine llm-d uses. Both use Kubernetes for deployment. The meaningful differentiation is in the routing and coordination architecture - Dynamo's centralised Planner versus llm-d's Kubernetes-native scheduler - and in hardware scope. Teams should expect continued convergence on the transport and engine layers while the coordination models remain distinct.
7 4 Key Ideas to Carry into Module 5.4
▾
01
llm-d is the Kubernetes-native path - it extends what teams already operate rather than introducing new systems to learn
Teams running LLM workloads on Kubernetes can adopt llm-d with a Helm chart. The routing layer is Kubernetes Inference Gateway, the data plane is Envoy, the inference engine is vLLM - all standard components. This is the primary differentiator from Dynamo, which introduces new processes with their own operational surface.
02
Hardware-agnosticism is llm-d's structural advantage - the only choice for teams with AMD, Intel, or Google TPU alongside NVIDIA
Dynamo is NVIDIA-optimised. llm-d actively tests and supports NVIDIA A100+, AMD MI250, Google TPU v5e+, and Intel GPU Max. For any customer running multi-cloud or multi-vendor GPU infrastructure, llm-d is the only framework in this module that covers the full fleet without vendor lock-in.
03
The well-lit paths approach means adoption is incremental - start with intelligent scheduling, add disaggregation when justified
Path 1 (inference scheduling alone) delivers 3x TTFT improvement and doubled throughput under SLO with minimal operational change. Paths 2 and 3 add disaggregation and MoE support for teams ready for more complexity. No team has to adopt the full stack on day one - each path is independently deployable and benchmarked.
04
llm-d and Dynamo are converging on transport and engine while remaining distinct on coordination - the right choice depends on existing infrastructure, not performance alone
Both now use NIXL for KV transfer. Both support vLLM. The enduring difference is the coordination model (centralised Planner vs sidecar + K8s scheduler) and hardware scope. Performance on NVIDIA hardware is competitive. The decision is primarily operational: does the customer already run Kubernetes with standard tooling, or are they building a new NVIDIA-focused fleet from scratch?
MODULE 5.3 - LLM-D CHEAT SHEET
What llm-d is
Kubernetes-native distributed LLM inference framework. Red Hat + Google + IBM. Open-source (llm-d/llm-d). Uses vLLM as primary engine, Kubernetes Inference Gateway for routing, Envoy for data plane.
Inference Gateway (IGW)
Official Kubernetes project. Extends Gateway API with LLM concepts: InferenceModel, InferencePool CRDs, serving priority. GA as of IGW v1.0 (Oct 2025). Control plane for llm-d's scheduler.
Endpoint Picker Protocol (EPP)
llm-d's inference scheduler. Runs as Envoy plugin. Filter phase removes unavailable pods. Scoring phase ranks remaining by: KV cache hit, load, predicted latency, SLA priority, P/D assignment. Customisable with own scorers.
Disaggregation sidecar
Container alongside each Decode pod. Receives x-prefiller-host-port header from scheduler. Proxies Prefill to designated Prefill worker. Coordinates KV transfer back. Kubernetes-native - no separate fleet coordinator needed.
Well-lit paths
Tested deployment blueprints: (1) Intelligent scheduling only - 3x TTFT, 2x throughput. (2) Disaggregated P/D - 40% TPOT reduction. (3) MoE wide-EP - 3,100 tok/s per B200 Decode GPU. Each independently deployable via Helm.
Hardware support
NVIDIA A100+, AMD MI250, Google TPU v5e+, Intel GPU Max. The only framework in Part 5 supporting non-NVIDIA hardware. Critical differentiator for multi-cloud or mixed-vendor fleets.
v0.5 highlights (Feb 2026)
Hierarchical KV offload, active-active HA, cache-aware LoRA routing, scale-to-zero autoscaling, UCCL transport. Validated 3,100 tok/s per B200 Decode GPU and 50k output tok/s on 16x16 B200 P/D topology.
Choose llm-d when
Team already runs Kubernetes with standard tooling. vLLM is the inference engine. Non-NVIDIA hardware in the fleet. Operational simplicity is a priority. Prefer community-governed open-source over vendor-led project.
QUICK CHECK
A customer's platform engineering team runs all their ML workloads on Kubernetes and has strong Kubernetes expertise. They serve Llama 3 70B on a mixed fleet - 12x NVIDIA H100 GPUs and 8x AMD MI300X GPUs. They are experiencing the same routing blindness and disaggregation coordination problems described in Module 5.1. A colleague recommends deploying NVIDIA Dynamo for its 30x performance claims. What is the most accurate assessment and recommendation?
A. Deploy Dynamo - the 30x performance improvement outweighs any hardware compatibility concern, and Dynamo's vLLM backend support means it will work on AMD GPUs.
B. llm-d is the right choice for three compounding reasons. First, the fleet includes AMD MI300X GPUs - Dynamo is NVIDIA-optimised and its maximum performance numbers assume NVIDIA hardware, particularly Blackwell; llm-d actively tests and supports AMD MI250+ and would serve the full mixed fleet. Second, the team has strong Kubernetes expertise and existing K8s tooling - llm-d's Helm chart deployment and IGW integration fits their operational model without introducing new processes to learn. Third, llm-d's Well-lit Path 1 (intelligent scheduling alone) already addresses the routing blindness problem with a 3x TTFT improvement and delivers comparable benefits to Dynamo's KV-aware router on H100 hardware. The correct recommendation is llm-d starting with Well-lit Path 1 (inference scheduling), migrating to Path 2 (disaggregated P/D) when the team is ready for additional complexity.
C. Neither Dynamo nor llm-d is production-ready - both were announced in 2025 and lack the maturity for enterprise deployment. The customer should wait for more stable releases.
D. Separate the fleet - deploy Dynamo on the NVIDIA H100 nodes and a standalone vLLM with Triton on the AMD nodes, with a custom load balancer routing between them.
