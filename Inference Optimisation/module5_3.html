LLM-d - Kubernetes-Native Distributed LLM Inference

Where Dynamo is NVIDIA's fleet coordination layer, llm-d is an open-source, hardware-flexible, Kubernetes-native answer - built by a broader community including Red Hat, Google, IBM Research and other contributors - on top of vLLM and the Kubernetes Gateway API Inference Extension.

5.4 - Dynamo vs LLM-d vs DIY

llm-d was introduced in 2025 as an open-source project for high-performance distributed LLM inference on Kubernetes. It targets the same fleet-scale problems Dynamo addresses - routing blindness, disaggregation coordination - but from a different starting point: instead of building a new proprietary coordination layer on top of existing tools, llm-d extends the tools and standards that Kubernetes teams already operate.

The key distinction is the deployment model. Dynamo introduces a new fleet-level process (the Dynamo router, planner, and NIXL-based transfer system) that teams adopt alongside their existing stack. llm-d routes through Kubernetes-native infrastructure - the Gateway API Inference Extension, Envoy proxy, and standard Kubernetes APIs - so teams already running LLM workloads on Kubernetes can adopt it incrementally without introducing a separate control plane to operate.

1 The Core Philosophy - Extend Kubernetes, Don't Replace It
▾

The name "well-lit paths" is llm-d's own description of its goal: tested, benchmarked deployment patterns that teams can follow without building bespoke infrastructure. The phrase reflects the project's philosophy - rather than providing a novel system that teams must learn and operate, llm-d provides Kubernetes-native implementations of proven inference optimisations that plug into tooling teams already know.

This matters practically. A team running LLM workloads on Kubernetes already has Helm charts, Prometheus monitoring, Grafana dashboards, Kubernetes RBAC, and standard GitOps workflows. Adding Dynamo means adopting a new process with its own operational surface. Adopting llm-d typically means installing Helm charts that deploy the inference-aware Gateway and a configured vLLM deployment - things that look and behave like the Kubernetes resources the team already manages.

The foundational choice that shapes everything: llm-d chose vLLM as its primary inference engine, not TRT-LLM. For teams running mixed-accelerator infrastructure or planning for non-NVIDIA hardware, llm-d’s vLLM-first design keeps more options open than a stack that is tightly coupled to TRT-LLM.
2 The Inference Gateway and Scheduler - Smart Routing the Kubernetes Way
▾

The Gateway API Inference Extension is an emerging Kubernetes standard - not just a llm-d component - that extends the standard Gateway API with LLM-specific concepts. llm-d builds on this extension rather than building its own bespoke routing protocol, which means teams get a routing system that integrates with standard Kubernetes tooling and is evolving in the open with the broader Kubernetes ecosystem.

Inference extension and scheduler components - click to explore

The Endpoint Picker Protocol (EPP) is where llm-d's intelligence lives. Rather than reimplementing a full-featured load balancer, the EPP plugs into Envoy's extensible balancing policies as a custom filter. This design means llm-d inherits Envoy's production-grade flow control, connection management, circuit breaking, and observability - while adding the LLM-specific scoring logic on top.

Why Envoy matters here: Envoy is the data plane used by Istio, AWS App Mesh, and many service meshes deployed in production Kubernetes environments. By building on Envoy, llm-d integrates naturally with existing mesh and gateway deployments. A team already running an Envoy-based gateway or service mesh can add llm-d's inference-aware routing by introducing the EPP filter into the existing Envoy configuration instead of standing up a new proxy stack.
3 How the Inference Scheduler Scores Pods - Filter, Score, Select
▾

When a request arrives at the inference gateway, the scheduler does not pick a pod arbitrarily. It runs a two-phase process: first it filters out pods that cannot serve the request (overloaded, wrong model version, unhealthy), then it scores the remaining candidates across multiple dimensions and routes to the highest-scoring pod.

Scheduler scoring - select a request type to see how pods are evaluated

Teams can customise the scoring algorithm by implementing their own scorer plugins - the EPP defines an interface that any scorer must implement, allowing organisations to add business-specific logic (e.g., prefer pods in a specific availability zone, penalise pods with recent errors) without forking the core scheduler. The default scorers cover KV Cache-awareness, load balancing, SLA priority, and P/D disaggregation routing. Experimental work adds predicted-latency-aware scoring that uses request length and queue state to estimate completion time, rather than relying only on current load.

Predicted latency balancing (experimental): llm-d’s scheduler includes support for a predicted latency scorer - rather than just looking at current load and cache state, it estimates the expected completion time for each pod given the request's input/output length and the pod's current queue. Early benchmarks show that this can significantly improve P90 latency for long-Prefill workloads compared to simple load-only balancing. This is a meaningful capability gap versus basic round-robin: a pod with low connection count but a large in-flight long-context request may have worse predicted latency than a busier pod with shorter requests.
4 Disaggregated Serving in llm-d - The Sidecar Model
▾

llm-d implements disaggregation differently from Dynamo. Rather than a dedicated fleet-level coordination process, disaggregation in llm-d is implemented via a sidecar container that runs alongside each Decode pod. The scheduler decides which Prefill pod should handle the Prefill phase and encodes that decision in an HTTP header (x-prefiller-host-port). The sidecar on the Decode pod reads this header and proxies the Prefill portion to the designated Prefill worker, then coordinates the KV Cache transfer back.

For KV Cache transfer, llm-d can integrate with high-bandwidth mechanisms such as NIXL, alongside vLLM’s own KV handling, so teams using NVIDIA hardware can align with the same class of RDMA-based transfers that Dynamo uses for inter-node handoff. The project is designed with hardware flexibility in mind, so support for different accelerators and transports continues to expand as the ecosystem matures.

The sidecar trade-off: The sidecar model has a smaller operational surface than a dedicated fleet coordinator - there is no new central process to deploy, monitor, or scale. But it also means disaggregation decisions are made per-request by the scheduler rather than by a planner with long-lived global state about worker health and queue depth. For very dynamic traffic patterns where Prefill pool capacity needs to be rapidly rebalanced, Dynamo's Planner can offer more centralised control. For teams who prioritise Kubernetes-native operations and simpler day-2 management, the sidecar model is attractive because it fits directly into existing pod lifecycle and deployment workflows.
5 Well-Lit Paths - llm-d's Tested Deployment Blueprints
▾

llm-d organises its capabilities around "well-lit paths" - pre-tested deployment configurations for specific goals. Each path is intended to come with Helm charts, deployment guidance, and benchmark results so teams can validate expected behaviour before committing. Conceptually, three primary well-lit paths are important to understand:

Well-lit path 1
Intelligent Inference Scheduling
Deploy vLLM behind an inference-aware Gateway and scheduler - KV Cache-aware, load-aware, SLA-aware routing - without any disaggregation. This is the simplest adoption path: existing vLLM deployments add the scheduler and immediately get smarter routing, without changing their model pods.
Impact: significantly improves TTFT and overall latency compared to naive round-robin, especially under mixed workload and multi-tenant conditions.
Well-lit path 2
Disaggregated Prefill / Decode
Deploy dedicated Prefill and Decode vLLM pods with the disaggregated sidecar and scheduler routing between them. This path takes advantage of the different characteristics of the two phases - Prefill is compute-bound, Decode is memory-bandwidth-bound - and uses the network for KV transfer between pods.
Impact: can reduce end-to-end latency and improve throughput for long-context and RAG-style workloads when network and accelerator topology are configured appropriately.
Well-lit path 3
MoE / Large Model Serving (Expert Parallelism)
Use vLLM with llm-d’s routing to serve very large models, including Mixture-of-Experts architectures, with wide expert parallelism (EP) scaling across many accelerators. The routing layer helps keep traffic balanced across experts and replicas while respecting KV-cache locality and SLAs.
Impact: enables high utilisation and competitive tokens-per-second on modern GPUs for large and MoE models, without teams having to design their own expert-aware routing layer.

6 llm-d vs Dynamo - The Meaningful Differences
▾

These two projects address similar problems with different design philosophies. Neither is universally superior - the right choice depends on the customer's infrastructure context, hardware, and operational preferences.

They are converging, not diverging: llm-d is designed to integrate with the same open components (vLLM, NIXL, Envoy, Gateway API Inference Extension) that also appear around Dynamo. Dynamo supports vLLM as a backend, the same engine llm-d uses. Both use Kubernetes as the deployment substrate. The meaningful differentiation is in the routing and coordination architecture - Dynamo's centralised Planner versus llm-d's Kubernetes-native scheduler and sidecar model - and in how tightly each stack is coupled to NVIDIA hardware. Teams should expect continued convergence on the transport and engine layers while the coordination models remain distinct.
7 4 Key Ideas to Carry into Module 5.4
▾
01
llm-d is the Kubernetes-native path - it extends what teams already operate rather than introducing new systems to learn
Teams running LLM workloads on Kubernetes can adopt llm-d with a Helm chart and CRDs. The routing layer is built on the Gateway API Inference Extension, the data plane is Envoy, and the inference engine is vLLM - all standard, well-understood components. This is a primary differentiator from Dynamo, which introduces a new, dedicated coordination layer with its own operational surface.
02
Hardware flexibility is a structural goal for llm-d - it is designed to work across accelerators, not just one vendor
Dynamo is tightly optimised for NVIDIA hardware. llm-d, by contrast, is positioned as a vendor-neutral framework that leverages open projects like vLLM, Envoy and the Gateway API Inference Extension and is intended to support “any model, any accelerator, any cloud” over time. For customers running multi-cloud or considering more than one accelerator vendor, llm-d’s design reduces the risk of hard lock-in to a single GPU stack.
03
The well-lit paths approach means adoption is incremental - start with intelligent scheduling, add disaggregation and MoE when justified
Path 1 (inference scheduling alone) improves routing quality with minimal operational change. Paths 2 and 3 add disaggregation and support for larger and MoE-style models for teams ready to invest in more sophisticated topologies. No team has to adopt the full stack on day one - each path is independently deployable and builds on familiar Kubernetes constructs.
04
llm-d and Dynamo are converging on engines and transports while remaining distinct on coordination - the right choice depends on existing infrastructure, not performance numbers alone
Both can use vLLM as an inference engine and both can integrate with modern KV transport libraries like NIXL. The enduring difference is the coordination model (centralised Planner vs. Envoy-based scheduler + sidecar) and how each project fits into a customer's Kubernetes and hardware strategy. For many customers, the decision is primarily operational: do they want a tightly NVIDIA-optimised coordination layer, or a Kubernetes-native, vendor-neutral blueprint they can extend?
MODULE 5.3 - LLM-D CHEAT SHEET
What llm-d is
Kubernetes-native distributed LLM inference framework. Created and driven by contributors from IBM Research, Red Hat, Google Cloud and the wider community. Open-source (llm-d/llm-d). Uses vLLM as the primary engine, the Gateway API Inference Extension for routing control, and Envoy as the data plane.
Inference Gateway / extension
An emerging Kubernetes Gateway API extension that adds LLM concepts like InferenceModel and InferencePool. Provides model-aware routing, serving priority and “smart” load-balancing hooks that llm-d uses as part of its control plane.
Endpoint Picker Protocol (EPP)
llm-d's inference scheduler. Runs as an Envoy plugin. Filter phase removes unavailable or incompatible pods. Scoring phase ranks remaining pods by KV cache signal, load, predicted latency, SLA priority and P/D role. Exposes an interface for custom scorer plugins so organisations can add their own routing policies.
Disaggregation sidecar
Container alongside each Decode pod. Receives the x-prefiller-host-port header from the scheduler. Proxies the Prefill portion of the request to the designated Prefill worker and coordinates KV transfer back. Uses Kubernetes-native deployment and lifecycle instead of a separate fleet coordinator process.
Well-lit paths
Deployment blueprints that map to common goals: (1) Intelligent scheduling only - better latency and utilisation without changing model pods. (2) Disaggregated P/D - dedicated Prefill and Decode pools with networked KV transfer. (3) MoE / large models - expert-parallel and large-model serving backed by vLLM and inference-aware routing.
Hardware posture
Designed to be vendor-neutral: built around open components (vLLM, Envoy, Gateway API Inference Extension, NIXL) with the goal of working across different accelerators and clouds, not only NVIDIA GPUs.
Project status
Donated as a CNCF sandbox project at KubeCon Europe 2026, giving it open governance and a path to broader standardisation while it continues to evolve technically and operationally.
Choose llm-d when
The team already runs Kubernetes with standard tooling, vLLM is the inference engine (or will be), avoiding tight coupling to a single accelerator vendor is important, and operational simplicity via Kubernetes-native patterns is a priority.
QUICK CHECK
A customer's platform engineering team runs all their ML workloads on Kubernetes and has strong Kubernetes expertise. They serve Llama 3 70B on a mixed fleet - 12x NVIDIA H100 GPUs and 8x AMD MI300X GPUs. They are experiencing the same routing blindness and disaggregation coordination problems described in Module 5.1. A colleague recommends deploying NVIDIA Dynamo for its headline performance claims on NVIDIA-only fleets. What is the most accurate assessment and recommendation?
A. Deploy Dynamo - the NVIDIA-side performance improvements justify adopting it even if AMD support is limited, and vLLM as a backend should allow it to handle the MI300X nodes with minimal trade-offs.
B. llm-d is the better strategic fit for three reasons. First, the fleet includes AMD MI300X GPUs - Dynamo is optimised for NVIDIA hardware and its strongest performance claims are for NVIDIA-only deployments, whereas llm-d is explicitly positioned as vendor-neutral and is designed to work across different accelerators via vLLM and open transports. Second, the team has strong Kubernetes expertise and existing K8s tooling - llm-d’s Helm chart deployment and Gateway API Inference Extension integration fit their operational model without introducing a separate coordination plane to learn. Third, llm-d’s “intelligent scheduling only” path already addresses routing blindness using KV cache-aware and load-aware routing, delivering many of the same benefits as Dynamo’s KV-aware router on the NVIDIA portion of the fleet while also covering the AMD nodes. The recommended starting point is llm-d with intelligent scheduling, adding disaggregation later if needed.
C. Neither Dynamo nor llm-d should be considered because they are both too immature for production - the customer should avoid them entirely until they reach a 1.0 GA release and rely on a simple round-robin load balancer instead.
D. Separate the fleet - deploy Dynamo on the NVIDIA H100 nodes and a standalone vLLM stack on the AMD nodes, with a custom load balancer that treats each sub-fleet as an opaque endpoint.
