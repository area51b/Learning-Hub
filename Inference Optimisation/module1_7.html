2.1 - What Is Inference?

In Module 1.6 you saw the complete transformer block: LayerNorm, Multi-Head Attention, residual connection, LayerNorm again, FFN, residual again - repeated N times. That architecture does not change, what changes is how it runs depending on what it is doing.

Every time a model processes a request, the transformer block operates in two distinct phases: Prefill, where it processes the entire input at once, and Decode, where it generates one output token at a time. These two phases stress the hardware in completely different ways - and that difference is the root cause behind almost every optimisation technique covered in this course.

This module makes that difference concrete, corrects a common misconception about where the bottleneck actually lives inside the GPU, and shows how Prefill and Decode map directly to the two latency metrics that surface in every inference conversation.

The transformer block architecture is identical in both modes. The difference is the batch size entering it: all input tokens at once during Prefill, one token per step during Decode. This single difference produces two completely different hardware bottlenecks.
The key insight: Prefill and Decode are not two different systems. They are the same transformer block running with a different batch size. Prefill runs with batch size equal to the full input length - massive parallelism, high GPU utilisation. Decode runs with batch size of one - serial, low GPU compute utilisation, dominated instead by the cost of reading data.
2 The Two Phases - Characteristics and Bottlenecks
▾

Because the batch size entering the transformer block is so different between phases, the hardware bottleneck is completely different in each one.

Phase 1
Prefill
Processing the input
All input tokens processed in parallel in one pass
K and V vectors written to the KV Cache for every input token
Runs once per request before any output appears
Bottleneck: compute (FLOPS) - the GPU is doing large parallel matrix multiplications
Hardware need: high Tensor Core throughput
GPU utilisation: high - Tensor Cores are working hard
Phase 2
Decode
Generating the output
One token generated per step - strictly sequential
Each step reads the full KV Cache for all previous tokens
Repeats for every output token - 500 tokens means 500 iterations
Bottleneck: HBM bandwidth - the compute cores wait for data reads
Hardware need: high HBM memory bandwidth (GB/s)
GPU Tensor Core utilisation: low - cores sit idle waiting for data
The core trade-off: Prefill wants raw compute throughput. Decode wants memory read speed. These are different hardware properties - and they cannot both be maximised on the same GPU at the same time. This is the fundamental tension that drives disaggregation, specialised hardware configurations, and most of the optimisation techniques in Part 3.
3 Inside the GPU: Why Decode Is Memory-Bandwidth-Bound
▾

A common point of confusion: if the bottleneck during Decode is memory, and the KV Cache is stored on the GPU, why is there a bottleneck at all? The answer requires understanding that a GPU is not one homogeneous thing - it has two distinct internal components that matter here.

Inside the GPU package - compute cores and HBM are separate components
GPU package (e.g. H100 SXM)
Compute Cores (SM)
Streaming Multiprocessors - where all matrix multiplications and attention calculations actually happen. Extremely fast. Only a few MB of on-chip SRAM (registers and shared memory). Cannot store the KV Cache here - it is far too small.
HBM (High Bandwidth Memory)
The GPU's main memory - the "80 GB" in an H100. Physically adjacent to the compute dies but a separate component. This is where model weights, the KV Cache, and activations are stored. Fast relative to system RAM, but reading from it takes time.
Every decode step:
KV Cache transferred from HBM to compute cores
3.35 TB/s
H100 SXM HBM3 bandwidth - the speed of this internal transfer is the Decode bottleneck
The compute cores and HBM are on the same physical package but are distinct components connected by a high-bandwidth bus. The bottleneck during Decode is the speed of that internal bus - not the compute cores themselves, and not HBM capacity. This is why HBM bandwidth (TB/s) is the critical spec for Decode performance, and why quantising the KV Cache (reducing bytes per entry) directly reduces Decode latency.

During every single Decode step, the compute cores must read the entire accumulated KV Cache from HBM across this internal bus, perform the attention calculation, and produce one token. As the response grows longer, the KV Cache grows, and each step requires transferring more data across the same bus. The compute cores are fast enough - they are waiting for the data to arrive.

A more precise analogy: Think of the compute cores as a chef at a preparation station with almost no counter space. HBM is a large pantry in the same building - nearby and fast, but a separate room. Every Decode step, an assistant must carry all the accumulated ingredients from the pantry to the counter, the chef uses them in seconds, and then the next step starts. As the recipe grows longer, there are more ingredients to carry each trip. The chef is never the bottleneck. The carrying is.

This is also why the fix for slow Decode is not a faster GPU in terms of raw FLOPS - it is higher HBM bandwidth, or reducing the amount of data that needs to cross the bus per step. KV Cache quantisation (covered in Module 3.3) directly addresses this by shrinking the size of each KV entry.

During Decode, the GPU compute utilisation drops to around 30% - not because the GPU is slow, but because it is waiting for memory reads. The GPU has to load the entire KV Cache from HBM (High Bandwidth Memory) for every single token it generates. This is the memory bandwidth bottleneck in action.

The restaurant analogy: Prefill is like a chef who gets all the ingredients delivered at once and does intense prep work - chopping, mixing, marinating - all in parallel. Decode is like a chef who must run back to the walk-in fridge for one ingredient before every single step of cooking. The fridge trips (memory reads) are what slow things down, not the actual cooking.

You learned in Module 1.3 that the KV Cache stores Key and Value vectors so they do not need to be recomputed. Now you can see exactly when and how it grows:

Blue = prompt tokens. Orange = generated tokens. Each decode step reads the entire growing cache - this is why decode gets progressively slower as the response gets longer, and why KV Cache memory management is critical.
The compounding problem: A 1,000-token output on a 4,000-token prompt means the decode phase eventually reads a 5,000-entry KV Cache on every single step. The last tokens take longer to generate than the first. For applications with long outputs - code generation, long-form writing, chain-of-thought reasoning - this KV Cache growth is the dominant inference cost driver.
7 The Two Metrics - TTFT and TPOT
▾

Because Prefill and Decode are distinct phases with different bottlenecks, they map to two distinct latency metrics that appear in almost every inference performance discussion.

TTFT
Time to First Token
Time from sending the prompt to receiving the first output token. Driven entirely by the Prefill phase. Longer input sequences produce higher TTFT.
Bottleneck: Prefill compute - Tensor Core throughput
TPOT
Time per Output Token
Time between each successive output token after the first. Driven entirely by the Decode phase. Longer responses accumulate a larger KV Cache, which increases TPOT over time.
Bottleneck: HBM bandwidth - KV Cache read per step
Diagnosing performance problems: When a system feels slow, identifying which metric is high determines which phase to investigate. High TTFT with acceptable TPOT points to a Prefill bottleneck - long input, insufficient compute throughput, or Prefill interference from concurrent requests. High TPOT with acceptable TTFT points to a Decode bottleneck - large KV Cache, insufficient HBM bandwidth, or memory pressure from too many concurrent sessions. The solutions are different for each.
8 Why Disaggregation Exists - Built From First Principles
▾

Now that Prefill and Decode are concrete, the logic behind disaggregation follows directly. The question is simply: if these two phases need different hardware resources and run at different times, why force them to share the same GPU?

The case for disaggregation - from first principles
Prefill needs
All tokens processed in one parallel pass
Bottleneck: Tensor Core compute (FLOPS)
Ideal GPU: high FLOPS, many compute cores
Runs once per request then hands off
KV Cache written, then transferred
≠
Decode needs
One token generated per step, serially
Bottleneck: HBM bandwidth (TB/s)
Ideal GPU: high HBM bandwidth, large HBM
Runs hundreds of times per request
KV Cache read on every single step
The disaggregation approach: Separate Prefill and Decode onto dedicated GPU pools, each configured for its specific bottleneck. Prefill GPUs handle only the compute-intensive first phase. Decode GPUs handle only the bandwidth-intensive generation phase. Once a request finishes Prefill, its KV Cache is transferred to a Decode GPU to complete generation. Each pool can be scaled independently based on the workload profile.
The trade-off to acknowledge: Disaggregation introduces KV Cache transfer overhead - the cache built during Prefill must be moved to the Decode GPU before generation can begin. Whether this trade-off is worth it depends on scale, input length, and the network fabric between nodes. Module 3.6 covers this in detail. MODULE 1.7 - CHEAT SHEET
Prefill Phase
The transformer block runs on all input tokens simultaneously. Compute-bound. Runs once per request. Writes K and V to the KV Cache. Determines TTFT.
Decode Phase
The transformer block runs once per output token. HBM-bandwidth-bound. Reads the full KV Cache on every step. Determines TPOT. Gets slower as responses grow longer.
Compute cores vs HBM
Compute cores (SMs) do the maths - fast but tiny on-chip memory. HBM stores the KV Cache and model weights - large but a separate component. Decode is slow because cores wait for HBM reads, not because cores are slow.
HBM Bandwidth
The speed of the internal transfer between HBM and compute cores. Measured in TB/s. The key spec for Decode performance. Quantising the KV Cache reduces bytes transferred per step, directly improving TPOT.
TTFT (Time to First Token)
Latency from sending the prompt to receiving the first output token. Driven by Prefill. Bottleneck: Tensor Core compute throughput.
TPOT (Time per Output Token)
Latency between successive output tokens. Driven by Decode. Bottleneck: HBM bandwidth. Grows as KV Cache accumulates across a long response.
Why disaggregation exists
Prefill needs Tensor Core throughput. Decode needs HBM bandwidth. These are different hardware properties. Disaggregation puts each phase on GPUs optimised for its specific bottleneck.
KV Cache Transfer
In disaggregated systems, the KV Cache built during Prefill is transferred to the Decode GPU before generation begins. The speed and cost of this transfer is the main operational trade-off of disaggregation.
QUICK CHECK
A team runs a coding assistant on a single H100 GPU. Users submit short prompts (around 200 tokens) but expect long, detailed code outputs (around 2,000 tokens). They report that the first line of code appears quickly, but the full response takes much longer than expected to complete. Which phase is the likely bottleneck and what is the most accurate explanation?
A. Prefill is the bottleneck. A 200-token prompt is relatively short but the GPU is spending too long computing the initial attention matrix before any output appears.
B. Decode is the bottleneck. With 2,000 output tokens, the KV Cache grows to over 2,200 entries by the end. The compute cores must read this entire cache from HBM on every single step, and HBM bandwidth becomes saturated - causing TPOT to be high even though TTFT is acceptable.
C. Both phases are equally slow. The H100 is undersized for this workload and needs to be replaced with a GPU that has both higher FLOPS and more HBM capacity.
D. Decode is the bottleneck, but the cause is insufficient Tensor Core compute. The GPU cannot perform the attention calculation fast enough for long output sequences.
