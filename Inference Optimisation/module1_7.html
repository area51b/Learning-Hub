Part 2 - How Inference Actually Works

Every time a model generates a response, it goes through exactly two phases: Prefill and Decode. These two phases are fundamentally different in what they demand from hardware - and understanding this difference is what unlocks the ability to explain almost every major inference optimisation technique to a customer.

This is the module where the entire Part 1 foundation comes together. By the end, you will understand not just what disaggregation is - but why it exists, why it works, and how to explain it confidently without notes.

1 The Two Phases - Side by Side
▾

When you send a prompt to an LLM, the model does not generate the entire response at once. It works in two distinct phases with completely different characteristics:

Phase 1
Prefill
Processing your input
Reads and processes all input tokens at once in parallel
Builds the KV Cache - one entry per input token
Happens once per request, before any output appears
Duration: milliseconds to seconds (depends on prompt length)
Bottleneck: compute - the GPU is doing massive parallel maths
Hardware need: high FLOPS (raw computing power)
Phase 2
Decode
Generating your output
Generates output tokens one at a time, sequentially
Each step reads the full KV Cache to maintain context
Repeats for every single output token - 500 tokens = 500 iterations
Duration: seconds to minutes (depends on output length)
Bottleneck: memory bandwidth - constantly reading the KV Cache
Hardware need: high memory bandwidth (GB/s throughput)
The single most important takeaway: Prefill and Decode are not just different speeds - they are bottlenecked by different hardware resources. Prefill needs raw compute (FLOPS). Decode needs memory bandwidth. This is why running them on the same hardware is inherently a compromise - and exactly why disaggregation exists.

3 Why They Need Different Hardware Resources
▾

The reason Prefill and Decode feel so different is because they stress the GPU in completely different ways. Here is a direct comparison of the four key resource dimensions:

During Decode, the GPU compute utilisation drops to around 30% - not because the GPU is slow, but because it is waiting for memory reads. The GPU has to load the entire KV Cache from HBM (High Bandwidth Memory) for every single token it generates. This is the memory bandwidth bottleneck in action.

The restaurant analogy: Prefill is like a chef who gets all the ingredients delivered at once and does intense prep work - chopping, mixing, marinating - all in parallel. Decode is like a chef who must run back to the walk-in fridge for one ingredient before every single step of cooking. The fridge trips (memory reads) are what slow things down, not the actual cooking.
4 How the KV Cache Grows Through Both Phases
▾

You learned in Module 1.3 that the KV Cache stores Key and Value vectors so they do not need to be recomputed. Now you can see exactly when and how it grows:

Blue = prompt tokens. Orange = generated tokens. Each decode step reads the entire growing cache - this is why decode gets progressively slower as the response gets longer, and why KV Cache memory management is critical.
The compounding problem: A 1,000-token output on a 4,000-token prompt means the decode phase eventually reads a 5,000-entry KV Cache on every single step. The last tokens take longer to generate than the first. For applications with long outputs - code generation, long-form writing, chain-of-thought reasoning - this KV Cache growth is the dominant inference cost driver.
5 The Two Metrics Every Customer Cares About
▾

Because Prefill and Decode are separate phases, they map directly to two distinct latency metrics that come up in almost every customer conversation about inference performance:

TTFT
Time to First Token
How long the user waits before anything appears on screen. Driven entirely by the Prefill phase. Long prompts = longer TTFT.
Typical target: under 500ms for chat applications
TPOT
Time per Output Token
How fast each subsequent token appears after the first. Driven by the Decode phase. Determines how "smooth" the stream feels to the user.
Typical target: under 30ms per token for real-time feel
Connecting to optimisation techniques: When a customer says "our model feels slow", you need to ask which type of slow. High TTFT? Focus on Prefill optimisation - chunked prefill, prefill-specific hardware, or Flash Attention. High TPOT? Focus on Decode optimisation - KV Cache management, Paged Attention, memory bandwidth improvements. They have different solutions.
6 Your Bridge to Disaggregation - Why It Exists
▾

You told me at the start that you are working on disaggregation but could not explain the foundation. You now have everything you need. Here is how it all connects:

The logical case for disaggregation - built from first principles
Prefill requirements
Processes all tokens at once - massive parallelism
Bottlenecked by GPU compute (FLOPS)
Ideal GPU: high core count, high FLOPS
Runs once per request, then done
KV Cache written, then handed off
≠
Decode requirements
Generates one token at a time - serial
Bottlenecked by memory bandwidth
Ideal GPU: high HBM bandwidth, large cache
Runs hundreds of times per request
KV Cache read on every single step
The disaggregation insight: If Prefill and Decode need different hardware and run at different times, why force them to share the same GPU? Disaggregation separates Prefill and Decode onto dedicated GPU pools. Prefill GPUs are optimised for compute throughput. Decode GPUs are optimised for memory bandwidth. Once a request finishes Prefill, its KV Cache is transferred to a Decode GPU to complete generation. Each pool can be scaled independently based on workload - more Prefill capacity for bursty chatbot loads, more Decode capacity for long-form generation workloads.
Why this matters in your customer conversations: When a customer asks "what is disaggregation and why should I care?", you can now answer from first principles - Prefill is compute-bound, Decode is memory-bound, they need different hardware, disaggregation lets you optimise each independently. You can also explain the trade-off: disaggregation adds complexity (KV Cache transfer overhead, more infra to manage) but unlocks significant throughput and cost improvements at scale.

MODULE 1.7 - CHEAT SHEET
Prefill Phase
Processes all input tokens in parallel. Compute-bound. Runs once. Builds the KV Cache. Determines TTFT.
Decode Phase
Generates output one token at a time. Memory-bandwidth-bound. Reads full KV Cache every step. Determines TPOT.
TTFT (Time to First Token)
Latency from sending prompt to first output token appearing. Driven by Prefill. Critical for interactive applications.
TPOT (Time per Output Token)
Latency between each successive output token. Driven by Decode. Determines how smooth generation feels.
Compute-bound
The bottleneck is GPU FLOPS - raw calculation speed. Prefill is compute-bound. More FLOPS = faster Prefill.
Memory-bandwidth-bound
The bottleneck is how fast data can be read from GPU memory. Decode is memory-bound. Higher HBM bandwidth = faster Decode.
Disaggregation
Separating Prefill and Decode onto dedicated GPU pools, each optimised for its specific bottleneck. Enables independent scaling of each phase.
KV Cache Transfer
In disaggregated systems, the KV Cache built during Prefill must be sent from the Prefill GPU to the Decode GPU. This transfer overhead is the main trade-off of disaggregation.
