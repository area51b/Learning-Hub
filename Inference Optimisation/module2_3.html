Batching Strategies

How grouping requests together transforms GPU utilisation - and why continuous batching is the single biggest throughput improvement in modern inference serving.

~10 min read Concept level

2.4 - Throughput vs Latency

A GPU running a single request at a time is like a 747 flying with one passenger - an extraordinary waste of capacity. Batching is the practice of processing multiple requests simultaneously on the same GPU forward pass. Done well, it is the primary mechanism for achieving high throughput and low cost per token.

This module covers three batching strategies in order of sophistication: static batching, dynamic batching, and continuous batching. Each solves a problem the previous one created. By the end, you will understand why continuous batching - the approach used in vLLM and modern inference servers - represents such a significant advance over earlier methods.

1 Why a GPU Needs Batching to Be Efficient
▾

Modern GPUs contain thousands of parallel compute cores - an H100 has over 16,000 CUDA cores. A single inference request uses a fraction of them. Running requests one at a time wastes the GPU's core parallelism and leaves most of the hardware idle.

GPU utilisation rises sharply with batch size during the Decode phase - this is where batching has its greatest impact. Running many decode steps in parallel costs almost the same as one, since the bottleneck is memory bandwidth (reading the model weights and KV Cache), not compute.
The key insight: During Decode, the GPU is memory-bandwidth-bound - it reads model weights and KV Cache from HBM regardless of how many tokens it generates. Processing 64 decode tokens simultaneously costs nearly the same memory reads as processing 1. This is why batching dramatically improves throughput at minimal latency cost per token - you are amortising the same memory reads across many requests.
2 Static Batching - The Naive Approach
▾
Static
Collect N requests, process together, return all, repeat
The simplest batching strategy: wait until you have a fixed number of requests (e.g. 8), run them all through the model together, wait for all to finish, then start the next batch. Used in early ML inference and still common in batch processing pipelines.
// Static batch of 4 - all must FINISH before next batch starts Req A [prompt=50tok, output=12tok ] ████░░░░░░░░░░░░ DONE at step 12 Req B [prompt=200tok, output=180tok] ████████████████████████████ DONE at step 180 Req C [prompt=30tok, output=8tok ] ████░░░░░░░░░░░░░░░░░░░░░░░░ DONE at step 8 Req D [prompt=100tok, output=95tok ] ████████████████████████░░░░ DONE at step 95 ^ Req A, C, D sit IDLE waiting for Req B
Simple to implement and reason about
Requests that finish early sit idle, wasting GPU slots
Predictable memory usage - all requests known upfront
The slowest request in the batch determines total time
Good for batch jobs with uniform output lengths
GPU utilisation collapses as output lengths diverge
The padding problem: LLMs cannot natively process sequences of different lengths in the same batch - all sequences must be padded to match the longest one. A batch where one request outputs 200 tokens and three output 10 tokens wastes 95% of compute on the shorter requests. This is the core inefficiency static batching cannot solve.
3 Dynamic Batching - Smarter Grouping
▾
Dynamic
Group requests by similarity; adjust batch size by load
Dynamic batching improves on static batching by forming batches intelligently rather than rigidly. Requests with similar sequence lengths are grouped together to minimise padding waste. Batch size adjusts based on current GPU load and available memory. A timeout ensures requests are not held waiting too long.
// Dynamic batch - similar lengths grouped, smaller padding waste Batch 1: Req A [50tok], Req C [30tok] → padded to 50 - minimal waste Batch 2: Req B [200tok], Req D [100tok] → padded to 200 - some waste Result: shorter requests no longer wait for long ones
Reduces padding waste vs static batching
Still waits for a batch to form (adds latency)
Adapts batch size to server load
Length prediction is imperfect - output lengths unknown upfront
Used in Triton Inference Server for non-LLM models
Does not solve the "one slow request blocks the batch" problem
The fundamental limitation remaining: Both static and dynamic batching treat a batch as an atomic unit - all requests enter together and exit together. The root problem is that LLM output lengths are highly variable and unpredictable. Even with smart grouping, a batch is held hostage by its longest request. What is needed is a way to add and remove requests from the batch mid-flight - which is exactly what continuous batching provides.
4 Continuous Batching - The Modern Standard
▾
Continuous
Insert new requests the moment a slot frees - no batch boundaries
Continuous batching (also called iteration-level scheduling or in-flight batching) eliminates the concept of a batch boundary entirely. The GPU runs one decode step at a time. After each step, any request that generated a stop token is removed from the active set and a waiting request is immediately inserted in its place - without waiting for other active requests to finish.
// Continuous batch - slots fill immediately when freed Step 1-8: [Req A] [Req B] [Req C] [Req D] Step 9: [DONE:C] → [NEW:E ] inserted immediately Step 10-12: [Req A] [Req B] [Req E] [Req D] Step 13: [DONE:A] → [NEW:F ] inserted immediately No idle slots. GPU stays fully utilised.
Near-100% GPU slot utilisation at all times
More complex KV Cache memory management required
Short requests do not wait for long ones
Requires per-step scheduling overhead
Throughput 5–10x higher than static batching
Latency for individual requests can vary under high load
Foundation of vLLM and TensorRT-LLM serving
Not useful for batch offline jobs with uniform lengths

Every slot is occupied at every step. When a request finishes, a new one enters immediately - no idle time, no waiting for the batch to synchronise.
Why this was a step change: The 2022 paper "Orca: A Distributed Serving System for Transformer-Based Generative Models" introduced iteration-level scheduling (what we now call continuous batching). Benchmarks showed 36x higher throughput compared to static batching for the same latency budget. vLLM implemented this alongside Paged Attention and became the dominant open-source inference server almost immediately. Continuous batching is now table stakes for any production LLM serving system.
5 Choosing the Right Strategy - and the Latency-Throughput Dial
▾
Strategy Best for Throughput Latency Used in
Static batching Offline batch jobs, uniform outputs Low Predictable Early systems, custom pipelines
Dynamic batching Mixed workloads, non-LLM models Moderate Moderate Triton, ONNX Runtime
Continuous batching Online LLM serving, variable outputs Very high Low per-request vLLM, TensorRT-LLM, SGLang

Even with continuous batching, there is a dial between latency and throughput. The maximum batch size you allow - and how aggressively you fill it - determines where you sit on the trade-off curve:

Batching aggressiveness dial
Latency-optimised (small batches) Throughput-optimised (large batches)
The practical guidance: For interactive chat applications, lean toward smaller batch sizes (4–16) with low max-wait time to keep TTFT under 500ms. For throughput-focused APIs where users accept slight latency increases for lower cost, larger batch sizes (32–128) can dramatically cut cost per token. Most production systems expose a max_batch_size and batch_timeout parameter for exactly this dial.
6 4 Key Ideas to Remember
▾
01
A GPU running one request at a time is severely underutilised
During Decode, the GPU's bottleneck is reading model weights from memory - a cost paid once regardless of batch size. Processing 64 requests in parallel costs nearly the same memory reads as processing 1, making batching the primary lever for cost efficiency.
02
Static batching wastes slots when output lengths vary
When one request in a static batch takes 10x longer than the others, those other slots sit idle for the duration. With highly variable LLM outputs, this can waste 80–90% of GPU capacity in the worst cases.
03
Continuous batching eliminates batch boundaries - slots never sit idle
By scheduling at the iteration (decode step) level rather than the batch level, completed requests are immediately replaced by waiting ones. This is the primary reason vLLM and TensorRT-LLM achieve such dramatically higher throughput than earlier systems.
04
Batch size is a latency-throughput dial - not a set-and-forget parameter
Larger batches increase throughput and reduce cost per token but increase individual request latency. The right setting depends on the workload: latency-sensitive chat applications want small batches with tight timeouts; cost-sensitive batch APIs want large batches with relaxed timeouts.
MODULE 2.3 - CHEAT SHEET
Static Batching
Fixed batch size, all requests enter and exit together. Simple but wastes GPU slots when output lengths differ. Good for uniform offline jobs.
Dynamic Batching
Groups requests by similarity, adjusts batch size by load, uses a timeout. Better padding efficiency than static. Still treats the batch as atomic.
Continuous Batching
Iteration-level scheduling - completed requests are replaced immediately by new ones. No idle slots. 5–10x throughput vs static batching. Used in vLLM and TensorRT-LLM.
Iteration-level Scheduling
Making scheduling decisions after every decode step (every token generated) rather than per batch. The mechanism that makes continuous batching work.
Padding
Extending shorter sequences with dummy tokens to match the longest in a batch. Required for matrix operations. Padding waste is the main inefficiency in static batching.
max_batch_size
Maximum number of concurrent requests the server will process together. A key tuning parameter - controls the latency-throughput trade-off.
batch_timeout
Maximum time the server waits for a batch to fill before processing a smaller one. Prevents latency spikes when traffic is low.
GPU Slot
One position in the active batch - one concurrent request the GPU is processing. Continuous batching keeps all slots occupied at all times.
QUICK CHECK
A customer's inference server uses static batching with a batch size of 8. They are seeing GPU utilisation of 15–25% even during peak traffic, and the team is frustrated that adding more requests to the queue does not speed up throughput. What is the most likely cause, and what should they do?
A. Their GPU is undersized - they need to upgrade to an H100 to handle the workload
B. Static batching is the bottleneck - requests with variable output lengths are finishing at different times, leaving idle GPU slots while the batch waits for the slowest request. Switching to continuous batching would fill those slots immediately with waiting requests and likely increase GPU utilisation to 70–90%+ without any hardware change
C. The batch size of 8 is too large - reducing it to 2 will improve GPU utilisation
D. The KV Cache is filling up - they need more GPU memory to serve more requests
