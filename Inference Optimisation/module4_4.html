4.5 - How It All Fits Together

In Module 4.3 the TensorRT-LLM deployment workflow ended with a step labelled "serve via Triton Inference Server" - and then moved on. This module fills that gap. Triton is the production serving layer that sits in front of TRT-LLM (and other backends), handling everything the inference engine itself does not: HTTP and gRPC endpoints, dynamic batching, request queuing, health checks, metrics, and multi-model routing.

Triton is frequently misunderstood in customer conversations. Some teams think it is just an API wrapper. Others think it is only relevant for TRT-LLM. Neither is accurate - Triton is a full inference serving platform that can front multiple backends simultaneously, orchestrate multi-step pipelines, and expose a standardised interface regardless of what runs underneath. Understanding it well lets you place it correctly in architecture discussions and recognise when it is the right layer to add.

1 What Triton Is - The Serving Layer vs the Inference Engine
▾

The distinction that matters most: TensorRT-LLM is an inference engine - it executes model computations on GPUs as fast as possible. Triton Inference Server is a serving layer - it handles everything else required to make that engine accessible to production applications at scale. The two are designed to work together, but they solve different problems.

Triton is backend-agnostic. The same Triton server can simultaneously serve a TRT-LLM engine on GPU, an ONNX model for a smaller classification task, and a Python-based pre/post-processing step - all addressable through a single endpoint with unified metrics.
Why TRT-LLM cannot serve itself: A compiled TRT-LLM engine is a GPU binary that accepts tensor inputs and returns tensor outputs. It has no concept of HTTP, JSON, streaming, request queuing, or health checks. Triton provides all of that - it translates incoming API requests into the tensor format TRT-LLM expects, manages the request queue, and streams token outputs back to the client. Without Triton (or an equivalent serving layer), TRT-LLM engines are not directly usable by production applications.
2 The Model Repository - How Triton Knows What to Serve
▾

Triton loads models from a directory structure called the model repository. Each subdirectory is one model, containing a config.pbtxt configuration file and versioned model artifacts. Triton watches this directory at runtime - adding a new model subdirectory hot-loads it without restarting the server. This is how teams manage model versioning and A/B deployments in Triton.

Model repository structure - click a scenario to explore
config.pbtxt is the critical file: Every Triton model requires a config.pbtxt that declares the model's name, backend type, input tensor shapes and dtypes, output tensor shapes and dtypes, batching configuration, and instance count. Misconfigured shapes are the most common Triton deployment error - if the config declares a different input shape than the model expects, requests will fail with cryptic tensor dimension errors. Always validate config against the actual model's expected input format.
3 Batching in Triton - Dynamic Batching and Sequence Batching
▾

Triton implements its own batching layer on top of whatever backend it fronts. For LLM serving with TRT-LLM, the relevant batching mode is sequence batching with in-flight batching - Triton manages the queue of incoming requests and feeds them to TRT-LLM's in-flight batching engine. For other model types, Triton's dynamic batching groups requests from different clients into batches automatically.

Triton batching modes - click to explore
The queue_delay_microseconds parameter: In dynamic batching mode, Triton waits up to queue_delay_microseconds for additional requests before dispatching a batch. Setting this to 0 dispatches immediately (lowest latency, smaller batches, lower throughput). Setting it to 1000μs (1ms) allows more requests to accumulate (slightly higher latency, larger batches, higher throughput). This is the Triton-level equivalent of the batch_timeout parameter discussed in Module 2.3 - it is the primary latency-throughput dial for non-LLM models served through Triton.
4 Ensemble Pipelines - Chaining Models in Triton
▾

One of Triton's most powerful features for production LLM deployments is ensemble scheduling - the ability to define multi-step pipelines where the output of one model feeds into the next, all managed by Triton without returning to the client between steps. This is particularly valuable for RAG and agentic workloads where preprocessing, embedding, retrieval, and generation all need to happen in sequence.

Triton manages the tensor passing between steps, handles errors at any stage, and streams the final output back to the client. Each component step can be independently scaled - the embedding model might run 4 concurrent instances while the LLM runs 1, because embedding is much faster and can handle more simultaneous requests per GPU.

Why this matters for agentic workloads: As customers move toward multi-step agentic workflows (plan → retrieve → generate → verify → refine), Triton's ensemble capability provides a structured way to orchestrate those steps with proper error handling, metrics per stage, and independent scaling. The alternative - orchestrating each step in application code - is harder to monitor, harder to scale, and introduces more network round-trips. Triton ensembles keep the heavy tensor data on-server between steps.
5 Metrics and Observability - What Triton Exposes Out of the Box
▾

Triton exposes a Prometheus-compatible metrics endpoint at /metrics out of the box. For production LLM deployments, these metrics are the primary source of truth for capacity planning, SLA monitoring, and debugging throughput issues. Knowing what Triton reports - and which metrics map to the concepts from Parts 1–3 - is essential for customer conversations about observability.

The metrics that matter most for LLM SLA monitoring: nv_inference_queue_duration_us maps to queueing latency - if this spikes, requests are waiting in Triton's queue before even reaching TRT-LLM, which means the serving layer is under-provisioned. nv_trt_llm_kv_cache_utilization is the KV Cache fill percentage - sustained values above 90% indicate you are approaching the OOM boundary described in Module 2.5. nv_trt_llm_generation_requests_gen_tput is your actual tokens/second throughput, the primary capacity metric.
Triton's health endpoint for load balancers: Triton exposes /v2/health/ready and /v2/health/live endpoints that return HTTP 200 when the server is ready to accept requests. These are what load balancers and orchestration systems (Kubernetes readiness probes) should poll - not custom health checks. A common deployment mistake is configuring Kubernetes to check a non-existent endpoint, causing the pod to be marked unhealthy immediately after startup and triggering an infinite restart loop.
6 Triton vs vLLM's Built-in API - When Does Triton Add Value?
▾

vLLM has its own built-in OpenAI-compatible HTTP server - so a natural question in customer conversations is: if we use vLLM, do we also need Triton? The answer depends on the deployment's complexity.

The honest answer for most vLLM customers: If a team is running vLLM with one model and needs an OpenAI-compatible endpoint, vLLM's built-in server is sufficient and adding Triton adds complexity without benefit. Triton becomes valuable when: (1) they move to TRT-LLM and need a serving layer, (2) they need to serve multiple models from one endpoint, (3) they need pipeline orchestration for RAG or agentic workloads, or (4) their organisation requires gRPC or KServe v2 protocol for integration with existing infrastructure.
7 4 Key Ideas to Remember
▾
01
Triton is the serving layer - TRT-LLM is the inference engine. They solve different problems and both are required for a TRT-LLM production deployment.
TRT-LLM executes GPU computation as fast as possible. Triton handles HTTP/gRPC endpoints, request queuing, batching, health checks, and metrics. A TRT-LLM engine without Triton is a GPU binary with no way to receive production traffic. Triton without TRT-LLM is a serving platform with no model to execute.
02
The model repository is directory-based and hot-reloadable - model versioning and A/B deployments are managed by dropping files into the right directories
Each subdirectory in the model repository is a model. Numbered subdirectories inside are versions (1/, 2/, 3/). Triton loads all versions and can route traffic to specific ones. Adding a new version directory hot-loads it without restarting. This is how production teams do zero-downtime model updates with Triton.
03
Ensemble pipelines keep tensor data on-server between steps - critical for RAG and agentic workflows where network round-trips between steps would add latency
A Triton ensemble defines the full pipeline (tokenise → embed → retrieve → generate → detokenise) in a config file. Triton executes all steps server-side, passing tensors directly between steps without serialisation or network overhead. Each step can be independently scaled. This is the production-grade alternative to orchestrating multi-step workflows in application code.
04
Triton adds value for TRT-LLM, multi-model, and pipeline workloads - for simple vLLM deployments its built-in server is sufficient
Adding Triton to a simple vLLM deployment is adding complexity without benefit. Triton earns its place when TRT-LLM is in the stack (mandatory), when multiple models need a unified endpoint, when pipelines need on-server orchestration, or when the organisation needs gRPC or KServe v2 protocol. Match the tool to the actual requirement.
MODULE 4.4 - CHEAT SHEET
Triton Inference Server
NVIDIA's production serving layer. Handles HTTP/gRPC, batching, queuing, health checks, metrics, multi-model routing, ensemble pipelines. Backend-agnostic - works with TRT-LLM, ONNX, PyTorch, Python.
Model repository
Directory structure Triton loads models from. Each subdirectory = one model. Versioned with numbered subdirs (1/, 2/). Hot-reloadable - add a directory to deploy a new model without restart.
config.pbtxt
Required config file per model. Declares backend type, input/output tensor shapes and dtypes, batching config, instance count. Misconfigured shapes = most common Triton deployment error.
Dynamic batching
Triton groups requests from different clients into batches automatically. queue_delay_microseconds sets how long to wait for more requests before dispatching. Key latency-throughput dial for non-LLM models.
Sequence batching
Batching mode for stateful models like LLMs - maintains state across steps of a generation. Used with TRT-LLM's in-flight batching. Triton manages the sequence queue; TRT-LLM manages the KV Cache.
Ensemble pipeline
Chain of models executed server-side with tensor passing between steps. No client round-trips. Defined in a config.pbtxt with an ensemble scheduler. Critical for RAG and agentic pipeline deployments.
Key metrics
nv_inference_queue_duration_us (queue wait time), nv_trt_llm_kv_cache_utilization (KV fill %), nv_trt_llm_generation_requests_gen_tput (tokens/sec). Exposed at /metrics (Prometheus format).
Health endpoints
/v2/health/ready and /v2/health/live. Use for Kubernetes readiness/liveness probes and load balancer health checks. Common mistake: checking wrong endpoint, causing infinite restart loops at deploy time.
QUICK CHECK
A customer is planning a production RAG deployment: an embedding model (BGE-M3, ONNX) runs on one GPU to embed queries, results are retrieved from a vector database, and Llama 3 70B (TRT-LLM compiled engine) runs on four H100s to generate answers. Their engineering team wants to expose a single API endpoint to their application developers and get per-stage latency metrics. What is the right Triton configuration approach, and what is the key operational concern to flag?
A. Run two separate Triton servers - one for the embedding model and one for the LLM - and have the application orchestrate calls between them with the vector database lookup in between.
B. Use a single Triton server with a model repository containing three models: the BGE-M3 ONNX model, a Python backend model for the vector database lookup, and the TRT-LLM engine for Llama 3 70B. Define a Triton ensemble that chains them in sequence - embedding output feeds the retrieval step, retrieval results feed the LLM prompt construction, LLM output streams back to the client through a postprocessing step. One endpoint, per-stage Prometheus metrics from Triton, independent instance scaling per step. The key operational concern: the Python backend step for vector database retrieval introduces a network call inside the Triton server's execution thread - if the vector database response is slow or times out, it blocks that ensemble slot. Set aggressive timeouts on the retrieval step and monitor queue_duration_us for the retrieval model separately to catch vector database latency before it cascades into LLM queue buildup.
C. Use vLLM instead of TRT-LLM - vLLM's built-in OpenAI-compatible server can handle all three steps and does not require Triton configuration.
D. Use Triton for the LLM only and handle the embedding and retrieval steps in the application layer - Triton ensemble pipelines are designed for GPU-to-GPU tensor passing and are not appropriate for steps that include external network calls.
