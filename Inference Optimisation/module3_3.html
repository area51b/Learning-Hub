Quantisation - INT8, FP8, INT4

Using smaller numbers to represent model weights reduces memory, increases throughput, and enables larger models on the same hardware - with carefully managed quality trade-offs.

~10 min read Concept level

Every number stored in a neural network takes up space. A standard FP16 weight uses 16 bits - two bytes. Quantisation compresses those numbers into fewer bits: INT8 uses 8 bits, FP8 uses 8 bits with a different layout, INT4 uses only 4 bits. Halving the bit-width halves the memory required and, because Decode is memory-bandwidth-bound, roughly doubles throughput.

The catch: fewer bits means less precision, and less precision means some loss of model quality. The art of quantisation is choosing the right format for the right layer so quality loss is acceptable - ideally imperceptible. Modern quantisation methods (GPTQ, AWQ, SmoothQuant, FP8) have made this trade-off so favourable that INT8 is now nearly lossless on most tasks, and FP8 is the new standard for H100 deployments.

1 What Quantisation Actually Does
▾

A model weight is just a number - typically a decimal like 0.3847291. In FP16 (16-bit floating point), that number is stored using 16 bits, giving very fine precision. Quantisation maps those precise values to a smaller set of representable values using fewer bits.

Think of it like rounding: instead of storing 0.3847291, you store 0.38 - close enough for most purposes, but using less space. The question quantisation research is answering is: how do you choose what to round to, so the rounding errors cancel out rather than compound across hundreds of billions of weights?

The critical insight: Quantisation does not change the model architecture - the number of layers, attention heads, or parameters stays identical. It only changes the precision of the stored values. The computational graph is the same; the numbers flowing through it are slightly less precise. Most of the time, for most tasks, the model cannot tell the difference.

FP8 vs INT8 - why both exist at 8 bits: INT8 stores integers (whole numbers from -128 to 127). FP8 stores floating-point values with an exponent, giving it a much wider dynamic range - it can represent both very large and very small numbers accurately. For neural network weights, which span a large dynamic range, FP8 typically achieves lower quality loss than INT8 at the same bit-width. H100 GPUs have dedicated FP8 Tensor Cores that make FP8 operations faster than INT8 on that hardware.

4 Quality Trade-offs - What Actually Degrades and When
▾

Quality loss from quantisation is not uniform - it depends on the task and the model size. Larger models are more robust to quantisation (they have redundant capacity to absorb precision loss). Harder tasks like multi-step reasoning are more sensitive than simple classification or summarisation.

Relative quality retention by precision and task - select a task type
Layers are not equally sensitive: Embeddings and often the final output projection tend to hurt more from aggressive rounding than many middle blocks. Methods like GPTQ (layer-wise error minimisation) and AWQ (activation-aware scaling) reduce damage in those fragile places while still using one low-bit format across the model. Some deployments also keep a few layers in FP16- that is a separate mixed-precision choice, not required by GPTQ or AWQ themselves.
The mandatory benchmark: Quality loss from quantisation is task-dependent and cannot be fully predicted from benchmarks alone. Before deploying INT4 (or even INT8 for critical applications), always run your actual task against your actual evaluation set. A 1% drop on MMLU may be acceptable; a 1% drop on your specific code generation or medical reasoning task may not be. Never skip this step.
5 Quantisation Methods - How It Is Actually Done
▾

Not all quantisation is equal. Format (INT8, INT4, FP8) is which number type you store; method is the recipe for choosing scales, rounding, and which weights matter most. The same INT8 weights can look very different depending on whether you used naive rounding or GPTQ. RTN, GPTQ, AWQ, and SmoothQuant are post-training recipes: start from an FP16/BF16 trained model, run a small calibration set, compress-no full retraining. The FP8 (native) row is different: it is about doing math in FP8 on supported GPUs, not a named calibration algorithm like GPTQ.

Reading the Speed column: For RTN through SmoothQuant, Speed means how long it takes to produce quantized weights (offline PTQ). For FP8 (native), Speed means inference matmul throughput on supported GPUs versus FP16-not the same measure as the rows above.

The practical recommendation matrix: On A100, prefer INT8 (SmoothQuant for full W8A8 stacks, or GPTQ for weight-only INT8 in many open-source flows). On H100/H200, prefer FP8 for speed and quality when your stack supports it; fall back to INT8 where FP8 is unavailable. For edge or severe memory limits, consider AWQ INT4 after benchmarking. Treat naive round-to-nearest INT4 as experimental only-not a production default.
6 KV Cache Quantisation - A Separate Lever
▾

Quantisation applies to two separate things in inference: the model weights, and the KV Cache. These are independent decisions with different trade-offs - and it is common to quantise them differently.

Weight Quantisation
Reduces model size in memory
Improves throughput (less memory to read)
Quality-sensitive - benchmark carefully
Done once at model load time
KV Cache Quantisation
Reduces KV Cache memory per token
Increases max concurrent requests
Less quality-sensitive than weights
Applied dynamically during serving
Why KV Cache is less sensitive to quantisation: Model weights are used in every forward pass - errors in weights compound over many layers and tokens. KV Cache values are read once per decode step and used for attention scoring - small errors in individual KV values tend to average out across hundreds of attention queries. This is why INT8 KV Cache quantisation is nearly lossless in practice even when INT8 weight quantisation might show some degradation on hard tasks. vLLM supports KV Cache quantisation with --kv-cache-dtype fp8.
7 4 Key Ideas to Remember
▾
01
Halving bit-width halves memory and roughly doubles throughput
FP16 to INT8 cuts weight memory in half. Because Decode is memory-bandwidth-bound, reading half the data in the same time roughly doubles how many tokens per second the GPU produces. This is the most direct path to throughput improvement short of adding hardware.
02
INT8 is nearly lossless; INT4 requires task-specific validation
Modern INT8 methods (SmoothQuant, GPTQ) achieve quality within 0.5–1% of FP16 on almost all tasks. INT4 can be 1–5%+ degraded on hard tasks and should always be validated on your specific workload before production deployment. FP8 on H100 is now the best balance of speed and quality.
03
Weight quantisation and KV Cache quantisation are independent decisions
A common and effective configuration: run FP16 weights (full quality) with INT8 KV Cache (doubles concurrency with negligible quality loss). Or INT8 weights with INT8 KV Cache for maximum throughput. Each lever can be turned independently based on where the constraint is.
04
FP8 on H100 is now the recommended default - not a compromise
The H100's hardware-native FP8 Tensor Cores, combined with the wider dynamic range of floating-point representation, make FP8 the best available option for H100/H200 deployments - not a trade-off. It is faster than FP16, uses half the memory, and has quality loss in the 0.1–0.5% range that is imperceptible on most tasks.

Quantisation
Storing model weights or KV Cache in fewer bits. Reduces memory, increases throughput. Does not change model architecture.
FP16 → INT8

8-bit floating point with hardware-native H100/H200 support. Better quality than INT8 at same bit-width. ~2× faster than FP16. Best option for H100 deployments.
INT4
4 bits per weight - 4× memory reduction vs FP16. 1–5%+ quality drop on hard tasks. Use for edge/memory-critical deployments. Always benchmark first.
GPTQ / AWQ
Post-training quantisation methods that minimise quality loss using calibration data. AWQ is better at INT4; both are better than naive rounding.
SmoothQuant
INT8 quantisation method that transfers difficulty from activations to weights. Near-lossless at INT8. Used in TensorRT-LLM.
KV Cache quantisation
Independent from weight quantisation. INT8 KV roughly doubles concurrency with negligible quality loss. Less sensitive than weight quantisation.
Recommendation matrix
A100: INT8 weights + INT8 KV. H100/H200: FP8 weights + FP8 KV. Edge: AWQ INT4. Always benchmark quality on your task before deploying INT4.
QUICK CHECK
A customer running Llama 3 70B on two A100 80GB GPUs wants to increase concurrent users without adding hardware. They are currently using FP16. What is your recommendation, and what should they expect?
A. Switch to INT4 - it gives the maximum memory savings and will quadruple concurrent users.
B. Switch model weights to INT8 (SmoothQuant or GPTQ) and KV Cache to INT8 quantisation. Weight INT8 frees ~70GB of the 160GB total - freeing headroom for more concurrent KV Caches. INT8 KV then roughly doubles KV capacity per GB. Together, this should increase concurrent users 3–4× with less than 1% quality degradation on most tasks, and no hardware change.
C. Quantisation will not help - the bottleneck is the number of attention heads, not memory.
D. Switch to FP8 - it is the best option available and will solve the problem on A100s.
