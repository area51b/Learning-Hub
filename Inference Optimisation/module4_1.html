1 GPU vs CPU - Why LLMs Need GPUs
▾

A CPU is designed for sequential, complex logic - a small number of powerful cores (8–128) that can each handle branching code, OS scheduling, and general-purpose tasks. A GPU is designed for parallel arithmetic - thousands of simpler cores (H100: 16,896 CUDA cores) that all execute the same operation simultaneously on different data.

LLM inference is almost entirely matrix multiplication - multiplying activations by weight matrices across hundreds of layers. Matrix multiplication is perfectly parallel: every output element is an independent dot product. A GPU can compute thousands of these simultaneously; a CPU must compute them one after another.

CPU - built for general logic
8–128 powerful cores
Complex branch prediction
Large cache per core (MB)
Clock: 3–5 GHz
FP32 throughput: ~1–5 TFLOPS
Best at: OS, branching logic, sequential code
GPU - built for parallel math
16,896+ simple CUDA cores
Thousands run same instruction
Small shared cache (SRAM)
Clock: 1.5–2 GHz
FP16 throughput: ~1,979 TFLOPS (H100)
Best at: matrix multiply, attention, FFN
The practical implication: Running a 70B LLM on a CPU would take minutes per token. The same model on a single A100 80GB produces tokens in milliseconds. The difference is not software - it is the fundamental architecture. Every serving framework (vLLM, TensorRT-LLM, Triton) is ultimately orchestrating GPU matrix multiplications. Understanding the GPU is understanding why the software works the way it does.
2 CUDA - The Software Layer Everything Runs On
▾

CUDA (Compute Unified Device Architecture) is NVIDIA's parallel computing platform - the programming model and runtime that allows software to talk to GPU hardware. Every LLM framework - PyTorch, vLLM, TensorRT-LLM, JAX - compiles down to CUDA kernels that execute on the GPU. CUDA is why all of these tools only work on NVIDIA GPUs.

Why this matters in customer conversations: When a customer says "we're using CUDA 12.4" or "we need to update our CUDA drivers," they are talking about this layer. CUDA version compatibility is a frequent deployment friction point - TensorRT-LLM and vLLM both have minimum CUDA version requirements, and mismatches between the driver version installed by IT and the version the software expects are a common cause of deployment failures. Always ask about CUDA and driver versions early in technical discovery.
Key CUDA libraries for LLMs: cuBLAS handles matrix multiplication (the dominant operation). cuDNN provides optimised attention implementations. NCCL (NVIDIA Collective Communications Library) handles AllReduce operations between GPUs in Tensor Parallelism - when you read about AllReduce in Part 3, NCCL is what actually executes those synchronisations.
3 HBM - Why Memory Bandwidth Is the Bottleneck
▾

HBM (High Bandwidth Memory) is the memory technology used in data-centre GPUs. Unlike DDR5 RAM in a CPU (which sits on a PCIe bus), HBM is stacked directly alongside the GPU die using silicon interposer connections - giving it dramatically higher bandwidth at the cost of high manufacturing complexity and price.

Recall from Module 2.5: Decode is memory-bandwidth-bound. Every token generation requires reading all model weights from HBM into the compute units. A 70B model at FP16 requires reading ~140GB of weights per decode step. At A100 bandwidth (2,000 GB/s), this takes ~70ms - which sets a hard floor on TPOT regardless of all other optimisations. The H100 (3,350 GB/s) does the same read in ~42ms, and the B200 (8,000 GB/s) in ~17ms.

HBM capacity vs HBM bandwidth - two separate specs: HBM has both a capacity (how many GB it holds) and a bandwidth (how fast data moves). The A100 comes in 40GB and 80GB variants - same bandwidth (2,000 GB/s), different capacity. Capacity determines how large a model fits and how many concurrent KV Caches you can hold. Bandwidth determines TPOT for a given model size. When a customer asks "should I get A100 40GB or 80GB?", the answer is almost always 80GB for LLM inference - the capacity difference is critical, the bandwidth is identical.
4 Tensor Cores - What Makes Modern GPUs Fast at Transformers
▾

Standard CUDA cores multiply one number at a time. Tensor Cores - introduced in Volta (V100) and substantially improved each generation - perform a 4×4 matrix multiply-accumulate operation in a single clock cycle. For LLM inference, which is almost entirely matrix multiplication, Tensor Cores are the primary throughput driver.

Tensor Core evolution - peak throughput per GPU
V100 (Volta, 2017)
First Tensor Cores. FP16 only.

A100 (Ampere, 2020)
Added BF16, TF32, INT8, sparsity support.

H100 (Hopper, 2022)
Added FP8, Transformer Engine (auto precision).
1,979 TFLOPS FP16  |  3,958 FP8
B200 (Blackwell, 2024)
FP4 support, 2nd gen Transformer Engine.
~9,000 TFLOPS FP8  |  ~18,000 FP4
TFLOPS = trillion floating point operations per second. Prefill is compute-bound - more TFLOPS directly improves TTFT. Decode is memory-bandwidth-bound - TFLOPS matter less than HBM bandwidth for Decode.
The Transformer Engine (H100+): The H100 introduced a Transformer Engine that automatically selects between FP8 and FP16 precision layer-by-layer during inference, maximising throughput without manual quantisation configuration. This is why FP8 is the recommended default for H100 deployments (Module 3.3) - the hardware handles the precision management automatically, and TensorRT-LLM exposes this through simple configuration flags.
5 NVLink & NVSwitch - The Interconnect That Enables Tensor Parallelism
▾

NVLink is NVIDIA's high-speed GPU-to-GPU interconnect. It is what makes Tensor Parallelism practical within a node - the AllReduce operations between GPUs that happen after every layer require bandwidth that PCIe (the standard CPU-GPU bus) cannot provide at acceptable latency.

NVLink generations - bandwidth and implications

NVSwitch is a dedicated switch chip that provides full all-to-all NVLink connectivity between all GPUs in a DGX node - without NVSwitch, GPU A connecting to GPU H would consume bandwidth on intermediate GPUs. NVSwitch ensures every GPU pair in a node communicates at full NVLink bandwidth simultaneously, which is what makes 8-way Tensor Parallelism within a DGX node practical.

PCIe vs NVLink - the crucial distinction: Cloud instances without NVLink (e.g. some AWS and Azure GPU instances use PCIe between GPUs) have ~64 GB/s bidirectional between GPUs. NVLink H100 provides ~900 GB/s. For Tensor Parallelism, which runs AllReduce after every layer, this 14× bandwidth difference is the difference between TP being net-positive for latency and being net-negative. Always ask customers whether their GPU interconnect is NVLink or PCIe before recommending TP configurations.

▾

These are the four GPUs you will encounter in almost every enterprise LLM inference conversation. Know them cold.

H100 PCIe vs H100 SXM - not the same GPU: NVIDIA sells the H100 in two form factors. SXM (used in DGX H100 servers) has NVLink 4.0 at 900 GB/s and higher TDP (700W). PCIe (used in standard rack servers) has PCIe 5.0 at 64 GB/s and lower TDP (350W). HBM bandwidth is also lower: 3,350 vs 2,000 GB/s. For Tensor Parallelism and high-throughput inference, always ask which form factor the customer has - the SXM variant is significantly more capable for LLM workloads.
7 4 Key Ideas to Remember
▾
01
HBM bandwidth sets the TPOT floor - it is the single most important spec for Decode
Decode reads all model weights from HBM every token. A100 at 2,000 GB/s, H100 at 3,350 GB/s, B200 at 8,000 GB/s - these numbers directly translate to minimum achievable TPOT for a given model size. When a customer asks why their TPOT is what it is, start here.
02
NVLink bandwidth determines whether Tensor Parallelism is viable
TP requires AllReduce after every layer. NVLink H100 (900 GB/s) makes this fast enough to be net-positive for latency. PCIe (64 GB/s) makes it net-negative. Before recommending a TP configuration, confirm NVLink vs PCIe. DGX nodes always have NVLink; cloud "GPU instances" may not.
03
The H100's Transformer Engine makes FP8 the right default - no extra work needed
The Transformer Engine auto-selects FP8 vs FP16 per layer at runtime. TensorRT-LLM exposes this through a single config flag. The quality-vs-speed analysis from Module 3.3 is real, but on H100 the hardware largely handles it. On A100, you need to choose and validate INT8 explicitly.
04
CUDA is the foundation everything builds on - version compatibility is a real deployment risk
vLLM, TensorRT-LLM, PyTorch - all compile to CUDA kernels. CUDA driver version mismatches are among the most common causes of deployment failures in the field. When a customer reports "it won't install" or "the framework crashes on startup," CUDA/driver version incompatibility is the first thing to check.
MODULE 4.1 - CHEAT SHEET
HBM (High Bandwidth Memory)
GPU memory stacked alongside the die. Higher bandwidth than DDR5. A100: 2TB/s, H100: 3.35TB/s, B200: 8TB/s. Bandwidth sets the TPOT floor for Decode.
CUDA
NVIDIA's parallel computing platform. Every LLM framework compiles to CUDA kernels. Version compatibility between CUDA runtime, driver, and framework is a common deployment issue.
Tensor Cores
Specialised matrix multiply units on NVIDIA GPUs. Process 4×4 matrix ops per clock cycle. Primary driver of Prefill (compute-bound) throughput. H100 Tensor Cores support FP8 natively.
Transformer Engine
H100+ feature that automatically selects FP8 vs FP16 per layer at inference time. Makes FP8 the recommended default without manual quantisation work.
NVLink
GPU-to-GPU interconnect within a node. H100: 900 GB/s. Required for Tensor Parallelism - AllReduce after every layer needs this bandwidth. PCIe (64 GB/s) is too slow for TP.
NVSwitch
Switch chip in DGX nodes that provides full all-to-all NVLink connectivity between all GPUs. Enables 8-way TP at full bandwidth without bandwidth sharing across intermediate GPUs.
SXM vs PCIe form factor
H100 SXM: 3,350 GB/s HBM, 900 GB/s NVLink, 700W. H100 PCIe: 2,000 GB/s HBM, 64 GB/s PCIe, 350W. For LLM inference at scale, SXM is substantially superior. Always confirm which form factor.
H200 vs H100
Same Tensor Cores as H100, but 141GB HBM (vs 80GB) and 4,800 GB/s bandwidth (vs 3,350 GB/s). Best current option for 70B models where memory capacity and Decode speed are both constraints.
QUICK CHECK
A customer has a cluster of 8× H100 PCIe GPUs (not SXM) and wants to serve Llama 3 70B with Tensor Parallelism=8 to minimise TPOT. Your colleague recommends this configuration. What concern should you raise?
A. The H100 PCIe does not support FP8, so quantisation will not work on this hardware.
B. H100 PCIe GPUs are connected via PCIe 5.0 (~64 GB/s), not NVLink (~900 GB/s). Tensor Parallelism=8 requires AllReduce communication after every layer across all 8 GPUs - at PCIe bandwidth, this communication overhead will likely dominate the forward pass time and make TP=8 net-negative for TPOT compared to running the model with TP=1 or TP=2. Recommend validating with a benchmark, and consider TP=2 at most, or switching to a configuration with NVLink-connected GPUs if budget allows.
C. 70B is too large for 8× H100 PCIe - the model will not fit in GPU memory.
D. Tensor Parallelism is only supported on A100, not H100.
