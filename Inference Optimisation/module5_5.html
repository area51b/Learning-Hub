3 Full-Course Concept Map - Where Everything Lives
▾

Every concept from the course lives in a specific layer of the production stack. This map is your reference for placing any customer symptom or question into its correct architectural context:

Parts 1–2 - Model internals
Transformer, attention, KV Cache, Prefill vs Decode
These concepts explain why the rest of the stack exists. Prefill is compute-bound because of parallel attention across all tokens. Decode is memory-bandwidth-bound because weights are read in full every step. The KV Cache is why memory fills under load. Flash Attention is why long context is possible.
Modules 1.1–2.5
Part 3 - Optimisations
Flash Attention, Paged Attention, quantisation, parallelism, speculative decoding, disaggregation
Each optimisation targets a specific bottleneck. Flash Attention targets Prefill memory. Paged Attention targets KV fragmentation. Quantisation targets Decode bandwidth. TP targets model memory. Disaggregation targets coupled phase contention. Speculative decoding targets serial token generation at low concurrency.
Modules 3.1–3.6
Part 4 - The single-node stack
NVIDIA hardware, vLLM, TRT-LLM, Triton
Hardware sets the physical limits (HBM bandwidth floor for TPOT, Tensor Core throughput for Prefill, NVLink for TP viability). vLLM and TRT-LLM implement the optimisations. Triton serves them. The per-instance stack is complete and production-grade for most deployments.
Modules 4.1–4.5
Part 5 - Fleet coordination
Routing blindness, KV locality, disaggregation coordination, Dynamo, llm-d
Fleet coordination addresses the limits of the per-instance stack at scale. Routing blindness negates prefix caching. KV locality creates multi-turn inefficiency. Static disaggregation breaks under dynamic traffic. Dynamo and llm-d provide the coordination layer that fixes these problems without replacing the per-instance stack.
Modules 5.1–5.5
The through-line across all five parts: Every concept in this course is ultimately about one of three physical constraints: GPU memory capacity (KV Cache, quantisation, offloading), GPU memory bandwidth (TPOT floor, Decode optimisation, TP for more bandwidth), or GPU compute throughput (Prefill speed, Flash Attention, Tensor Cores). Parts 1–2 explain why these constraints exist at the model level. Parts 3–4 show how to push against them per-instance. Part 5 shows how to operate efficiently when pushing against them across a fleet.
4 The SME Diagnostic Framework - From Symptom to Root Cause in 60 Seconds
▾

An SME's most visible skill in a customer conversation is the ability to hear a symptom and immediately name the layer, component, and concept responsible - and the fix. Here is the complete diagnostic map across every concept in the course:

SYMPTOM: High TTFT that varies widely (200ms to 3s for similar prompts)
Single node: Head-of-line blocking - long Prefill jobs monopolising the GPU (Module 3.6). Fix: chunked Prefill, disaggregation.
Multi-node fleet: Routing blindness destroying prefix cache hit rates (Module 5.1). Fix: Dynamo KV router or llm-d EPP.
SYMPTOM: Slow TPOT at all concurrency levels, even single user
Root cause: HBM bandwidth floor (Module 2.5). TPOT minimum = model bytes / HBM bandwidth. No scheduler or software change can go below this.
Fix: Quantise weights (FP16→FP8 halves bytes read), upgrade GPU (A100→H100→H200→B200), or use smaller model.
SYMPTOM: OOM errors under load despite apparent free GPU memory
Root cause: KV Cache pool exhaustion - not raw GPU memory (Module 2.5). vLLM pre-allocates 90% of HBM; KV Cache fills the pool before raw memory appears full.
Fix: Prefix caching, lower --max-model-len, lower --max-num-seqs, KV Cache FP8 quantisation (doubles pool capacity).
SYMPTOM: TPOT spikes mid-stream - tokens arrive then pause for 200–500ms
Root cause (1): KV Cache preemption - scheduler swapping sequences to CPU RAM as HBM fills (Module 2.5).
Root cause (2): Long Prefill interrupting Decode on coupled server (Module 3.6).
Fix (1): KV Cache quantisation, lower concurrency limits. Fix (2): Disaggregation.
SYMPTOM: Adding GPUs with TP didn't improve (or worsened) latency
Root cause: PCIe interconnect - AllReduce bandwidth (64 GB/s) slower than compute benefit (Module 3.4, 4.1).
Fix: Verify NVLink presence. TP is net-positive only on NVLink-connected GPUs. On PCIe, use Pipeline Parallelism for model-doesn't-fit scenario; use Data Parallelism for throughput scaling.
SYMPTOM: Prefix caching enabled but TTFT still high across fleet
Root cause: Routing blindness at fleet scale - round-robin routing sends requests to replicas without the cache (Module 5.1).
Fix: Dynamo KV-aware router or llm-d EPP. This is the single highest-ROI fix for multi-replica deployments with shared system prompts.
SYMPTOM: Triton 503 at startup / Kubernetes infinite restart loop
Root cause: Kubernetes readiness probe on wrong endpoint or initialDelaySeconds too short (Module 4.4).
Fix: Use /v2/health/ready. Set initialDelaySeconds ≥ 120s for large models. Set failureThreshold ≥ 5 on liveness probe.
5 What's on the Horizon - Concepts That Will Matter Next
▾

The field is moving quickly. These are the developments most likely to surface in customer conversations over the next 6–12 months - and a brief framing for each so you can engage when they come up:

Emerging concepts to watch
Disaggregation at rack scale - GB200 NVL72 as a single disaggregated unit
The GB200 NVL72 connects 72 GPUs with NVLink in a single rack, enabling ~900 GB/s inter-GPU bandwidth across all 72 devices. This makes the entire rack a single disaggregated inference unit where Prefill and Decode workers communicate at NVLink speed - not InfiniBand speed. Dynamo's 30x number is specifically this configuration. As customers upgrade to Blackwell, this architecture becomes the reference design for high-throughput reasoning model serving.
Speculative decoding at scale - draft models co-located in disaggregated fleets
Module 3.5 noted that speculative decoding degrades at high concurrency. Disaggregated serving opens a new configuration: dedicated small draft model workers (Llama 3 8B) that generate candidate tokens in parallel for multiple Decode workers simultaneously, reducing the overhead that makes speculative decoding impractical at high concurrency. Early results suggest this can recover the TPOT benefits of speculative decoding without the concurrency penalty.
Multi-modal inference - KV Cache for vision tokens adds a new memory dimension
Vision-language models (GPT-4V successors, Llama 3.2 Vision, Gemma 3) process image tokens alongside text tokens. Image tokens can be 1,000–4,000 tokens per image. KV Cache for vision tokens follows the same principles as text, but with different caching opportunities: a document with multiple images across a long session may have vision KV blocks that are even more expensive to recompute than text. Prefix caching for shared image tokens is an active area of development in both vLLM and Dynamo.
LoRA adapter serving at fleet scale - routing by adapter, not just by model
Many enterprise customers fine-tune base models with LoRA adapters for specific use cases. Serving dozens of LoRA adapters on a shared base model requires the routing layer to be adapter-aware - routing requests to workers that already have the right adapter loaded, to avoid expensive adapter swaps. llm-d v0.5 added cache-aware LoRA routing (Module 5.3). This is likely to become a standard feature of both frameworks as LoRA-based customisation becomes the dominant enterprise deployment pattern.
Heterogeneous memory pooling - CPU RAM and NVMe as first-class KV Cache tiers
Both Dynamo and llm-d now support KV Cache offload to CPU RAM and NVMe. The next frontier is making this tiering transparent and automatic at fleet scale - a distributed KV Cache that spans all GPUs in a fleet plus their CPU RAM, managed by a single block manager. This turns the entire fleet's aggregate CPU memory (potentially petabytes across hundreds of nodes) into a KV Cache tier that dramatically reduces recomputation for long-context and multi-session workloads.
6 The Course in Five Sentences
▾

If you had to explain everything in this course to someone who had 90 seconds to listen, these five sentences carry the essential knowledge:

Part 1–2
LLMs have two fundamentally different phases: Prefill (compute-bound, parallel, fast) and Decode (memory-bandwidth-bound, serial, slow) - and the KV Cache is the memory structure that connects them, growing with every token generated.
Part 3
Flash Attention makes long-context Prefill possible, Paged Attention and prefix caching make KV Cache memory efficient and reusable, quantisation lowers the TPOT floor and fits larger models, and disaggregation lets each phase run at its independent optimum.
Part 4
GPU HBM bandwidth sets the hard TPOT floor, NVLink determines whether tensor parallelism helps or hurts, vLLM is the right default for most teams, TRT-LLM earns its complexity at fixed-model high-scale deployments, and Triton serves both.
Part 5
At fleet scale, round-robin routing destroys prefix caching, static disaggregation configuration breaks under dynamic traffic, and heterogeneous workloads need specialised pools - Dynamo and llm-d solve these problems as a fleet coordination layer above the existing inference engines.
The meta-principle
Every performance problem in LLM inference is ultimately about one of three physical constraints - GPU memory capacity, GPU memory bandwidth, or GPU compute throughput - and the right fix is always the one that directly addresses the constraint that is actually binding.
MODULE 5.5 - FULL COURSE INTEGRATION CHEAT SHEET
The three physical constraints
All LLM inference problems trace to: GPU memory capacity (OOM, KV size), GPU memory bandwidth (TPOT floor, Decode speed), or GPU compute throughput (Prefill speed, Tensor Core utilisation).
Prefill vs Decode in one line
Prefill: all tokens in parallel, compute-bound, determines TTFT. Decode: one token per step, memory-bandwidth-bound, determines TPOT. Everything else follows from this.
Optimisation decision order
1. Right GPU. 2. Right inference engine. 3. Quantisation. 4. Parallelism. 5. Prefix caching. 6. Disaggregation. 7. Fleet coordination (KV router, Planner). Never skip ahead.
Prefix caching caveat
Prefix caching only works if requests reach the replica with the cached blocks. At multi-replica scale, a KV-aware router (Dynamo or llm-d EPP) is a prerequisite for prefix caching to deliver its benefits.
Fleet coordination threshold
~16+ GPUs, multiple replicas, shared system prompts, disaggregated P/D, or mixed workload types. Below this, the Part 4 stack is right. Above this, Dynamo or llm-d.
Dynamo vs llm-d in one line
Dynamo: NVIDIA-optimised, centralised Planner, TRT-LLM native, maximum Blackwell performance. llm-d: Kubernetes-native, hardware-agnostic, sidecar disaggregation, vLLM primary. Converging via NIXL and shared KV components.
The fastest single improvement
For multi-replica deployments with shared system prompts: deploy a KV-aware router. Restores prefix cache hit rates from ~5% to 85–95%. Often delivers 3–4x TTFT improvement with no other changes.
The meta-answer when uncertain
Identify the binding constraint (memory capacity / bandwidth / compute), confirm the scale (single node vs fleet), match to the layer that owns it, recommend the minimum change that addresses that specific constraint.
FINAL INTEGRATION CHECK - PARTS 1–5
A fintech customer runs Llama 3 70B on 16x H100 SXM GPUs across 2 nodes (8 GPUs each, TP=4 per node, 2 replicas). They use vLLM with FP8 quantisation, Triton in front, and round-robin routing. Their workload is a customer-facing chatbot with a 2,500-token shared compliance prompt. At peak (500 concurrent users), they see: TTFT averaging 1.9s, TPOT degrading to 38ms (SLA: 25ms), and OOM errors on roughly 5% of sessions. Their engineering team has strong Kubernetes expertise and wants to fix all three symptoms. Rank the interventions in the correct order and identify which frameworks, if any, they need.
A. Deploy Dynamo immediately for all three problems - it handles routing, TPOT via disaggregation, and KV Cache pressure via the Block Manager in one integrated system.
B. The three symptoms have three distinct root causes requiring fixes in order. (1) TTFT 1.9s: routing blindness - round-robin across 2 replicas means the 2,500-token compliance prefix cache hit rate is ~50% at best. Fix: deploy llm-d EPP scheduler (Kubernetes-native, fits team expertise) to restore cache hit rates to 90%+, dropping TTFT to under 150ms. (2) TPOT 38ms: the FP8 TPOT floor for 70B on H100 is ~10ms, so 38ms indicates KV Cache pressure forcing preemption - sequences are being swapped as HBM fills at 500 concurrent users. Fix: enable FP8 KV Cache quantisation to double pool capacity; also lower --max-num-seqs to prevent overcommitting. (3) OOM errors: same root cause as (2) - KV Cache pool exhaustion, not raw GPU memory. The FP8 KV Cache fix resolves both (2) and (3) together. No disaggregation is needed yet: with only 2 replicas and the above fixes, headroom is restored. Revisit disaggregation if TPOT SLA is still missed after KV Cache quantisation.
C. Upgrade to INT4 quantisation to further reduce model size and fix TPOT, then add more replicas to fix TTFT variance.
D. Migrate from vLLM to TRT-LLM - the compiled engine will deliver better TPOT and handle higher concurrency before OOM errors occur.
