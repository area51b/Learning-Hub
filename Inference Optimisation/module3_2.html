Paged Attention

How vLLM borrowed the OS concept of virtual memory to eliminate KV Cache fragmentation - and why it was the single biggest throughput advance in LLM serving.

~10 min read Concept level

Before Paged Attention, LLM inference servers wasted up to 80% of their KV Cache memory - not because they ran out, but because of fragmentation. Memory was reserved in large contiguous blocks per request, and chunks sat unused as requests finished at different lengths. The result: GPUs that looked full were actually mostly empty.

Paged Attention, introduced in the vLLM paper (Kwon et al., 2023), solved this by borrowing a 50-year-old idea from operating systems: virtual memory paging. Instead of one big contiguous block per request, KV Cache is stored in small fixed-size pages that can be scattered anywhere in GPU memory. The result was a 2–4× throughput improvement on the same hardware, with no quality change. It is now the standard approach in every major inference server.

1 The Problem - KV Cache Fragmentation
▾

In a naive inference server, when a request arrives the system reserves a contiguous block of GPU memory for its entire potential KV Cache - sized for the maximum possible output length. This creates three types of waste:

Internal fragmentation
A request reserved space for 2,048 tokens but only generated 200. The remaining 1,848 token-slots sit empty but blocked - no other request can use them.
External fragmentation
After several requests finish, freed blocks are scattered across memory. A new large request cannot fit even though total free space is sufficient - there is no single contiguous block big enough.
Over-reservation
Because output length is unknown at request start, the server reserves space for the worst case. Most requests generate far less - so the reserved-but-unused space accumulates across the entire batch.

The scale of the waste: The original vLLM paper measured fragmentation in real serving scenarios and found that naive KV Cache allocation wasted 60–80% of available GPU memory. A GPU that looked 90% full was actually only 10–30% productively utilised. This is why adding more GPU memory alone does not fix the problem - the waste scales with the memory you add.
2 The OS Analogy - Virtual Memory Paging
▾

Operating systems solved an identical fragmentation problem for RAM decades ago with virtual memory paging. Every CS textbook covers it. The vLLM authors applied the same concept to GPU KV Cache.

OS Virtual Memory
Each process sees a clean, contiguous virtual address space
RAM is divided into fixed-size pages (e.g. 4KB)
A page table maps virtual pages to physical RAM pages - which can be scattered anywhere
Processes cannot tell their memory is non-contiguous - the OS handles the mapping transparently
Paged Attention
Each request sees a clean, contiguous logical KV sequence
GPU memory is divided into fixed-size KV blocks (e.g. 16 tokens each)
A block table maps logical KV positions to physical GPU memory blocks - scattered anywhere
The attention kernel handles non-contiguous lookups transparently - output is identical
Why this is powerful: A request that generates 47 tokens needs exactly 3 physical blocks (at 16 tokens/block). It wastes at most 1 token-slot in the last partial block - not hundreds. When the request finishes, all 3 blocks are immediately available for any future request. There is no fragmentation beyond one partial block per request.

Req A's blocks (2, 7, 11) are scattered - but each request's block table records the mapping. The attention kernel follows the table to find KV vectors regardless of physical location. Free blocks (3, 5, 6, 8, 10, 12) are immediately available for any new request.
Allocation on demand: Unlike naive allocation that reserves a full block upfront, Paged Attention allocates blocks one at a time as tokens are generated. A request that will eventually produce 500 tokens starts with zero blocks reserved - blocks are added as each group of 16 tokens is generated. This means unused "future" space is never reserved, eliminating over-reservation entirely.
4 Copy-on-Write - Sharing Pages Across Requests
▾

Because blocks are now independently addressable, multiple requests can point their block tables to the same physical block - as long as that block's content is identical. This unlocks two powerful capabilities: prefix caching and beam search memory sharing.

All three requests read from these shared blocks freely - no copies are made. Copy-on-write triggers only if a request needs to write new data into a shared block, for example during beam search when two beams diverge. At that point the system makes a private copy of just that block for the writing request, leaving the other request's view of the block untouched.

This is exactly how prefix caching works in vLLM. A system prompt that is 6,000 tokens long would normally require every request to recompute and store those 6,000 tokens' KV values. With Paged Attention and prefix caching, that KV data is stored once - in a set of physical blocks - and every subsequent request that shares that prefix simply gets pointers to the same blocks. TTFT drops from seconds to milliseconds for those requests. For prefix caching - where shared blocks are never modified after Prefill - the copy never triggers and the sharing costs nothing.

Beam search benefit: Beam search generates multiple candidate continuations from a shared prefix. With naive allocation, each beam gets its own full copy of the shared tokens. With Paged Attention copy-on-write, all beams share the same physical blocks for their common prefix - only diverging tokens get new blocks. Memory use for beam search drops by the beam width factor.

Paged Attention is the core architectural innovation in vLLM. TensorRT-LLM adopted an equivalent block-based memory manager. SGLang extends it further with RadixAttention for more sophisticated prefix sharing. Every modern high-throughput inference server uses some variant of this approach.

When customers ask why vLLM is faster than naive serving: The short answer is Paged Attention plus continuous batching. Continuous batching keeps GPU slots filled. Paged Attention keeps GPU memory efficiently used. Together, they transform the same hardware from serving 5–10 concurrent requests to serving 50–100+. That is the core of vLLM's value proposition relative to a basic inference loop.
The one limitation: Paged Attention adds a small overhead per decode step - the attention kernel must follow block table pointers to find KV data scattered in memory, rather than reading a single contiguous array. On very short sequences or tiny batch sizes, this indirection can add measurable latency. In practice, for any workload where throughput matters, the fragmentation savings vastly outweigh this overhead.
6 4 Key Ideas to Remember
▾
01
Naive KV Cache allocation wastes 60–80% of GPU memory through fragmentation
Reserving contiguous blocks sized for worst-case output leaves most of that space empty but blocked. A GPU that appears 90% full is often only 10–30% productively used. This is the problem Paged Attention was built to solve.
02
Paged Attention stores KV Cache in small fixed-size blocks - anywhere in GPU memory
A block table per request maps logical token positions to physical GPU blocks. Blocks can be scattered across all of HBM, allocated on demand as tokens are generated, and freed immediately when a request completes. Internal fragmentation is reduced to at most one partial block per request.
03
Copy-on-write enables prefix sharing - the foundation of prefix caching
Multiple requests can point to the same physical blocks for any shared prefix. This makes prefix caching possible: a 6,000-token system prompt stored once in GPU memory can be shared by thousands of requests. TTFT for those requests drops from seconds to milliseconds.
04
Paged Attention is why vLLM exists - and why every serious inference server uses it
The 2–4× throughput improvement vs prior systems with no hardware change was the vLLM paper's central result. Combined with continuous batching, it defined what modern LLM inference serving looks like. TensorRT-LLM, SGLang, and others all implement equivalent block-based KV managers.
MODULE 3.2 - CHEAT SHEET
Paged Attention
KV Cache management system that stores KV data in fixed-size non-contiguous blocks. Eliminates fragmentation. Core innovation in vLLM (2023).
KV Block
Fixed-size chunk of GPU memory holding KV vectors for a set number of tokens (e.g. 16). Allocated on demand, freed immediately on request completion.
Block Table
Per-request lookup structure mapping logical token positions to physical GPU memory blocks. Equivalent to a page table in OS virtual memory.
Internal Fragmentation
Wasted space inside a reserved block - e.g. reserving 2,048 tokens but using 200. Paged Attention reduces this to at most one partial block per request.
External Fragmentation
Free memory scattered in unusable small chunks between allocated blocks. Paged Attention eliminates this - any free block can satisfy any allocation.
Copy-on-Write
Multiple requests share the same physical blocks for identical content. A private copy is made only when a request modifies its content. Enables prefix sharing.
Prefix Caching
Storing computed KV blocks for a shared prefix (e.g. system prompt) once, then sharing them via block tables across all requests with that prefix. Drops TTFT by orders of magnitude for prefix-heavy workloads.
RadixAttention
SGLang's extension of Paged Attention - uses a radix tree to share KV blocks across arbitrary common prefixes, not just fixed system prompts. More general-purpose prefix sharing.
QUICK CHECK
A customer's vLLM deployment serves a customer support chatbot. Every request starts with the same 4,000-token system prompt describing the product and tone guidelines. They are seeing TTFT of 1.8 seconds and want to cut it below 300ms without changing hardware. What do you recommend, and why will it work?
A. Reduce the system prompt to under 500 tokens - long system prompts are inherently slow to process.
B. Enable prefix caching in vLLM. The 4,000-token system prompt will be computed once and its KV blocks stored in GPU memory. Every subsequent request with the same prefix will skip Prefill for those 4,000 tokens entirely - loading pre-computed KV blocks instead. TTFT will drop from ~1.8s to under 100ms for cached requests, with no hardware change and no quality impact.
C. Switch from vLLM to TensorRT-LLM - it handles system prompts more efficiently by default.
D. Enable Flash Attention - it reduces Prefill time for long sequences like system prompts.
