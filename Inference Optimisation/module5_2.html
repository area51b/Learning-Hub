NVIDIA Dynamo - Fleet-Scale Disaggregated Inference

Announced at GTC 2025, Dynamo is NVIDIA's answer to every scaling problem from Module 5.1 - a distributed inference framework built to coordinate fleets of GPU workers with KV Cache-aware routing, dynamic disaggregation, and NIXL-accelerated transfer.

NVIDIA Dynamo was announced at GTC March 2025 as the successor to Triton Inference Server - a ground-up redesign for the era of large-scale, multi-node LLM inference. Where Triton unified per-node model serving across frameworks, Dynamo introduces the fleet-level coordination layer that was absent from the Part 4 stack: KV-aware routing, disaggregated Prefill/Decode scheduling, standardised KV transfer, and tiered memory management. It wraps existing inference engines - TRT-LLM, vLLM, and SGLang - without replacing them, and is implemented in Rust for performance with Python integration for extensibility. Triton continues under the name Dynamo-Triton for teams that need its broad multi-framework, multi-hardware coverage.

NVIDIA's published benchmarks show two distinct performance tiers. On Blackwell-based GB200 NVL72 hardware running DeepSeek-R1 671B, Dynamo's disaggregated serving delivered up to 30x throughput improvement - a figure that reflects both the new hardware's architecture and Dynamo's orchestration working together and should not be generalised to other configurations. On existing Hopper-class hardware running Llama 70B with vLLM, Dynamo more than doubled throughput - a more representative baseline for customers on current infrastructure. This module explains the four mechanisms behind those gains - the Smart Router, the Planner, NIXL, and the KV Block Manager - and maps each one back to the structural gaps identified in Module 5.1.

1 What Dynamo Is - And What It Is Not
▾

The most important framing to establish first: Dynamo is not a new inference engine. It does not replace TRT-LLM or vLLM. It sits above them as a coordination and orchestration layer - a fleet-level scheduler that fills the gap identified in the current stack. Conceptually, you can think of it as providing for a multi-node fleet what Triton provides for a single node, but with deep awareness of LLM-specific concerns like KV Cache state and Prefill/Decode phase separation.

The Triton relationship: Dynamo is positioned as a fleet-scale coordination layer that complements Triton, not as a universal replacement. Existing Triton deployments continue to be supported under NVIDIA AI Enterprise, and Triton remains the per-node inference server in many architectures. Dynamo becomes the right choice to introduce when deployments grow into multi-node or rack-scale fleets, especially where disaggregation, multi-node serving, or KV Cache-aware routing are required. Teams running single-node Triton + TRT-LLM can continue with that pattern; Dynamo is an additional layer for when fleet-scale coordination becomes the bottleneck.

3 The KV Cache-Aware Router - How It Actually Works
▾

The router is the component that directly addresses Module 5.1's routing blindness problem. It operates by maintaining a real-time registry of which prefix hashes each worker currently has cached, then using that registry to make routing decisions that maximise cache hits while still balancing load.

When a request arrives, Dynamo's router computes a hash of the prompt prefix - typically the system prompt portion that is shared across many requests. It looks up that hash in its registry, finds which workers currently have that block cached, checks those workers' current load and memory pressure, and routes to the best candidate. If no worker has a cache hit, it routes to the least-loaded worker and updates the registry once that worker builds the cache.

KV Cache-aware router in action - compare routing modes

The router operates in two modes: basic mode (simple load balancing, no cache awareness) and KV-aware mode (full prefix hash tracking). KV-aware mode adds a small overhead to each routing decision - typically under 1ms - which is negligible compared to the TTFT savings from cache hits. The registry is updated asynchronously as workers build and evict cache blocks, so routing decisions are always based on near-real-time cache state.

The impact on prefix caching: In Module 3.2 we showed an example where prefix caching dropped TTFT from around 1-2 seconds to well under 200ms for a cached multi‑thousand‑token system prompt on Hopper GPUs. In Module 5.1 we saw how round-robin routing can negate this benefit at fleet scale, driving cache hit rates toward zero as replicas increase. With Dynamo’s KV-aware router, prefix cache hit rates can be restored into the 80-90%+ range for workloads that genuinely reuse a common prefix - effectively making prefix caching behave as intended even with many replicas. In practical customer tests, this routing change alone frequently delivers multi‑x TTFT improvements before any other Dynamo feature is adopted, assuming the underlying engine is already well tuned.
4 Disaggregated Serving - Dynamic P/D with NIXL Transfer
▾

Module 3.6 covered the concept of Prefill/Decode disaggregation. Module 5.1 identified that the operational coordination problem - dynamic ratio adjustment, standardised KV transfer - was the gap. Dynamo provides both. It coordinates dedicated Prefill and Decode worker pools through its Planner, and transfers KV Cache between them via NIXL (NVIDIA Inference Xfer Library), a purpose-built low-latency data transfer library.

NIXL uses GPU-aware, low-copy transfer paths (including GPUDirect RDMA where available) to move KV Cache blocks between Prefill and Decode workers with minimal CPU involvement. The effective interconnect bandwidth and the size of the KV block together determine how much latency the handoff adds, and therefore whether disaggregation is net beneficial. On NVLink within a node or rack, the handoff overhead is usually negligible compared to the hundreds of milliseconds of Prefill time it allows you to isolate. On modern InfiniBand fabrics, tens of milliseconds of additional latency are often acceptable for long-context or reasoning workloads. On standard Ethernet, KV transfers for large contexts can easily be slower than simply recomputing Prefill locally, which is why disaggregation over Ethernet is generally not recommended for production.

The Planner component continuously monitors the Prefill and Decode pools and adjusts the ratio dynamically. If Prefill queue depth rises (long prompts accumulating), the Planner can spin up additional Prefill workers from an idle pool or reassign Decode workers temporarily. If Decode demand spikes (many concurrent generations), it shifts the balance the other way. This dynamic adjustment is the operational capability that teams building manual disaggregation with static configurations have always had to build by hand.

Different TP for Prefill and Decode: One important capability Dynamo can unlock, when supported by the underlying engine and model configuration, is using different tensor parallelism settings for the two phases. Prefill is typically more compute-bound and can benefit from smaller TP (less AllReduce and communication overhead), while Decode is more memory-bandwidth-bound and can benefit from larger TP (more HBM pooled across GPUs). In a single, tightly coupled server you normally choose one TP setting for both phases. With Dynamo-style disaggregation and compatible engines (for example, specific TRT-LLM configurations), Prefill workers can run at a different TP than Decode workers - allowing each phase to be tuned closer to its own optimum.
5 KV Cache Offloading - Extending GPU Memory to Storage Tiers
▾

Beyond inter-GPU transfer, NIXL also enables KV Cache offloading to lower memory tiers - CPU RAM, NVMe SSD, or even object storage (S3, PowerScale). This addresses a different problem from the routing and disaggregation features: the GPU memory wall for long-context and multi-turn workloads.

In a multi-turn conversation, KV Cache for previous turns accumulates. A 20-turn conversation with 500 tokens per turn has 10,000 tokens of history - and all of it must be in GPU HBM to generate the next response. For a deployment serving many concurrent long-context users, this quickly exhausts GPU memory. The traditional solution is to evict old KV blocks and recompute them (Prefill) when needed - which is expensive.

Dynamo's KV Block Manager moves blocks between these tiers automatically based on access recency and memory pressure. Frequently accessed blocks stay in GPU HBM. Blocks from earlier conversation turns move to CPU RAM. Very old blocks that might still be needed can spill to NVMe or object storage. When a block is needed and is not in GPU HBM, NIXL (together with the surrounding storage stack) can fetch it back asynchronously. In vendor tests with integrated storage platforms, keeping KV blocks in a warm tier instead of recomputing them has translated into significantly higher effective GPU utilisation and large TTFT reductions for long-context, multi-turn workloads - especially when the alternative is repeatedly re-running long Prefill stages.

When KV offloading matters most: Multi-turn conversational agents, long-context document analysis, and agentic workflows where the same document is referenced across many tool calls. A customer running a legal document review agent where each conversation references a 50,000-token contract across 30+ turns is the canonical use case. Without KV offloading, each turn after the GPU cache fills pays full Prefill cost on the entire document. With KV offloading, the document's KV blocks persist in CPU RAM or NVMe and are restored to GPU HBM on demand - eliminating the recomputation cost.
6 Performance Numbers - What Is Real and What Is Context-Dependent
▾

The “up to 30x throughput” claim needs careful context to be useful in customer conversations. It comes from a specific configuration: a very large reasoning model (for example, a DeepSeek-R1–class MoE with long thinking chains) running on a GB200 NVL72 rack-scale system (Blackwell GPUs with 72 devices in a single high‑bandwidth NVLink domain), compared against a more conventional Hopper-based baseline. Under those conditions, both the new hardware and Dynamo’s disaggregation and routing capabilities contribute to the gain. Disaggregation is maximally effective because long thinking chains make the output length extremely large – Decode dominates, and separating Prefill onto dedicated GPUs prevents Prefill workload from interrupting the Decode pool.

Realistic throughput improvement by scenario - Dynamo vs vLLM/TRT-LLM without Dynamo
The honest framing for customers on H100: For most enterprise customers on H100 hardware running production LLM workloads, Dynamo’s KV-aware router on top of a well-tuned engine can often deliver multi‑x TTFT improvements by restoring prefix cache hit rates where there is real prefix reuse. When fleet size and workloads justify it, adding disaggregation with a suitable interconnect (typically InfiniBand or NVLink) can unlock further throughput gains. In practice, a well-configured Dynamo deployment on Hopper should be positioned as a “3–6x improvement is achievable in the right workloads and configurations”, not as a universal 30x promise. The largest published numbers require both next‑generation Blackwell hardware and carefully chosen reasoning workloads with ideal traffic patterns.

Planner / Scheduler

NIXL (NVIDIA Inference Xfer Library)
Zero-copy GPU-to-GPU KV Cache transfer library. Uses GPUDirect RDMA. NVLink: ~2ms (same node), InfiniBand: ~20-40ms (cross-node), Ethernet: not viable. Also handles CPU RAM and NVMe offload tiers.
KV Block Manager
Fleet-wide KV Cache block registry. Tracks which worker holds which blocks. Manages tiered offload: GPU HBM (hot) → CPU RAM (warm) → NVMe/object storage (cold). Eliminates KV recomputation for long-context multi-turn.
30x throughput claim

Different TP per phase
Disaggregation enables different tensor parallelism for Prefill (lower TP, less AllReduce) vs Decode (higher TP, more HBM bandwidth pooled). Not possible in coupled single-server deployments.

16+ GPU fleet with multiple replicas, high prefix cache reuse (shared system prompts), disaggregated serving needed, mixed workload types, or multi-turn / long-context agentic workloads with KV memory pressure.

They have a RAG application where every request includes the same 6,000-token document index as a system prompt - so prefix caching is enabled. They use a standard load balancer with round-robin routing. TTFT is averaging 2.1 seconds and they cannot explain why, since prefix caching should bring it under 200ms. A colleague suggests they also consider full Dynamo disaggregation with dedicated Prefill and Decode pools. Which Dynamo adoption sequence gives the highest ROI most quickly, and what is the key infrastructure prerequisite to check before committing to full disaggregation?
A. Deploy full Dynamo disaggregation immediately - dedicated Prefill and Decode pools with NIXL transfer is the right end-state, so there is no reason to stage the adoption.
B. Replace the round-robin load balancer with Dynamo's KV Cache-aware router as the first step - this alone will restore prefix cache hit rates from near zero to 80-95% and drop TTFT from 2.1s to under 200ms. C. The 2.1 second TTFT is caused by insufficient quantisation - upgrading from FP8 to INT4 will reduce model size further and drop TTFT. Dynamo does not address TTFT directly.
D. Increase the number of replicas to 16 - more replicas means more instances with warm prefix caches and higher probability of a cache hit per request.
