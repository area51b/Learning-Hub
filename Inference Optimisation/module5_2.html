NVIDIA Dynamo - Fleet-Scale Disaggregated Inference

Announced at GTC 2025, Dynamo is the successor to Triton Inference Server - a ground-up redesign for multi-node LLM inference that closes every gap identified in Module 5.1 with KV-aware routing, dynamic disaggregation, and NIXL-accelerated transfer.

NVIDIA Dynamo was announced at GTC March 2025 as the successor to Triton Inference Server - a ground-up redesign for the era of large-scale, multi-node LLM inference. Where Triton unified per-node model serving across frameworks, Dynamo introduces the fleet-level coordination layer that was absent from the Part 4 stack: KV-aware routing, disaggregated Prefill/Decode scheduling, standardised KV transfer, and tiered memory management. It wraps existing inference engines - TRT-LLM, vLLM, and SGLang - without replacing them, and is implemented in Rust for performance with Python integration for extensibility. Triton continues under the name Dynamo-Triton for teams that need its broad multi-framework, multi-hardware coverage.

NVIDIA's published benchmarks show two distinct performance tiers. On Blackwell-based GB200 NVL72 hardware running DeepSeek-R1 671B, Dynamo's disaggregated serving delivered up to 30x throughput improvement - a figure that reflects both the new hardware's architecture and Dynamo's orchestration working together and should not be generalised to other configurations. On existing Hopper-class hardware running Llama 70B with vLLM, Dynamo more than doubled throughput - a more representative baseline for customers on current infrastructure. This module explains the four mechanisms behind those gains - the Smart Router, the Planner, NIXL, and the KV Block Manager - and maps each one back to the structural gaps identified in Module 5.1.

1 What Dynamo Is - And What It Is Not
▾

The most important framing to establish first: Dynamo is not a new inference engine. It does not replace TRT-LLM or vLLM. It sits above them as a coordination and orchestration layer - a fleet-level scheduler that fills the gap identified in the current stack. Conceptually, you can think of it as providing for a multi-node fleet what Triton provided for a single node, but with deep awareness of LLM-specific concerns like KV Cache state and Prefill/Decode phase separation.

3 The KV Cache-Aware Router - How It Actually Works
▾

The router is the component that directly addresses Module 5.1's routing blindness problem. It operates by maintaining a real-time registry - implemented as a global radix tree - of which prefix hashes each worker currently has cached, then using that registry to make routing decisions that maximise cache hits while still balancing load.

When a request arrives, Dynamo's router computes a hash of the prompt prefix - typically the system prompt portion shared across many requests. It looks up that hash in the radix tree, finds which workers have that block cached, checks those workers' current load and memory pressure, and routes to the best candidate. If no worker has a cache hit, it routes to the least-loaded worker and updates the registry once that worker builds the cache. Workers publish KV cache events asynchronously as blocks are created or evicted, keeping the global registry continuously up to date.

KV Cache-aware router in action - compare routing modes

The router operates in two modes: basic mode (simple load balancing, no cache awareness) and KV-aware mode (full prefix hash tracking via global radix tree). KV-aware mode adds a small overhead to each routing decision - typically under 1ms - which is negligible compared to the TTFT savings from cache hits. In NVIDIA's published data on 100,000 real DeepSeek-R1 user queries, KV-aware routing delivered a 3x improvement in TTFT and a 2x reduction in average request latency compared to load-based routing alone.

The impact on prefix caching: In Module 3.2 we showed that prefix caching can drop TTFT dramatically for a shared multi-thousand-token system prompt. In Module 5.1 we saw how round-robin routing negates this benefit at fleet scale, driving cache hit rates toward zero as replicas increase. With Dynamo's KV-aware router, prefix cache hit rates are restored into the 80–95%+ range for workloads with genuine prefix reuse - effectively making prefix caching behave as intended even across many replicas. In NVIDIA's own testing, the routing change alone delivers a 2x TTFT speedup before any other Dynamo feature is adopted, assuming the underlying engine is already well tuned.
4 Disaggregated Serving - Dynamic P/D with NIXL Transfer
▾

Module 3.6 covered the concept of Prefill/Decode disaggregation. Module 5.1 identified that the operational coordination problem - dynamic ratio adjustment, standardised KV transfer - was the gap. Dynamo provides both. It coordinates dedicated Prefill and Decode worker pools through its SLO Planner, and transfers KV Cache between them via NIXL (NVIDIA Inference Xfer Library), a purpose-built low-latency data transfer library.

NIXL uses GPU-aware, low-copy transfer paths (including GPUDirect RDMA where available) to move KV Cache blocks between Prefill and Decode workers with minimal CPU involvement. The effective interconnect bandwidth and the size of the KV block together determine whether disaggregation is net beneficial. On NVLink the handoff overhead is usually negligible. On modern InfiniBand fabrics, tens of milliseconds of additional latency are often acceptable for long-context or reasoning workloads. On standard Ethernet, KV transfers for large contexts can easily exceed the cost of simply recomputing Prefill locally - which is why disaggregation over Ethernet is not recommended for production.

The SLO Planner component continuously monitors the Prefill and Decode pools and adjusts the ratio dynamically based on queue depth, worker utilisation, and SLO targets (TTFT and inter-token latency). If the Prefill queue backs up - long prompts arriving faster than Prefill workers can process them - the Planner can allocate more workers to Prefill or temporarily shift Decode workers to handle prefill tasks in aggregated mode. If Decode demand spikes, it rebalances the other way. On Hopper-class hardware, NVIDIA's published figures show a single-node disaggregated setup delivers approximately 30% throughput improvement per GPU, scaling to over 2x gains with two-node disaggregated setups (tested with R1 Distilled Llama 70B FP8 using vLLM, 3K ISL / 150 OSL).

Different TP for Prefill and Decode: One important capability Dynamo can unlock, when supported by the underlying engine and model configuration, is using different tensor parallelism settings for the two phases. Prefill is typically more compute-bound and can benefit from smaller TP (less AllReduce communication overhead), while Decode is more memory-bandwidth-bound and can benefit from larger TP (more HBM pooled across GPUs). In a single, tightly coupled server you normally choose one TP setting for both phases. With Dynamo-style disaggregation and compatible engines (for example, specific TRT-LLM configurations), Prefill workers can run at a different TP than Decode workers - allowing each phase to be tuned closer to its own optimum.
5 KV Cache Offloading - Extending GPU Memory to Storage Tiers
▾

Beyond inter-GPU transfer, NIXL also enables KV Cache offloading to lower memory tiers - CPU RAM, NVMe SSD, or even object storage (S3, Azure Blob). This addresses a different problem from the routing and disaggregation features: the GPU memory wall for long-context and multi-turn workloads.

In a multi-turn conversation, KV Cache for previous turns accumulates. A 20-turn conversation with 500 tokens per turn has 10,000 tokens of history - and all of it must be in GPU HBM to generate the next response without recomputing Prefill. For a deployment serving many concurrent long-context users, this quickly exhausts GPU memory. The traditional solution is to evict old KV blocks and recompute them when needed - which is expensive.

Dynamo's KV Block Manager moves blocks between these tiers automatically based on access recency and memory pressure. Frequently accessed blocks stay in GPU HBM. Blocks from earlier conversation turns move to CPU RAM. Very old blocks can spill to NVMe or object storage. When a block is needed and is not in GPU HBM, NIXL fetches it back asynchronously. In NVIDIA's published testing - 10 multi-turn conversations with 80 users, 100K requests using R1 Distilled Llama 70B FP8 on 2 nodes of H100s - CPU memory offloading resulted in a 40% improvement in TTFT compared to inference-engine-level prefix caching alone. This demonstrates that the tiered offload capability adds meaningful benefit on top of what vLLM or SGLang provide natively.

When KV offloading matters most: Multi-turn conversational agents, long-context document analysis, and agentic workflows where the same document is referenced across many tool calls. A customer running a legal document review agent where each conversation references a 50,000-token contract across 30+ turns is the canonical use case. Without KV offloading, each turn after the GPU cache fills pays full Prefill cost on the entire document. With KV offloading to CPU RAM, the document's KV blocks persist in the warm tier and are restored to GPU HBM on demand - eliminating the recomputation cost for the bulk of the context.
6 Performance Numbers - What Is Real and What Is Context-Dependent
▾

The "up to 30x throughput" claim needs careful context to be useful in customer conversations. It comes from a specific configuration: a very large reasoning model (DeepSeek-R1 671B MoE with long thinking chains) running on a GB200 NVL72 rack-scale system (Blackwell GPUs with 72 devices in a single high-bandwidth NVLink domain), compared against a Hopper-based baseline without disaggregation. Under those conditions, both the new hardware architecture and Dynamo's disaggregation capabilities contribute to the gain - the figure should not be attributed to Dynamo's software alone.

Throughput improvement by scenario - Dynamo vs baseline without Dynamo
The honest framing for customers on H100: For most enterprise customers on H100 hardware running production LLM workloads, Dynamo's KV-aware router on top of a well-tuned engine delivers approximately 2x TTFT improvement by restoring prefix cache hit rates where there is real prefix reuse. When fleet size and workloads justify it, adding disaggregation with a suitable interconnect (InfiniBand or NVLink) can unlock further throughput gains - NVIDIA's own Hopper data shows roughly 2x throughput with two-node disaggregated setups. In practice, "2–3x improvement is achievable on Hopper in the right workloads and configurations" is the defensible positioning. The 30x figure requires both next-generation Blackwell hardware and long reasoning-model thinking chains - it is not a general Dynamo software claim.

What Dynamo does not solve - heterogeneous workload routing: Neither Dynamo nor LLM-d provides a feature that classifies incoming requests by workload type (chat vs RAG vs batch) and routes them to purpose-configured worker pools. Dynamo's SLO Planner adjusts P:D ratios for a single deployment based on queue depth - it does not maintain separate pools per request category. Teams handle heterogeneous workloads through infrastructure-level separation: deploying separate model endpoints per workload class and routing at the API gateway above Dynamo. This is a valid architecture, but it is outside what Dynamo itself provides.
What Dynamo also does not solve: Per-instance inference engine performance. If vLLM or TRT-LLM is misconfigured, quantisation is wrong, or the GPU is underpowered for the model, Dynamo does not fix those problems. Dynamo is a fleet coordination layer - it amplifies good per-instance performance across a fleet. It does not substitute for correct engine configuration at the node level.

SLO Planner / Scheduler

KV Block Manager (KVBM)
Fleet-wide KV Cache block registry. Manages tiered offload: GPU HBM (hot) → CPU RAM (warm) → NVMe/object storage (cold). NVIDIA data: 40% TTFT improvement from CPU offload on multi-turn workloads (80 users, 10-turn conversations on H100). Extends effective KV capacity beyond GPU HBM.
Performance numbers to use
Hopper + KV router: ~2x TTFT improvement (sourced). Hopper + two-node disaggregation: ~2x throughput (sourced). GB200 NVL72 + reasoning model: up to 30x (hardware + software combined, not Dynamo alone). Use "2–3x on H100 in the right workload" as the defensible H100 claim.
Different TP per phase
Disaggregation enables different tensor parallelism for Prefill (lower TP, less AllReduce) vs Decode (higher TP, more HBM bandwidth pooled). Not possible in coupled single-server deployments. Engine and model configuration must support it.
When to recommend Dynamo
16+ GPU fleet with multiple replicas, high prefix cache reuse (shared system prompts), disaggregated serving needed, or multi-turn / long-context agentic workloads with KV memory pressure. Not needed for single-node or small deployments - the Part 4 stack is sufficient there.
QUICK CHECK
A customer has 32x H100 SXM GPUs running 8 vLLM replicas of Llama 3 70B FP8 (4 GPUs each, TP=4). They have a RAG application where every request includes the same 6,000-token document index as a system prompt - so prefix caching is enabled. They use a standard load balancer with round-robin routing. TTFT is averaging 2.1 seconds and they cannot explain why, since prefix caching should bring it under 200ms. A colleague suggests they also consider full Dynamo disaggregation with dedicated Prefill and Decode pools. Which Dynamo adoption sequence gives the highest ROI most quickly, and what is the key infrastructure prerequisite to check before committing to full disaggregation?
A. Deploy full Dynamo disaggregation immediately - dedicated Prefill and Decode pools with NIXL transfer is the right end-state, so there is no reason to stage the adoption.
B. Replace the round-robin load balancer with Dynamo's KV Cache-aware router as the first step - this alone will restore prefix cache hit rates from near zero to 80-95% and drop TTFT significantly. The infrastructure prerequisite to check before full disaggregation is interconnect: if the 8 nodes are connected only by standard Ethernet, cross-node NIXL KV transfer bandwidth (~3 GB/s) makes disaggregation impractical. InfiniBand or NVLink (if within one rack) is required for disaggregation to be net-positive. With InfiniBand confirmed, the second phase is enabling Dynamo's SLO Planner with a split Prefill/Decode pool - which delivers further throughput gains for this RAG workload where Prefill is the bottleneck.
C. The 2.1 second TTFT is caused by insufficient quantisation - upgrading from FP8 to INT4 will reduce model size further and drop TTFT. Dynamo does not address TTFT directly.
D. Increase the number of replicas to 16 - more replicas means more instances with warm prefix caches and higher probability of a cache hit per request.
