Disaggregation - Prefill/Decode Split

Separating the Prefill and Decode phases onto dedicated hardware pools - the frontier technique for simultaneously optimising TTFT and TPOT at scale.

~10 min read Concept level

4.1 - vLLM Architecture

Every serving system covered so far runs Prefill and Decode on the same GPU. A user's prompt is processed (Prefill) and then tokens are generated (Decode) on the same hardware, back to back. This seems natural - but it creates a fundamental tension: Prefill is compute-bound and wants large batches of long prompts, while Decode is memory-bandwidth-bound and wants many small ongoing generations. These two workloads have opposite resource profiles, and running them together forces constant compromise.

Disaggregation - sometimes called Prefill/Decode (P/D) separation - solves this by splitting the phases onto separate GPU pools, each tuned for its workload. Prefill GPUs handle only prompt processing; Decode GPUs handle only generation. After Prefill, the computed KV Cache is transferred to a Decode GPU over high-speed interconnect. This is the approach behind DistServe, Mooncake, and the direction NVIDIA's TRT-LLM and vLLM are both moving. It is the current frontier of production inference optimisation.

1 The Fundamental Tension - Why Coupled Serving Is a Compromise
▾

Recall from Module 1.7 that Prefill and Decode have different bottlenecks. Prefill processes all prompt tokens in parallel - it is compute-bound and runs fast. Decode generates one token at a time - it is memory-bandwidth-bound and runs slowly, reading the entire model weight set on every step.

Bottleneck: HBM bandwidth - reading weights and KV

When both phases share the same GPU, every scheduling decision is a trade-off. A long Prefill job blocks the GPU from running Decode steps, causing TPOT to spike for in-flight requests. Batching many Decode requests together improves GPU utilisation for Decode but leaves the GPU under-utilised during Prefill. There is no single configuration that satisfies both phases simultaneously.

The "head-of-line blocking" problem: In a coupled system, a large Prefill (e.g. a 10,000-token document) monopolises the GPU for hundreds of milliseconds. All Decode requests that arrived during that time experience a TPOT spike - tokens stop arriving mid-stream. Users perceive this as the model "pausing." Disaggregation eliminates this entirely by isolating Prefill from Decode.
2 How Disaggregation Works - The Architecture
▾

In a disaggregated system, the serving cluster is divided into two pools. Prefill workers receive incoming requests, run the full forward pass over the prompt, build the KV Cache, and then ship the KV Cache to a Decode worker via high-speed interconnect. The Decode worker takes over and generates tokens, streaming them back to the user.

The KV Cache transfer is the critical link: After Prefill computes the KV Cache, it must be shipped to the Decode worker before generation can begin. This transfer adds latency to TTFT - the KV Cache for a 2,000-token prompt with a 70B model at FP16 is roughly 1–2GB. Over NVLink (~600 GB/s) this takes ~2–3ms. Over InfiniBand (~50 GB/s) it takes ~20–40ms. The transfer latency therefore constrains how far apart Prefill and Decode workers can be placed, and whether the overall TTFT improvement outweighs the transfer cost.

4 KV Cache Transfer - The Mechanics and the Constraint
▾

The KV Cache transfer is what makes disaggregation work - and also what limits it. After Prefill computes the KV tensors, they must reach the Decode worker before a single generation token can be produced. The transfer size and network speed determine whether disaggregation adds or subtracts from TTFT.

KV Cache size formula (from Module 2.2): tokens × KV_heads × layers × head_dim × 2 (K+V) × bytes. For 70B Llama 3 at FP16 with 2,048 tokens: 2048 × 8 × 80 × 128 × 2 × 2 bytes ≈ 1.3GB. At 4,096 tokens this doubles to ~2.6GB. Transfer latency scales linearly with context length.
Prefix caching synergy: Disaggregated systems benefit even more from prefix caching (introduced in Module 2.2). If a system prompt or RAG context is already cached on a Decode worker from a previous request, the Prefill worker only needs to process the new portion - and the transfer is only the delta KV Cache, not the full context. This can reduce both Prefill time and KV transfer size by 70–90% for common system prompts.
The placement constraint: Disaggregation works well within a single node (NVLink KV transfer: ~2–5ms) and between nodes connected by high-bandwidth InfiniBand (HDR/NDR: ~20–40ms for 1–2GB). It does not work across commodity Ethernet - transfer times of hundreds of milliseconds would negate any TTFT benefit. This is why disaggregation is a data-centre architecture pattern, not something deployable on general cloud VMs without specialised networking.
5 The Prefill:Decode Ratio - Tuning the Fleet
▾

Once you separate the pools, you must decide how many Prefill GPUs to allocate relative to Decode GPUs. This ratio is the primary tuning lever in a disaggregated system and depends entirely on the workload's prompt-to-output ratio.

Dynamic ratio adjustment: Production systems like Mooncake (Kimi's serving infrastructure) adjust the P/D ratio dynamically based on real-time queue depths. When Prefill queue is backing up, more GPUs are temporarily allocated to Prefill. When Decode queue backs up, GPUs shift to Decode. This elasticity is a major advantage over coupled systems where the ratio is implicitly fixed at 1:1 by the hardware.

The scale threshold: Disaggregation's complexity overhead is fixed - you pay it once regardless of scale. At small scale (1–4 GPUs), coupled serving is almost always the right choice. At large scale (16+ GPUs serving thousands of requests per minute with mixed workloads), disaggregation's throughput and latency gains compound, and the operational cost amortises quickly. The crossover point is roughly where head-of-line blocking starts appearing regularly in your P95 TTFT metrics.
7 4 Key Ideas to Remember
▾
01
Prefill and Decode have opposing resource profiles - coupling them means perpetual compromise
Prefill is compute-bound and wants large batches. Decode is memory-bandwidth-bound and wants many steady streams. On shared hardware, every scheduling decision helps one phase and hurts the other. Disaggregation ends this conflict by giving each phase its own hardware.
02
The KV Cache transfer is the enabling mechanism - and the key constraint
After Prefill, the computed KV Cache is sent to a Decode worker via NVLink or InfiniBand. Transfer latency adds to TTFT - 2–5ms over NVLink, 20–40ms over InfiniBand. This constrains deployment topology: Prefill and Decode workers must be on the same node or connected by high-bandwidth fabric.
03
The Prefill:Decode ratio is the primary tuning lever - match it to your workload
Long prompts, short outputs (RAG, document Q&A) need more Prefill GPUs. Short prompts, long outputs (code generation, creative writing) need more Decode GPUs. Systems like Mooncake adjust this ratio dynamically in response to real-time queue depths, achieving further efficiency gains over static allocation.
04
Disaggregation is a fleet-scale pattern - complexity must be justified by scale
At small scale, coupled serving is simpler and sufficient. At large scale - thousands of requests per minute, mixed workloads, SLA-sensitive applications - disaggregation's throughput improvement (often 1.5–2.5×) and latency consistency (P99 TTFT and TPOT both improve) justify the operational investment. It is the current frontier of production LLM serving.
MODULE 3.6 - CHEAT SHEET
Disaggregation (P/D split)
Separating Prefill and Decode onto dedicated GPU pools, each optimised for its workload. KV Cache transferred between pools after Prefill completes.
Head-of-line blocking
In coupled serving, a long Prefill job monopolises the GPU and stalls all in-flight Decode requests. Users see TPOT spikes - tokens stop arriving mid-stream.
KV Cache transfer
After Prefill, KV tensors are shipped to the Decode worker. Size scales with context length (~1.3GB per 2k tokens for 70B FP16). NVLink: ~2ms. InfiniBand: ~20–40ms.
Prefill:Decode ratio
The GPU allocation split between pools. Long prompts / short outputs → more Prefill GPUs. Short prompts / long outputs → more Decode GPUs. Adjust dynamically with queue depth.
DistServe / Mooncake
Research systems and production deployments demonstrating disaggregation at scale. Mooncake is Kimi's serving infrastructure; DistServe is from UC Berkeley. Both show 1.5–2.5× throughput gains.
Prefill pool optimisation
Prefill GPUs can be tuned for compute: larger per-batch sizes, Flash Attention, higher MFU. They do not need to hold large KV Caches since KV is transferred out immediately after each request.
Decode pool optimisation
Decode GPUs need large HBM for KV Cache accumulation across many concurrent streams. Paged Attention and prefix caching are critical here. TPOT consistency is the primary SLA metric.
When to use
16+ GPUs, mixed prompt/output lengths, SLA-sensitive workloads, high-BW interconnect available. Not worth the complexity at small scale or on commodity networking.
QUICK CHECK
A customer operates a large-scale RAG application: users submit a question, the system retrieves 5 relevant documents (~3,000 tokens total) and passes them with the question to Llama 3 70B, which produces a 150-token answer. They are seeing inconsistent TTFT (ranging from 200ms to 2,000ms) and occasional TPOT spikes mid-response. They have a 16-GPU H100 cluster with NVLink within each 8-GPU node and InfiniBand between nodes. What is your diagnosis and recommendation?
A. The TPOT spikes are caused by insufficient KV Cache memory - enable INT8 KV quantisation to fit more concurrent requests.
B. This is a classic head-of-line blocking pattern from coupled Prefill/Decode serving. The 3,000-token Prefill jobs are intermittently monopolising GPUs that are also serving Decode streams. Recommendation: implement P/D disaggregation - allocate 4 GPUs (one node) as a dedicated Prefill pool and 12 GPUs (1.5 nodes) as Decode. The Prefill pool handles the long RAG prompts in large batches; the Decode pool maintains steady token streams uninterrupted. KV Cache transfer over InfiniBand (~28ms for ~2GB at 3k tokens) adds to TTFT but eliminates the variance. Also enable prefix caching on the Decode pool - the retrieved documents often share segments, reducing both transfer size and Prefill time.
C. Use speculative decoding - the short 150-token outputs are ideal for it and will fix the TPOT spikes.
D. The inconsistent TTFT is caused by network latency to the retrieval system - optimise the vector database response time.
