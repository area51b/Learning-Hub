5.1 - SME Readiness

Parts 1 through 4 built each concept in isolation: Prefill and Decode, KV Cache, Paged Attention, continuous batching, quantisation, parallelism, Flash Attention, speculative decoding, disaggregation, NVIDIA hardware, vLLM, TensorRT-LLM, and Triton. This module puts all of them together in the context of real serving architectures.

You will walk through a production request from the moment an API call arrives to the moment the last token streams back - touching every component in sequence and seeing exactly where each concept applies. Then you will see three reference architectures for different customer scenarios and the reasoning behind every technology choice. By the end, you should be able to look at any customer's LLM serving setup and immediately map it to these concepts, identify what is missing, and explain why each component is there.

1 The Complete Production Stack - Every Layer in One View
▾

A production LLM serving system has five conceptual layers. Understanding which layer each component lives in - and why - is the foundation of every architecture conversation.

Production LLM serving stack - all components
Layer 1 - Client
Application / API gateway
User-facing app, LangChain agent, REST client. Sends HTTP/gRPC requests. Handles auth, rate limiting, routing to inference endpoint.
Layer 2 - Serving
Triton Inference Server (or vLLM built-in API)
HTTP/gRPC endpoint. Request queuing. Dynamic/sequence batching. Ensemble pipeline orchestration. Prometheus metrics. Health checks. Model versioning.
Layer 3 - Inference engine
TensorRT-LLM engine
Compiled GPU kernels. Kernel fusion. FP8/INT8 execution. In-flight batching (continuous batching). KV Cache management. TP/PP execution.
or
vLLM engine
PyTorch + custom CUDA. Paged Attention. Continuous batching. KV Cache mgmt. OpenAI API built-in.
Layer 4 - Optimisation
Flash Attention
Tiled SRAM attention. Fast Prefill. Enables long context. (Modules 1.3, 3.1)
Paged Attention
Block KV Cache. Near-zero fragmentation. Prefix caching. (Modules 2.2, 3.2)
Quantisation
FP8/INT8/INT4. Smaller weights. More memory for KV Cache. Faster Decode. (Module 3.3)
Speculative decoding
Draft + verify. Fewer large-model passes. Best at low batch sizes / concurrency. (Module 3.5)
Layer 5 - Hardware
NVIDIA GPUs - HBM, Tensor Cores, NVLink
A100 / H100 / H200 / B200. HBM bandwidth sets the Decode TPOT floor. NVLink enables efficient TP. Tensor Cores drive Prefill throughput. (Module 4.1)
Every component in this stack maps to at least one concept from Parts 1-4. A production architecture conversation is really a question of which choices to make at each layer - and the answer always depends on the customer's specific workload, scale, SLAs, and team.
The two paths through the inference engine layer: vLLM and TRT-LLM are not hierarchical - they are alternatives at the same layer. The choice between them determines whether Triton is mandatory (TRT-LLM always needs Triton) or optional (vLLM can use its own built-in API for simple deployments). Everything above and below the inference engine layer is the same regardless of which engine you choose.

5 Symptom-to-Root-Cause Map - Diagnosing Production Issues
▾

In customer conversations, symptoms arrive before causes. Here are the most common production LLM serving symptoms and the root cause and fix that each one points to - drawing on every concept from Parts 1-4:

High TTFT that varies widely (200ms to 3s for similar prompts)
This is head-of-line blocking from coupled Prefill/Decode serving. Long Prefill jobs intermittently monopolise the GPU. Other requests wait. The variance correlates with whether a long-prompt request is running when yours arrives.
Fix: Enable prefix caching for shared system prompts (drops TTFT for cached content). At scale: implement P/D disaggregation (Module 3.6). Intermediate: chunked Prefill in vLLM breaks long Prefills into steps.
TPOT spikes mid-stream - tokens arrive steadily then pause for 200-500ms
Either (1) a large Prefill job interrupted the Decode phase on a shared GPU (coupled serving - same root cause as above, different symptom), or (2) KV Cache pressure forced preemption - the scheduler evicted an in-flight sequence to HBM (swapping) to free GPU memory for a new request.
Fix for (1): same as above. Fix for (2): reduce --max-num-seqs in vLLM, or increase --gpu-memory-utilisation, or quantise the KV Cache (Module 3.3) to fit more sequences before swapping occurs.
OOM errors at 20-30 concurrent users despite GPU having apparent free memory
vLLM pre-allocates its memory pool at startup (for example, 90% of GPU by default). Model weights consume part of that pool; the rest is KV Cache blocks. With 20+ concurrent long-context requests, the KV Cache pool fills before the raw GPU memory appears full. The "free memory" is outside vLLM's pool.
Fix: enable prefix caching (shared prefixes reduce per-request KV blocks), set --max-model-len to a realistic value (reduces block pre-allocation), reduce --max-num-seqs, or quantise KV Cache to FP8/INT8 (2x more sequences fit in same pool).
TPOT is consistently slow regardless of concurrency - even single-user requests are slow
This is the HBM bandwidth floor from Module 2.5. Decode reads all model weights every token. For a given model size and GPU, TPOT has a hard minimum set by (model size in bytes) ÷ (effective HBM bandwidth). No software optimisation can go below this floor - only quantisation (smaller weights) or a faster GPU (more bandwidth) can.
Fix: quantise weights (e.g. FP16 → INT8 → FP8 → INT4, each roughly reducing bytes read and lowering the TPOT floor), upgrade to a higher-bandwidth GPU (A100 → H100 → H200 → B200), or use a smaller model.
Adding more GPUs with TP did not improve latency - sometimes made it worse
The AllReduce communication between TP GPUs takes time proportional to data size and inversely proportional to interconnect bandwidth. On PCIe (for example, 64 GB/s), AllReduce for a 70B model's layer output can take longer than the compute time. Adding more TP GPUs means more AllReduce operations at the same bandwidth, making it worse.
Fix: verify GPU interconnect - NVLink is typically required for TP to be net-positive. On PCIe, use Pipeline Parallelism instead if the model doesn't fit on one GPU. On NVLink, TP=4 or TP=8 is often a good starting point for 70B-class models.
Triton returns 503 at deployment - pod restarts in Kubernetes infinite loop
Almost certainly a Kubernetes readiness probe misconfiguration. The probe is checking a non-existent endpoint and marking the pod unhealthy before it has finished loading the model (model loading can take 30-120 seconds for large models on first start).
Fix: use /v2/health/ready as the readiness probe endpoint with initialDelaySeconds set to a generous value (for example, ≥120s) for large models. Use /v2/health/live for the liveness probe with a longer failureThreshold. Avoid custom health paths - Triton only honours the standard health endpoints.
6 The Decision Sequence - How to Architect from Scratch
▾

When a customer says "we want to self-host an LLM," there is a specific sequence of decisions that leads to the right architecture. Each decision constrains and informs the next:

Decision 1 - Which model?
Model size strongly influences GPU count. As a rough guide: 7B fits comfortably on 1x GPU at FP16. 70B typically needs either a single high-memory GPU (for example, H200 with FP8 or lower precision) or 2x H100 with tensor or pipeline parallelism. 405B may need 4x H200 or 8x H100. Model architecture (GQA, MoE, context window) determines KV Cache size and memory requirements. Nail this before choosing anything else.
Decision 2 - Which GPU?
HBM capacity must fit the model (+ KV Cache budget) at your chosen precision. Effective HBM bandwidth determines the Decode TPOT floor. NVLink (or equivalent high-bandwidth interconnect) determines whether TP is viable. H100 SXM is the current default for many production stacks. H200 is attractive for 70B+ where memory is the main constraint. Check SXM vs PCIe form factor - they are not equivalent.
Decision 3 - Which inference engine?
vLLM by default - faster to deploy, broad model support, OpenAI-compatible. TRT-LLM when: the model is relatively fixed, concurrency is high (for example, >50 users), TPOT SLA is tight (for example, <30ms), and there is a team to manage engine builds. This decision determines whether Triton is mandatory (TRT-LLM) or optional (vLLM).
Decision 4 - Which quantisation?
FP8 on H100/H200 - native Transformer Engine, minimal quality impact, excellent general default. INT8 + SmoothQuant on A100 - well-tested, solid quality. INT4 AWQ only when a model absolutely must fit into a smaller memory footprint and you have validated quality on your workload. Always validate quality after quantising - never assume the theoretical quality loss matches real-world quality loss.
Decision 5 - Which parallelism?
If the model fits on one GPU after quantisation: no parallelism needed. If not: Tensor Parallelism within a node (NVLink or equivalent high-bandwidth interconnect strongly recommended), Pipeline Parallelism across nodes (InfiniBand). For throughput at scale with a model that fits: Data Parallelism (replicate across GPU groups). Avoid recommending TP on PCIe-connected GPUs for latency-sensitive workloads.
Decision 6 - Advanced optimisations?
Prefix caching: always on if any system prompt is reused. Speculative decoding: most useful at low concurrency / smaller batch sizes with predictable outputs; benefits often shrink or become negative at very high concurrency. Disaggregation: only at large scale (for example, 16+ GPUs) with mixed prompt lengths and SLA-sensitive workloads. Add complexity only when simpler optimisations have been exhausted.
The sequence matters: Teams that start at Decision 6 (trying speculative decoding before fixing their KV Cache configuration) waste weeks on optimisations that cannot move the needle while the real bottleneck sits untouched. Starting from hardware and working up ensures each decision is grounded in physical constraints before software tuning begins.
7 4 Key Ideas to Carry Into Part 5
▾
01
Every symptom traces to one of three root causes: memory, bandwidth, or scheduling
OOM is a memory problem. Slow TPOT at low concurrency is a bandwidth problem (HBM floor). Slow TPOT at high concurrency or TTFT variance is a scheduling problem (head-of-line blocking, KV Cache pressure). Identifying the category first narrows the fix immediately.
02
The inference engine choice is the pivotal decision - it determines the rest of the stack
vLLM brings its own API and is self-contained. TRT-LLM typically sits behind Triton and requires an engine build pipeline below it. Choosing TRT-LLM without understanding this cascade is how teams end up with a half-built architecture. The engine choice is not just a performance decision - it is an operational architecture decision.
03
Optimisations compound but must be applied in the right order - fix hardware and engine before tuning software
Quantisation on the wrong GPU yields little. Prefix caching with no repeated prompts yields nothing. Speculative decoding at very high concurrency can hurt throughput. Every technique from Parts 1-3 has a precondition - understanding those preconditions is what separates a recommendation that helps from one that adds complexity with no benefit.
04
The stack is not fixed - it should evolve as the workload evolves
Teams might start with vLLM and one GPU. They grow to TRT-LLM and TP=4. They add prefix caching when system prompts standardise. They consider disaggregation when RAG prompts get long. Good architecture conversations are not about finding the perfect final stack - they are about identifying the right next step for where the customer is today.
MODULE 4.5 - PART 4 INTEGRATION CHEAT SHEET
Decision sequence
Model → GPU → Inference engine → Quantisation → Parallelism → Advanced optimisations. Try not to skip ahead - each decision constrains the next.
TTFT variance
Typical root cause: head-of-line blocking (coupled P/D serving). Fix: prefix caching for shared prompts, chunked Prefill for short term, disaggregation for scale.
TPOT slow at low concurrency
Root cause: HBM bandwidth floor. Fix: quantise weights (reduces bytes read per token), upgrade GPU, or use smaller model. Scheduler changes do not beat the physical floor.
OOM under load
Root cause: KV Cache pool exhaustion (not raw GPU memory). Fix: prefix caching, lower --max-model-len, lower --max-num-seqs, KV Cache quantisation.
TP not helping latency
Root cause: limited interconnect bandwidth (for example, PCIe) makes AllReduce slower than compute. Fix: verify NVLink (or similar) present, or switch to PP for cross-node distribution.
vLLM vs TRT-LLM trigger
TRT-LLM when: relatively fixed model + high concurrency + tight TPOT SLA + experienced team. vLLM for most other cases. TRT-LLM stacks almost always place Triton as the serving layer.
Triton 503 loop
Root cause: K8s readiness probe on wrong endpoint or too-short initialDelay. Fix: use /v2/health/ready, set initialDelaySeconds generously (for example, ≥120s for large models).
Stack evolution path
1 GPU + vLLM → quantisation → multi-GPU TP/PP where needed → TRT-LLM + Triton → prefix caching → disaggregation. Add complexity only when simpler steps are exhausted.
QUICK CHECK - PART 4 INTEGRATION
A customer has the following setup: 8x H100 SXM GPUs, Llama 3 70B at BF16 with TP=4 and PP=2, vLLM, 60 concurrent users at peak. Symptoms: TTFT is acceptable (~400ms), but TPOT degrades from 18ms at 20 users to 45ms at 60 users - far above their 25ms SLA. GPU utilisation shows 85% on the Decode pool during peak. KV Cache utilisation is 88%. Which combination of changes most directly addresses the TPOT degradation?
A. Add 4 more H100s and increase TP to 8 - more GPUs will reduce TPOT by splitting the model across more devices.
B. Enable speculative decoding with Llama 3 8B as the draft model - this reduces the number of large-model forward passes per token and will restore TPOT at high concurrency.
C. Apply FP8 quantisation to weights and KV Cache, then migrate to a TRT-LLM compiled engine. FP8 weights substantially reduce the bytes read per Decode step compared to BF16, lowering the Decode TPOT floor and giving more headroom as concurrency grows. FP8 (or INT8) KV Cache reduces the 88% utilisation, giving more margin before the scheduler is forced to preempt sequences or slow down to manage memory pressure - which is the likely cause of TPOT degrading from 18ms to 45ms as concurrency grows. A TRT-LLM FP8 engine on H100 can often add a further ~20-30% throughput from kernel fusion and native Tensor Core usage.
D. Reduce max concurrent users to 30 via rate limiting - this will bring TPOT back under 25ms by reducing load.
