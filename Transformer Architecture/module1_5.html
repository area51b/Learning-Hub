1.6 — The Full Transformer Block

Every Transformer block has two major components. You have learned about the first — Attention. Now meet the second: the Feed Forward Network (FFN).

While Attention handles relationships between tokens, the FFN handles something different — it processes each token individually, applying deep reasoning to refine what the token means after attention has told it what to focus on. Together, they are the engine of every transformer layer.

This module also introduces Mixture of Experts (MoE) — a modern FFN variant that is reshaping how large models are built and served, and that you will encounter in customer conversations about models like Mixtral and GPT-4.

1 What the FFN Actually Does
▾

After Attention has gathered information from across the sequence, each token now knows what other tokens are relevant to it. But that raw gathered information still needs to be processed and refined. That is what the FFN does.

Think of it in two distinct steps:

1
Expand — look at it from many angles
The token's embedding is projected into a much higher-dimensional space — typically 4x larger. This is like taking a concept and breaking it down into thousands of sub-questions: "Is this technical? Is it a question? Is it positive? Does it relate to infrastructure? Does it require a list response?" The model considers all of these simultaneously.
2
Contract — synthesise into a refined answer
The expanded representation is projected back down to the original embedding size. The model has now "thought through" the token from many angles and produced a refined, richer understanding of it. This output flows to the next transformer layer.
The key difference from Attention: Attention is about relationships between tokens — token A looks at tokens B, C, D. The FFN is about reasoning within a single token — token A thinks deeply about itself after gathering that context. Both are essential. Attention gathers. FFN processes.
2 The Analogy: A Research Team's Workflow
▾

Imagine a research analyst preparing a briefing on a complex topic. Their workflow has two distinct phases:

Phase 1 — Attention (Gathering)
The analyst reaches out to colleagues, reads related reports, and gathers information. They ask: "What do others know about this topic that I need?" This is cross-token, relational work.
Phase 2 — FFN (Processing)
The analyst sits at their desk alone, thinks deeply, synthesises what they gathered, and writes their analysis. This is individual, token-level reasoning — no looking outward, just thinking through what has been collected.

A transformer layer always does these in sequence: Attention first (gather context), then FFN (process it). This pair repeats at every layer, each time producing a richer understanding of the tokens.

Why this matters for inference: The FFN is not just a minor component. In a standard Transformer, the FFN accounts for roughly two-thirds of the total model parameters. Attention is powerful but compact. The FFN is where most of the model's "knowledge" and reasoning capacity is stored — which is also why it is the biggest contributor to model weight size on disk and in GPU memory.
3 FFN vs Attention — Different Bottlenecks
▾

For inference optimisation, it is important to understand that Attention and the FFN create different kinds of bottlenecks. This shapes which techniques fix which problem.

Attention Bottleneck
Scales with sequence length (quadratic)
Bottleneck grows as context gets longer
KV Cache is the main memory burden
Fixed by: Flash Attention, Paged Attention, GQA
FFN Bottleneck
Scales with model size (number of parameters)
Bottleneck is constant regardless of context length
Model weights are the main memory burden
Fixed by: Quantisation, MoE, tensor parallelism
Practical takeaway for customer conversations: When a customer says "our model is slow", the first question is: is it slow because of long contexts (Attention bottleneck) or because the model is just too large for their GPU (FFN / weight bottleneck)? These require different solutions. Understanding the distinction makes you a much more credible technical advisor.
4 How FFN Size Scales with Model Size
▾

The FFN intermediate size (the expanded dimension in Step 1) is typically 4x the embedding dimension. For large models, this adds up quickly:

Model Size Embedding Dim FFN Intermediate % of Total Params Inference Implication
7B 4,096 ~11,008 ~65% FFN Fits on a single A100 80GB in FP16
13B 5,120 ~13,824 ~65% FFN Tight on one A100, comfortable with quantisation
70B 8,192 ~28,672 ~67% FFN Requires multiple GPUs or heavy quantisation
405B 16,384 ~53,248 ~68% FFN Requires large GPU clusters or MoE architecture
Why quantisation targets the FFN first: Since the FFN holds most of the model's weight, compressing it (e.g. from FP16 to INT8 or INT4) yields the biggest memory savings. Quantising just the FFN weights can reduce model size by 50–75% with manageable quality impact. You will learn this in detail in Module 3.3.
5 Mixture of Experts (MoE) — The Modern FFN Evolution
▾

Standard FFNs run the same set of weights for every single token. This is effective but wasteful — a token about "GPU architecture" and a token about "cooking recipes" both go through the exact same weights, even though they need completely different reasoning.

Mixture of Experts (MoE) solves this. Instead of one FFN, it has many smaller FFNs called experts. A lightweight router decides which 1–2 experts are most relevant for each token, and only those experts are activated.

For a technical reasoning token, only Expert 1 (Technical) and Expert 4 (Reasoning) activate. The other 6 experts sit idle — saving compute while keeping quality high.
Why MoE matters for inference: A Mixtral 8x7B model has 8 experts per layer, but only activates 2 per token. It has 47B total parameters but only uses ~13B active parameters per token — giving near-70B quality at 13B compute cost. The catch: all 47B weights still need to fit in GPU memory. This is the key MoE trade-off — high quality at lower compute, but higher memory than a dense model of equivalent active size.
6 4 Key Ideas to Remember
▾
01
Attention gathers. FFN processes.
Every transformer layer does both in sequence. Attention collects relevant context from across the sequence. The FFN then reasons through that context at the individual token level. Neither works well without the other.
02
The FFN holds ~65–70% of a model's parameters
This is why quantisation — compressing weight precision — is so effective. Reducing FFN weight size from FP16 to INT4 cuts model memory requirements by 75%. It is the single biggest lever for fitting large models onto fewer GPUs.
03
FFN bottleneck is about model size, not context length
Unlike Attention (which gets worse with longer sequences), the FFN cost is the same regardless of how long the input is. If a model is slow on short prompts too, the bottleneck is the FFN — likely a GPU memory or compute capacity issue, not a context issue.
04
MoE gives large-model quality at lower active compute cost
By activating only the most relevant experts per token, MoE models like Mixtral achieve high quality with much lower per-token compute. The trade-off is total memory: all experts must be loaded. MoE is ideal when you have GPU memory headroom but want to reduce compute cost.
MODULE 1.5 — CHEAT SHEET
Feed Forward Network (FFN)
Second half of every transformer block. Processes each token individually using a two-step expand-then-contract operation.
FFN Intermediate Size
The expanded dimension in the first FFN step. Typically 4x the embedding dimension. Larger = more reasoning capacity.
Dense FFN
Standard FFN — all parameters activate for every token. Simple but computationally uniform regardless of input type.
Mixture of Experts (MoE)
Multiple smaller FFNs (experts) with a router that selects which ones activate per token. High quality at lower compute cost.
Active Parameters
In MoE: the parameters actually used per token. Much smaller than total parameters. Determines compute cost per token.
Total Parameters
All parameters including idle MoE experts. Determines GPU memory requirement regardless of how many activate.
FFN Bottleneck
Slowness caused by model weight size. Constant regardless of context length. Fixed by quantisation, MoE, or tensor parallelism.
Attention Bottleneck
Slowness caused by long sequences. Grows quadratically with context. Fixed by Flash Attention, GQA, Paged Attention.
QUICK CHECK
A customer is running a Mixtral 8x7B model (47B total parameters, ~13B active per token). They have 80GB of GPU memory but find the model is slow even on very short prompts. What is the most likely explanation and recommended direction?
A. The KV Cache is overflowing — they need to reduce their context window length
B. Even though only 13B parameters are active per token, all 47B parameters must be loaded into GPU memory — if the GPU is near capacity from model weights alone, it will be slow regardless of prompt length. The fix is quantisation or adding more GPU memory.
C. Mixtral does not support short prompts efficiently — they should switch to a dense model
D. The attention mechanism is the bottleneck — Flash Attention should fix it
