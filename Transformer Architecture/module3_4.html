Part 3 — Inference Optimisation Techniques

Tensor & Pipeline Parallelism

How to split a model that is too large — or too slow — for a single GPU across multiple GPUs, and the distinct trade-offs of cutting along different dimensions.

~10 min read Concept level No math

A 405B parameter model in FP16 requires about 810GB of memory — far beyond any single GPU. Even a 70B model at FP16 needs 140GB, which requires two A100 80GB GPUs. Parallelism is how the inference stack spreads a model across multiple GPUs to make large models feasible — and to make smaller models faster by letting multiple GPUs work simultaneously.

There are two primary strategies. Tensor Parallelism splits the work within each layer — every GPU processes part of every forward pass together. Pipeline Parallelism splits the model across layers — each GPU owns a block of consecutive layers and passes activations to the next. Each has different communication costs, latency characteristics, and sweet spots. Most production deployments use both.

1 Why Parallelism — Two Reasons to Spread Across GPUs
▾

There are two separate reasons to run a model across multiple GPUs, and they lead to different parallelism strategies:

Reason 1 — Model does not fit
The model weights alone exceed single-GPU memory. A 70B FP16 model (140GB) cannot fit on one A100 (80GB). You must distribute the weights.
Primary solution: Tensor Parallelism — each GPU holds a shard of every weight matrix.
Reason 2 — Model is too slow
The model fits on one GPU but latency is too high. You want multiple GPUs to share the compute burden so each forward pass completes faster.
Primary solution: Tensor Parallelism within a node (fast interconnect required).
The third reason — more throughput: Even when a model fits and latency is acceptable on one GPU, spreading across more GPUs multiplies throughput — more concurrent requests can be served. This is data parallelism: running multiple independent copies of the model on separate GPU groups, each serving different requests. This is how production clusters at scale work — many parallel model replicas behind a load balancer.
2 Tensor Parallelism — Splitting Within Each Layer
▾

Tensor Parallelism (TP) splits each weight matrix in a layer across multiple GPUs. During a forward pass, every GPU simultaneously computes its portion of the output, then they communicate their partial results to assemble the full answer. This happens at every single layer — the GPUs work together in lock-step on every token.

The NVLink requirement: Tensor Parallelism only makes economic sense within a node where GPUs are connected via NVLink (~600 GB/s bidirectional). Across nodes via InfiniBand (~400 Gb/s = ~50 GB/s), the AllReduce communication overhead at every layer makes TP impractical — the network latency between layers adds up to hundreds of milliseconds per token. This is why TP degree is almost always equal to the number of GPUs per node (typically 8).
3 Pipeline Parallelism — Splitting Across Layers
▾

Pipeline Parallelism (PP) divides the model's layers into consecutive stages, with each stage running on a separate GPU or node. GPU 0 processes layers 1–20, GPU 1 processes layers 21–40, and so on. After GPU 0 finishes its layers, it passes the activations to GPU 1 and can immediately begin processing the next request's layers — like an assembly line.

Each GPU holds only its own layers — no weight matrix needs to span GPUs. Communication is only the activation tensor between stages (~small compared to weight matrices). This makes PP suitable for across-node deployments where interconnect is slower.
The pipeline bubble — PP's key limitation: When processing a single request, only one GPU is active at any time — the others wait. GPU 1 cannot start until GPU 0 finishes. GPU 2 cannot start until GPU 1 finishes. This idle time is called the pipeline bubble. With 4 GPUs and one request, 75% of GPU time is wasted. The bubble shrinks as batch size grows (more microbatches keep all stages busy), but for low-latency single-request scenarios, PP adds significant latency overhead.

Micro-batching — the standard mitigation: Instead of waiting for one request to flow through the entire pipeline before starting the next, micro-batching splits a batch into smaller chunks. GPU 0 processes micro-batch 2 while GPU 1 processes micro-batch 1 — keeping all stages busy most of the time. With enough micro-batches, the bubble becomes a small fraction of total time. This is how GPipe and Megatron-LM handle pipeline parallelism in practice.
5 Combining Strategies — TP + PP + Data Parallelism
▾

Production deployments of large models almost always combine multiple parallelism strategies simultaneously. The standard Megatron-style 3D parallelism uses all three:

Example: Llama 3 405B on 16× H100 nodes (128 GPUs total)
Tensor Parallelism = 8
8 GPUs per node share each layer. NVLink handles AllReduce within each node at full speed.
Pipeline Parallelism = 8
8 node groups each hold a stage of the model. InfiniBand handles activation passing between nodes.
Data Parallelism = 2
Two complete replicas of the TP×PP setup serve different requests independently for higher throughput.
Total GPUs: TP(8) × PP(8) × DP(2) = 128 GPUs. Each serves half the traffic, doubling total throughput.
The rule of thumb: TP within a node (NVLink), PP across nodes (InfiniBand), DP for throughput scaling. TP reduces latency and memory per GPU. PP enables models that span multiple nodes. DP multiplies throughput without reducing latency further. A customer asking "how many GPUs do I need for 405B?" is really asking about this three-way split.

7 4 Key Ideas to Remember
▾
01
Tensor Parallelism reduces latency and fits large models — within a node
TP splits weight matrices across GPUs that work in lock-step. Every forward pass is faster because each GPU does less work. But AllReduce communication at every layer requires NVLink bandwidth — across nodes with InfiniBand, the overhead is too high. TP degree is almost always equal to GPUs per node (typically 8).
02
Pipeline Parallelism scales to very large models across nodes — at a latency cost
PP assigns consecutive layer groups to different GPUs/nodes. Communication is only activations between stages — small enough for InfiniBand. The cost is the pipeline bubble: with a small batch and many PP stages, most GPUs sit idle most of the time. Micro-batching mitigates this but adds complexity.
03
Data Parallelism is the cleanest path to higher throughput
Running multiple independent replicas of the same model (each handling different requests) scales throughput linearly with zero inter-GPU communication during inference. If a model fits within one node's GPUs using TP, adding more nodes for DP is straightforward and efficient. This is how most large-scale serving clusters are built at steady state.
04
Real deployments combine TP + PP + DP — and the split follows the interconnect
The standard pattern: TP within each node (NVLink handles the AllReduce), PP across nodes (InfiniBand handles activation passing), DP for throughput by replicating the whole setup. The parallelism degree at each axis is chosen to match hardware topology. vLLM and TensorRT-LLM both configure all three through simple parameters.
MODULE 3.4 — CHEAT SHEET
Tensor Parallelism (TP)
Splits weight matrices within each layer across GPUs. All GPUs work together on every forward pass. Requires NVLink. Reduces latency and memory per GPU.
Pipeline Parallelism (PP)
Assigns consecutive layer groups to different GPUs. Only activation tensors cross GPU boundaries. Works across nodes. Introduces pipeline bubble idle time.
Data Parallelism (DP)
Multiple complete model replicas, each serving different requests. Zero inter-GPU communication during inference. Linearly scales throughput.
AllReduce
Communication operation where all GPUs share their partial results and each receives the full combined result. Happens after every layer in TP. Requires fast NVLink.
Pipeline Bubble
Idle GPU time in PP while waiting for upstream stages to complete. With batch=1 and PP=4, 75% of GPU time is wasted. Mitigated by micro-batching.
NVLink vs InfiniBand
NVLink: ~600 GB/s within a node — fast enough for TP AllReduce. InfiniBand: ~50 GB/s across nodes — sufficient for PP activation passing, too slow for TP.
3D Parallelism
TP × PP × DP combined. Standard for 100B+ model production deployments. TP within node, PP across nodes, DP for throughput replicas.
TP Degree
Number of GPUs sharing each layer's weights. Typically equals GPUs per node (8 for H100 DGX). Each GPU holds 1/TP of each weight matrix.
QUICK CHECK
A customer wants to deploy Llama 3 405B for a latency-sensitive interactive application. They have access to a cluster of 4 nodes, each with 8 × H100 80GB GPUs (32 GPUs total, NVLink within each node, InfiniBand across nodes). What parallelism configuration do you recommend and why?
A. Use Pipeline Parallelism = 32 across all GPUs — this gives the maximum layer distribution and lowest memory per GPU.
B. TP=8 within each node (NVLink handles AllReduce), PP=4 across the 4 nodes (InfiniBand handles activation passing). This distributes 405B's ~810GB FP16 weight across 32 GPUs (each holds ~25GB of weights) while using the fast NVLink for the frequent TP communication and the slower InfiniBand only for the infrequent inter-stage activation transfers. For a latency-sensitive workload, keep the PP degree low to minimise pipeline bubble.
C. Use Data Parallelism = 32 — run 32 independent model replicas for maximum throughput.
D. Use Tensor Parallelism = 32 across all GPUs — more TP means faster forward passes.
