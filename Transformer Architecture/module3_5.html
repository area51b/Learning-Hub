Speculative Decoding

Using a fast small model to draft candidate tokens that a large model verifies in a single parallel pass — cutting TPOT by 2–4× with mathematically guaranteed identical output.

~10 min read Concept level

Every technique covered so far in Part 3 has targeted memory efficiency or compute distribution. Speculative Decoding attacks something different: the fundamental seriality of the Decode phase. Standard Decode generates exactly one token per forward pass — no matter how fast each pass is, you cannot do two at once. Speculative Decoding breaks this constraint.

The insight: a small, fast "draft" model proposes a sequence of tokens speculatively. The large "target" model then verifies all proposed tokens in a single parallel forward pass — accepting the ones it agrees with and rejecting from the first disagreement. When the draft is mostly right (which it often is for predictable text), you get multiple tokens output for roughly the cost of one large-model forward pass. TPOT drops 2–4× with zero change to output quality — the target model's outputs are statistically identical to what it would have produced without speculation.

1 The Problem Speculative Decoding Solves
▾

Recall from Part 1 that Decode is inherently serial — each token depends on all previous tokens, so you cannot generate token N+1 until token N is complete. This means:

Standard decode — strictly serial
Forward pass 1 → token: "The"
Forward pass 2 → token: "capital"
Forward pass 3 → token: "of"
Forward pass 4 → token: "France"
Forward pass 5 → token: "is"
5 tokens = 5 large model forward passes. Each pass reads all model weights from HBM. No parallelism possible within a single request.

Every other optimisation in Part 3 has made each forward pass faster or more memory-efficient. Speculative Decoding is the first technique that reduces the number of large-model forward passes needed to produce a given number of tokens.

Why this matters for TPOT specifically: Flash Attention, Paged Attention, and quantisation all improve TPOT somewhat by making each forward pass faster or lighter. But even an infinitely fast forward pass still produces only one token — TPOT is bounded by the number of passes. Speculative Decoding is the only technique that breaks this bound by producing multiple tokens per large-model pass.
2 How Speculative Decoding Works — Step by Step
▾
Standard Decoding
1
Large model runs one forward pass → produces 1 token
2
Append token to context
3
Repeat — one full pass per token
N tokens = N large-model forward passes
Speculative Decoding
1
Draft model (small, fast) runs K forward passes → proposes K candidate tokens
2
Target model (large) runs 1 forward pass over all K candidates simultaneously
3
Accept matching tokens from the front; target generates correction at first mismatch
4
Repeat — net output: 1–K tokens per large-model pass
N tokens ≈ N/acceptance_rate large-model passes — often 2–4× fewer
The verification guarantee: The acceptance/rejection algorithm uses a mathematically rigorous rejection sampling scheme. The distribution of tokens accepted by the target model is provably identical to what the target model would have produced without speculative decoding. This is not an approximation. When the target model rejects a draft token, it samples a corrected token from its own distribution and the process restarts. The final output is statistically indistinguishable from standard decoding.

Why acceptance rate varies by content: The draft model is most accurate on predictable, formulaic text — common phrases, code patterns, factual continuations. On creative writing, ambiguous contexts, or domain-specific terminology, it diverges more often from the target. The speedup from speculative decoding is therefore workload-dependent, and measuring acceptance rate on your specific task is essential before deciding to deploy it.
4 Acceptance Rate — The Key Variable
▾

Everything in speculative decoding flows from one number: the acceptance rate — the fraction of draft tokens the target model accepts. A higher acceptance rate means more tokens produced per large-model pass and greater speedup.

Acceptance rate → speedup (draft length K=5)
Low acceptance (creative/novel) High acceptance (predictable/formulaic)
What happens at rejection: When the target model rejects a draft token at position i, all subsequent draft tokens (positions i+1 to K) are also discarded — they were conditioned on the wrong token. The target model generates its own correct token at position i and the process restarts. If the draft model is very inaccurate, the overhead of running K draft passes plus one target verification pass can make speculative decoding slower than standard decoding. Acceptance rate must stay above roughly 50% for a net benefit.
5 Draft Model Options — What to Use as the Drafter
▾
ApproachDraft sourceAcceptance rateComplexityBest for
Small model of same family e.g. Llama 3 8B drafts for Llama 3 70B High (75–90%) Moderate — run two models General text, most production use cases
Self-speculative / MEDUSA Extra draft heads added to the target model — no separate model Medium-high (65–85%) Low — one model, extra heads Memory-constrained deployments
Prompt lookup decoding Copies candidate tokens directly from the prompt — no separate model Task-dependent (high for RAG/summarisation) Very low — zero additional model RAG, summarisation, document Q&A
N-gram / retrieval draft Predicts tokens from n-gram statistics or a retrieval index Lower (40–65%) Low Domain-specific repetitive patterns
The same-family advantage: When the draft model is a smaller version of the same model family — trained with the same tokeniser, vocabulary, and similar data — its token distribution closely matches the target. Llama 3 8B and Llama 3 70B share tokenisation and training distribution. This high alignment translates directly to high acceptance rates, which is why same-family pairing is the default recommendation.
6 When Speculative Decoding Helps — and When It Does Not
▾
GO
Low-to-medium concurrency, latency-sensitive
Speculative decoding shines when few requests share a GPU — extra compute from draft runs is available. Reduces TPOT 2–4×. Best ROI for interactive chat, copilot, and real-time applications where users feel per-token delay.
GO
Predictable, formulaic output
Code completion, structured data extraction, common phrasing, RAG summarisation — tasks where the target model's output is predictable and the draft model achieves 80%+ acceptance. High acceptance rate → high speedup.
NO
High-concurrency throughput-focused serving
When the GPU is fully saturated with many concurrent requests, speculative decoding adds draft model overhead without proportional benefit. At high batch sizes, continuous batching and quantisation outperform it.
NO
Highly creative or unpredictable generation
If the draft model has a low acceptance rate (below ~50%), the overhead of K draft passes plus one verification pass exceeds the cost of just running the target model K times. Measure acceptance rate before deploying. Creative writing and complex reasoning tasks are risky.
The memory overhead caveat: Running both a draft and a target model simultaneously requires enough GPU memory to hold both. Llama 3 8B (draft) + Llama 3 70B (target) at INT8 requires roughly 7GB + 70GB = 77GB — barely fitting on a single A100 80GB. In memory-constrained environments, self-speculative methods (MEDUSA heads) or prompt lookup decoding (zero additional model) are better choices.
7 4 Key Ideas to Remember
▾
01
Speculative decoding targets TPOT — the metric everything else cannot break
Flash Attention, quantisation, and Paged Attention all make each forward pass faster or cheaper — but they still produce one token per pass. Speculative decoding is the only technique that produces multiple tokens per large-model pass, directly reducing TPOT by 2–4× on suitable workloads.
02
Output quality is mathematically guaranteed identical — this is not an approximation
The rejection sampling algorithm ensures the accepted token distribution matches the target model's distribution exactly. A customer asking "will this change the model's outputs?" has a clear answer: no. The only observable difference is that responses arrive faster.
03
Acceptance rate determines everything — measure it before committing
High acceptance (80–90%): 3–4× TPOT improvement, strong ROI. Medium acceptance (60–75%): 1.5–2.5× improvement, still worthwhile. Low acceptance (below 50%): negative ROI, slower than standard decoding. Measure on your actual task and model pairing before deploying.
04
Best at low concurrency — at high batch sizes the benefit inverts
Speculative decoding uses extra GPU compute on draft passes. At low concurrency, that compute is idle anyway — free speedup. At high concurrency, every GPU cycle is needed for requests — draft overhead reduces throughput. Use speculative decoding for latency-sensitive, low-concurrency workloads; rely on batching and quantisation for high-throughput ones.
MODULE 3.5 — CHEAT SHEET
Speculative Decoding
A fast draft model proposes K tokens; the target model verifies all K in one parallel pass, accepting matches and correcting from the first rejection. Output is identical to standard decoding.
Draft Model
The small, fast model that proposes candidate tokens. Best results with a same-family smaller model (e.g. Llama 3 8B drafting for Llama 3 70B).
Target Model
The large model that verifies draft tokens. Its output distribution is mathematically preserved — identical to running it without speculation.
Acceptance Rate
Fraction of draft tokens accepted by the target. Primary determinant of speedup. Must exceed ~50% for net benefit. 80%+ gives 3–4× TPOT improvement.
Draft Length (K)
Number of tokens the draft model proposes per cycle. Typical values: 4–8. Higher K = more potential tokens per target pass, but longer draft overhead if acceptance is low.
MEDUSA / Self-Speculative
Extra prediction heads added to the target model itself — no separate draft model needed. Lower memory overhead. Slightly lower acceptance rate than same-family pairing.
Prompt Lookup Decoding
Drafts candidates by looking them up directly in the prompt. Zero additional model needed. High acceptance for RAG and summarisation where output mirrors input.
When NOT to use
High batch sizes / fully saturated GPU. Low acceptance rate tasks (creative writing, complex reasoning). Memory-constrained setups without MEDUSA or prompt lookup.
QUICK CHECK
A customer runs a legal document review application using Llama 3 70B. The task involves extracting structured fields (dates, parties, obligations) from contracts — highly formulaic, repetitive output patterns. They are serving 2–4 concurrent users at a time and their TPOT of 45ms per token feels too slow for their reviewers. What is your recommendation?
A. Add more GPUs to reduce TPOT — parallelism is the right tool for latency problems.
B. Enable speculative decoding using Llama 3 8B as the draft model. This workload has ideal characteristics: low concurrency (GPU has spare compute for draft passes), highly structured output (high expected acceptance rate on legal field extraction), and latency-sensitivity (users are waiting on TPOT). Expect 2–3× TPOT improvement bringing 45ms down to roughly 15–22ms. Also ensure INT8 quantisation is in place to fit both models on available GPU memory.
C. Speculative decoding will not help because legal text is too complex for a small draft model to predict accurately.
D. Switch to a smaller model like Llama 3 8B — 70B is overkill for structured field extraction.
