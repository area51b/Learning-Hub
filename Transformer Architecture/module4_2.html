vLLM — Architecture, What It Solves, When to Recommend It

The open-source inference server that brought Paged Attention and continuous batching to production. What it actually is under the hood, what it does well, and when TensorRT-LLM is the better answer.

~10 min read Concept level

4.3 — TensorRT-LLM

vLLM is the most widely deployed open-source LLM inference server. Released by UC Berkeley in 2023, it became the de-facto standard for self-hosted LLM serving by solving the KV Cache fragmentation problem that made all prior systems leave 60–80% of GPU memory wasted. It is the first thing most teams reach for, and understanding it well positions you to have credible conversations about when it is the right tool and when something else (TensorRT-LLM, Module 4.3) would serve better.

This module covers what vLLM actually does internally — its architecture, its two core innovations (Paged Attention and continuous batching), its key configuration parameters, and its limitations. By the end you should be able to explain vLLM accurately in a customer discovery call and know which questions to ask to determine whether it is the right fit.

1 What vLLM Is — The Full Architecture Stack
▾

vLLM is an inference server — it sits between your application and the GPU, handling all the complexity of batching, KV Cache management, and scheduling so that the GPU stays utilised and requests are served efficiently. It exposes an OpenAI-compatible REST API, which is why most teams can swap from OpenAI to a self-hosted vLLM endpoint by changing one URL.

vLLM internal architecture — request lifecycle
OpenAI-compatible API
HTTP endpoint (/v1/chat/completions, /v1/completions). Accepts requests, streams tokens back. Drop-in replacement for OpenAI API in most SDKs.
▼
LLM Engine / Scheduler
Core of vLLM. Runs continuous batching — decides each iteration which requests join the batch, which wait, which are preempted. Manages request priority and sequence groups.
▼
Block Manager (Paged Attention)
Manages GPU memory as fixed-size blocks (default 16 tokens/block). Assigns blocks to requests on demand, frees blocks when requests complete. Enables prefix caching and copy-on-write sharing.
▼
Worker / Model Executor
One worker per GPU. Executes the forward pass using PyTorch + custom CUDA kernels. Handles Tensor Parallelism via NCCL AllReduce between workers on same node.
▼
GPU / CUDA runtime
NVIDIA GPU executing CUDA kernels. vLLM uses PyTorch for model execution and custom kernels (Flash Attention, Paged Attention) for attention computation.
vLLM is Python-first with custom CUDA kernels for the performance-critical paths. This makes it highly readable and extensible — a key reason for its rapid community adoption — but also means it carries more Python overhead than TensorRT-LLM.
The OpenAI-compatible API is a deliberate design choice: By mirroring the OpenAI API exactly, vLLM enables zero-code migration from the OpenAI service to self-hosted inference. Any application using the openai Python SDK or making HTTP calls to OpenAI endpoints can point to a vLLM server with one line changed. This is one of vLLM's most powerful adoption drivers in enterprise accounts.
2 Paged Attention — vLLM's Core Innovation
▾

Before vLLM, every inference system allocated KV Cache memory contiguously — one large pre-reserved block per request, sized for the maximum possible output length. A request with a 2,048-token max output reserved 2,048 tokens of KV Cache upfront, even if it only generated 50 tokens. The result: 60–80% of GPU memory sat wasted as internal fragmentation. Paged Attention solves this by managing KV Cache the same way an OS manages RAM — in fixed-size pages allocated on demand.

The OS analogy is exact: Paged Attention maps KV Cache pages to physical GPU memory blocks the same way an OS maps virtual memory pages to physical RAM. A logical sequence of tokens (the request's KV Cache) can occupy scattered, non-contiguous physical blocks. A block table (like a page table) maps logical token positions to physical block addresses. The GPU's attention kernels are rewritten to follow these indirections — which is the custom CUDA work vLLM ships that made this possible.

Paged Attention also enables two features that build on top of block-level management. Prefix caching: if two requests share a common prefix (e.g. the same system prompt), their KV Cache blocks for that prefix are identical — vLLM can reference the same physical blocks for both requests without duplicating data. A 4,000-token system prompt cached once and shared by 100 concurrent users reduces TTFT from ~1.8s to under 100ms for those users. Copy-on-write: beam search and parallel sampling need multiple sequence branches that share a prefix — vLLM shares blocks until a branch diverges, then copies only the diverging blocks, saving memory proportional to the shared prefix length.

3 Continuous Batching — Keeping the GPU Busy
▾

Without continuous batching, requests are grouped into fixed batches that enter and exit together. A 5-request batch stays together even if request 1 finishes after 20 tokens and request 5 needs 500 tokens — the GPU idles waiting for the slowest request before accepting new work. Continuous batching (also called iteration-level scheduling) fixes this by making batching decisions at every single decode step.

5–10× throughput improvement vs static batching: The original vLLM paper (Kwon et al., 2023) demonstrated 2–4× throughput improvement from Paged Attention alone, and an additional factor from continuous batching. Combined, real-world deployments commonly see 5–10× more requests per second versus naive static-batch serving at the same GPU count. This is the primary reason vLLM became the standard so quickly.
4 Key Configuration Parameters — What to Set and Why
▾

vLLM is started from the command line (vllm serve model-name --args) or via Python API. Most defaults are sensible but several parameters have large effects on throughput, latency, and memory usage. Select a workload profile to see the recommended configuration:

vLLM configuration guide by workload
Workload type
GPU
gpu-memory-utilisation is the most important parameter to get right: The default of 0.90 reserves 90% of GPU HBM for vLLM (weights + KV Cache). If other processes share the GPU, lower this. If OOM errors appear under load, lower this to 0.85 or 0.80. If the GPU has spare memory and you want more concurrent KV Cache capacity, raising to 0.95 is reasonable. Never go to 1.0 — CUDA itself needs headroom for its own allocations.
5 vLLM's Limitations — When TensorRT-LLM Wins
▾

vLLM is the right default for most teams, but it has genuine limitations that matter in specific scenarios. Knowing these makes you a more credible advisor than someone who recommends vLLM for everything.

Python overhead limits peak throughput
vLLM's scheduler and most of its logic runs in Python. At very high concurrency (100+ simultaneous requests), Python GIL overhead and scheduling latency can become measurable. TensorRT-LLM's C++ runtime eliminates this. For maximum-throughput batch workloads, TensorRT-LLM typically delivers 20–40% more tokens/second at the same GPU count.
Model compilation is build-time vs runtime
vLLM runs models via PyTorch (with optional torch.compile). TensorRT-LLM compiles models ahead of time into optimised TensorRT engines — fusing operations, selecting optimal kernels per GPU. The compilation step takes time (minutes to hours for large models) but produces faster inference. vLLM's easier startup is a feature in development contexts; in production at fixed scale, compilation pays off.
Quantisation support is improving but uneven
vLLM supports AWQ, GPTQ, and FP8 quantisation, but the implementation maturity varies by model and quantisation type. TensorRT-LLM's quantisation (via NVIDIA's Model Optimizer / AMMO) is more deeply integrated with the hardware and consistently delivers the performance gains that quantisation theory promises. If a customer is trying to squeeze every last token/second out of INT8 or FP8, TensorRT-LLM is the stronger tool.
Where vLLM clearly wins: model breadth and iteration speed
vLLM supports almost every open-weight model within days of release (Llama, Mistral, Qwen, Gemma, Phi, DeepSeek, and hundreds more). TensorRT-LLM takes longer to add new models — each needs a TRT-optimised implementation. If a customer is experimenting with multiple models, switching frequently, or using a less common architecture, vLLM is the right choice. Speed to deployment beats peak throughput efficiency for most teams.
The honest framing for customers: vLLM is the right answer for 80% of teams. If you are in production at scale with a fixed model, high concurrency (>50 simultaneous users), and latency SLAs of under 50ms TPOT, TensorRT-LLM is worth the operational investment. If you are still evaluating models, iterating on prompts, or serving at moderate scale, vLLM gets you to production faster with lower complexity.
6 4 Key Ideas to Remember
▾
01
Paged Attention eliminates KV Cache fragmentation — from 20–40% utilisation to near 100%
By managing KV Cache in fixed 16-token blocks allocated on demand (like OS virtual memory), vLLM eliminated the 60–80% GPU memory waste of naive contiguous allocation. This directly increases concurrency — more active requests per GPU — which is the primary driver of throughput improvement.
02
Continuous batching keeps the GPU busy at every decode step — GPU slots never idle between requests
The scheduler checks every decode iteration whether a finished request's slot can be filled with a new request. A request that finishes after 20 tokens immediately makes room for a waiting request, rather than holding its slot until the full static batch completes. Combined with Paged Attention, this is the source of the 5–10× throughput improvement.
03
The OpenAI-compatible API means near-zero migration cost from OpenAI to self-hosted serving
Any application built with the OpenAI Python SDK or standard HTTP calls can switch to a vLLM endpoint by changing one URL and one API key. This is a significant commercial argument for customers who want to reduce API spend or keep data on-premises.
04
vLLM is the right default — TensorRT-LLM wins only at the intersection of fixed model, high scale, and tight SLAs
Python overhead and runtime (vs compiled) execution mean vLLM leaves 20–40% throughput on the table vs TensorRT-LLM at peak load. But vLLM's model support breadth, faster startup, and lower operational complexity make it the right choice for the vast majority of deployments. Recommend TRT-LLM when the customer has definitively chosen their model and needs to squeeze maximum performance from fixed hardware.
MODULE 4.2 — CHEAT SHEET
vLLM
Open-source inference server from UC Berkeley (2023). Implements Paged Attention + continuous batching. OpenAI-compatible API. Python-first with custom CUDA kernels.
Paged Attention
Manages KV Cache in fixed 16-token blocks allocated on demand. Eliminates fragmentation (20% → ~99% utilisation). Enables prefix caching and copy-on-write sharing.
Continuous batching
Scheduler makes batching decisions at every decode step. Finished request slots filled immediately with waiting requests. No idle GPU time between requests. 5–10× throughput vs static batching.
gpu-memory-utilisation
Key config param. Default 0.90 (90% of HBM for vLLM). Lower to 0.85 if OOM under load. Never set to 1.0 — CUDA needs headroom.
max-num-seqs
Maximum concurrent sequences in the scheduler. Caps active batch size. Higher = more throughput but more KV Cache memory pressure. Default 256, tune down if OOM.
Prefix caching
Enabled via --enable-prefix-caching. Identical KV Cache blocks (e.g. shared system prompts) are stored once and referenced by multiple requests. TTFT drops from ~1.8s to <100ms for cached prefixes.
vLLM vs TensorRT-LLM
vLLM: easier setup, broad model support, right for most teams. TRT-LLM: compiled engines, 20–40% more throughput, right for fixed-model high-scale production.
Tensor Parallelism in vLLM
Set via --tensor-parallel-size N. Requires N GPUs on same node with NVLink. vLLM handles NCCL AllReduce automatically. TP=4 or TP=8 for models that don't fit on one GPU.
QUICK CHECK
A customer running vLLM with a 13B model on a single A100 80GB is seeing OOM (out of memory) errors when concurrent users exceed 20, despite the model only using ~26GB of the 80GB GPU. They have not changed any vLLM defaults. What is the most likely cause and fix?
A. The A100 80GB cannot run a 13B model — they need to use a smaller model or a larger GPU.
B. The model weights use 26GB but vLLM pre-allocates 90% of the 80GB GPU (72GB) for its memory pool at startup — of which 26GB goes to weights and the remaining 46GB to KV Cache blocks. With 20 concurrent users sending medium-length requests, the KV Cache fills the 46GB pool. The fix is not to reduce --gpu-memory-utilisation but instead to first check whether prefix caching is enabled (it reduces KV Cache pressure significantly if they have a shared system prompt), then consider reducing --max-num-seqs to cap concurrent sequences, and finally ensure --max-model-len is set to a realistic value rather than the model's maximum context window, which inflates block pre-allocation.
C. vLLM's continuous batching is causing too many requests to run simultaneously — disable continuous batching and switch to static batching with a fixed batch size of 16.
D. The OOM is caused by CUDA driver version incompatibility — they need to update their CUDA installation.
