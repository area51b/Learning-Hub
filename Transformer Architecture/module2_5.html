The Memory Bottleneck

Why GPUs run out of memory, what happens when they do, and the practical toolkit for fitting more into the memory you have.

Part 3 - Optimisation Techniques

GPU memory - HBM (High Bandwidth Memory) - is the most constrained resource in LLM inference. It is expensive, finite, and shared between three competing consumers that all need it at the same time. Understanding exactly how memory fills up, what happens when it runs out, and how to reclaim space is the practical foundation for every infrastructure sizing conversation.

This module completes Part 2 by tying the KV Cache, batching, and throughput concepts together into a single coherent picture of GPU memory pressure - and giving you the decision ladder for addressing it.

1 What Lives in GPU Memory - The Three Tenants
▾

Every byte of GPU HBM is occupied by one of three things. They do not take turns - all three are present simultaneously whenever inference is running.

WGT
Model Weights
Every parameter of the model - all the learned values in every attention head, every FFN layer, every embedding. These are fixed once the model is loaded and never change during inference. They consume the same amount of memory regardless of how many requests are being served.
7B FP16: ~14GB  |  70B FP16: ~140GB  |  405B FP16: ~810GB
KV
KV Cache
Key and Value vectors for every token in every active request, across every layer. This is the dynamic portion - it grows with each new token generated and shrinks when requests complete. With high concurrency and long contexts, KV Cache can consume more memory than the model weights themselves.
Grows with: active requests × context length × layers × precision
ACT
Activations and Runtime Overhead
Intermediate tensors computed during the forward pass - the outputs of each layer before they are written to the next. These are temporary, existing only for the duration of a single decode step. They also include CUDA kernel buffers, framework overhead, and memory for sampling (picking the next token). Typically 1–5GB depending on batch size.
Roughly constant per forward pass - not the primary concern
The fundamental constraint: Model weights are fixed and large. Activations are small and transient. KV Cache is dynamic and grows unboundedly with more requests and longer outputs. The contest for GPU memory is almost always between weights and KV Cache - and it determines how many users you can serve simultaneously.

3 The HBM Bandwidth Wall
▾

Even when there is enough GPU memory to fit a model and its KV Cache, performance can still be limited by how fast data can be read from that memory. This is the HBM bandwidth bottleneck - the wall that makes Decode memory-bound.

Every decode step requires reading the model weights and the full KV Cache from HBM. The speed at which this can happen is capped by the GPU's memory bandwidth - measured in terabytes per second (TB/s).

Each GPU generation roughly doubles the bandwidth available for Decode. This is why upgrading from A100 to H100 improves TPOT more than it improves Prefill speed - Decode is bandwidth-bound, Prefill is compute-bound.
The roofline model in plain English: A GPU has two limits - maximum compute (FLOPS) and maximum memory bandwidth (TB/s). Prefill hits the compute ceiling. Decode hits the bandwidth ceiling. You cannot accelerate Decode with more FLOPS - you need more bandwidth. This is why H200's 4.8 TB/s (vs H100's 3.35 TB/s) matters so much for interactive applications: it directly reduces TPOT for every request on every model.
4 When Memory Runs Out - Three OOM Scenarios
▾

Out-of-memory (OOM) errors in inference are not all the same. There are three distinct scenarios, each with different causes and fixes. Knowing which one you are facing is the first step to solving it:

OOM Type 1
Model does not fit at load time
The model weights alone exceed available GPU memory before any requests arrive. The server crashes on startup. Common when trying to run a 70B FP16 model on a single 80GB GPU (needs ~140GB).
Signals: Server fails to start. CUDA OOM error at model load. No requests can be served.
Fix: Add GPUs (tensor/pipeline parallelism) OR quantise the model (INT8 reduces 70B to ~70GB - fits on one A100) OR switch to a smaller model.
OOM Type 2
KV Cache exhaustion under load
The model loads fine, but as concurrent requests accumulate and contexts grow, the KV Cache fills all remaining GPU memory. New requests cannot be served or existing ones must be evicted. Happens gradually under load - the system degrades rather than crashes immediately.
Signals: Increasing TTFT under load. vLLM logs showing cache evictions. Requests queuing despite GPU being "available". Throughput plateau even as traffic grows.
Fix: Enable KV Cache quantisation (INT8 KV doubles effective KV capacity) OR reduce max context length OR enable Paged Attention (reduces fragmentation) OR add more GPUs OR reduce max concurrent requests.
OOM Type 3
Single long-context request OOM
A single request with an extremely long context (e.g. 128k tokens) requires a KV Cache so large it exhausts all remaining memory on its own, even if only one request is being served. Increasingly common with 128k+ context window models.
Signals: OOM error tied to specific long requests, not general load. Fails intermittently based on input length. Normal requests succeed fine.
Fix: Enforce a max context length limit on the API OR use KV Cache quantisation to reduce per-token KV size OR use a model with GQA (fewer KV heads) OR add GPU memory (H200 141GB per GPU helps significantly for long-context workloads).
Diagnostic shortcut: OOM at startup = Type 1 (weights). OOM under sustained load with normal inputs = Type 2 (KV Cache accumulation). OOM on specific long requests = Type 3 (single context). Each has a different fix, and mixing up the diagnosis leads to wasted engineering effort.

6 The Memory Mitigation Ladder - In Order of Cost
▾

When a customer is hitting memory limits, these are the interventions - ordered by cost and complexity. Always work up the ladder before jumping to hardware:

1
Enable Paged Attention
Eliminates KV Cache fragmentation - can recover 20–40% of "wasted" memory in naive systems. Zero quality impact. Config change only in vLLM (enabled by default).
+20–40% concurrency Zero cost
2
KV Cache quantisation (INT8)
Halves the memory used per KV Cache entry. Roughly doubles the number of concurrent requests or max context length. Minimal quality impact - KV Cache values are less sensitive to precision than model weights.
~2× KV capacity Near-zero cost
3
Model weight quantisation (INT8 or INT4)
Reduces model weight footprint by 50–75%, directly expanding the KV Cache headroom. INT8 has minimal quality impact. INT4 can have measurable impact on complex tasks - validate on your use case first.
2–4× weight reduction Low - quality validation needed
4
Switch to a model with GQA
Models using Grouped Query Attention (Llama 3, Mistral, Gemma) have far fewer KV heads than MHA models. Llama 3 70B has 8 KV heads vs Llama 2 70B's 64 - reducing KV Cache size by 8×. If the model supports GQA, this is a large, free KV saving.
Up to 8× KV reduction Model swap - re-evaluate quality
5
Downsize the model
A 13B model requires ~8× less KV Cache memory than a 70B model for the same context, and ~5× less weight memory. If quality requirements can be met with a smaller model, this is the most impactful memory reduction available - at the cost of capability.
5–10× total memory reduction Quality trade-off - benchmark first
6
Add GPU memory (more or larger GPUs)
More GPU memory is always effective but is the most expensive option. Justified when: software optimisations are exhausted, quality requirements prevent model changes, and workload growth projections warrant the investment. H200 (141GB) vs A100 (80GB) nearly doubles memory per GPU.
Linear with memory added High - last resort
7 4 Key Ideas to Remember
▾
01
GPU memory has three tenants - weights are fixed, KV Cache is the variable
Model weights and activations are relatively predictable. KV Cache is the one that scales with load and context length. Every memory optimisation conversation is really about freeing up more space for KV Cache to enable higher concurrency.
02
HBM bandwidth limits Decode speed - more memory also means more bandwidth
Larger, newer GPUs are not just bigger storage - they also have proportionally higher HBM bandwidth. The H200's 4.8 TB/s vs the A100's 2.0 TB/s directly translates to faster Decode. When evaluating GPU upgrades, bandwidth per GB is as important as total capacity.
03
There are three distinct OOM types - diagnose before prescribing
Model load OOM, KV Cache accumulation OOM, and single-request long-context OOM each have different causes and different fixes. Applying the wrong solution wastes time. The diagnostic shortcut: when does the OOM happen - startup, under load, or on specific requests?
04
Work up the mitigation ladder - hardware is the last step
Paged Attention, KV quantisation, weight quantisation, and GQA model selection can collectively deliver 5–20× more effective memory utilisation with no hardware change. The correct sequence: software optimisations first, hardware expansion last.
MODULE 2.5 - CHEAT SHEET
HBM (High Bandwidth Memory)
The GPU's on-chip memory. Fast but expensive and finite. Split between model weights, KV Cache, and activations during inference.
Memory Bandwidth
How fast data can be read from HBM. Measured in TB/s. The bottleneck for Decode speed. A100: 2.0 TB/s. H100: 3.35 TB/s. H200: 4.8 TB/s.
OOM Type 1 - Load failure
Model weights exceed GPU memory. Fails at startup. Fix: quantise weights or add GPUs.
OOM Type 2 - KV accumulation
KV Cache fills under sustained load. Causes evictions and latency spikes. Fix: KV quantisation, Paged Attention, reduce concurrency.
OOM Type 3 - Long context
Single request's KV Cache exhausts memory. Intermittent, tied to input length. Fix: max context limit, GQA model, KV quantisation.
KV Cache quantisation
Storing KV Cache in INT8 or INT4 instead of FP16. Halves or quarters KV memory use. Roughly doubles concurrency capacity with minimal quality impact.
Roofline Model
Framework for understanding whether a workload is compute-bound (Prefill) or memory-bandwidth-bound (Decode). Each has a different hardware bottleneck and fix.
Mitigation Ladder
Paged Attention → KV quantisation → weight quantisation → GQA model → model downsize → add hardware. Always work in this order.
QUICK CHECK
A customer is running Llama 2 70B in FP16 across 2 × A100 80GB GPUs. They report that the system works fine at low traffic but begins throwing OOM errors and slowing down significantly as concurrent users exceed 8. What type of OOM is this, and what is the most cost-effective first intervention?
A. Type 1 OOM - the model does not fit. They need to add 2 more GPUs immediately.
B. Type 2 OOM - KV Cache accumulation under load. The model loads fine but KV Cache exhausts remaining memory beyond ~8 concurrent users. Most cost-effective first step: enable KV Cache INT8 quantisation, which roughly doubles effective KV capacity and should push the concurrency ceiling to ~16 users with zero hardware change.
C. Type 3 OOM - individual long-context requests are causing the issue. They should enforce a 4,096-token maximum context limit.
D. The GPU bandwidth is the bottleneck - they need to upgrade to H100s for better memory bandwidth.
