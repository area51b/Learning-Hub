Why the Current Stack Has Limits

vLLM, TRT-LLM, and Triton are excellent tools - but at fleet scale they hit a ceiling. Understanding exactly where and why sets up everything in Part 5.

5.2 - NVIDIA Dynamo

Part 4 ended with a production stack that works well for most teams: vLLM or TRT-LLM on a DGX node, Triton in front, quantisation applied, parallelism configured. For a team serving thousands of requests per day on a handful of GPUs, this stack is close to optimal. The problems begin when the scale jumps by two orders of magnitude.

At the scale of hyperscaler inference deployments - hundreds of GPUs, millions of requests per day, dozens of model variants, mixed workloads with vastly different prompt and output lengths - the current stack exposes four structural limits that no amount of configuration tuning can fully overcome. These limits are not bugs in vLLM or TRT-LLM. They are architectural gaps between what single-node inference servers were designed to do and what fleet-scale inference actually requires. NVIDIA Dynamo and LLM-d exist specifically to close these gaps.

1 The Scale Where Problems Emerge
▾

The problems in this module are not visible at small deployments. They are emergent - they appear as a deployment grows from one node to many, from one model to several, from a few dozen concurrent users to thousands. Understanding the threshold at which each problem becomes significant helps you apply this knowledge correctly: a customer with 4 GPUs does not need Dynamo; a customer with 128 GPUs almost certainly does.

Problem severity by deployment scale - click a problem to see where it bites
The important framing for customer conversations: When a customer with a large deployment complains that vLLM "isn't scaling well" or that their P99 TTFT is unpredictable despite having plenty of GPU capacity, the root cause is almost always one of these four structural problems - not a misconfiguration of their existing stack. Knowing these problems by name and mechanism is what lets you move the conversation from "have you tried tuning --max-num-seqs" to "you have a KV Cache locality problem and you need a smarter router."
2 Problem 1 - Routing Blindness: Load Balancers Don't Know About KV Cache
▾

When a deployment grows beyond a single inference server instance, requests must be routed across multiple servers. The standard approach is a load balancer using round-robin or least-connections routing. This works well for stateless services. LLM inference is not stateless - it has KV Cache.

Consider a deployment with four vLLM replicas, all serving the same model with prefix caching enabled. A customer application has a 4,000-token system prompt that is identical for every user. With prefix caching, the first request that arrives on each replica pays the full Prefill cost for that system prompt (~400ms). Subsequent requests to the same replica find the KV Cache already built and skip it entirely (~20ms TTFT). With round-robin routing, each of the four replicas handles roughly 25% of requests - and each replica's prefix cache only covers the requests it receives. If a new replica spins up, it starts cold.

The routing blindness problem
A load balancer routing by least-connections has no visibility into which replica has which KV Cache blocks already built. It might route a request with a cached 4,000-token system prompt to the one replica that just restarted and has an empty cache - paying full Prefill cost unnecessarily. Meanwhile, the three replicas that have the cache sit idle or handle unrelated requests.

At scale with thousands of distinct system prompts across hundreds of replicas, the fraction of requests that hit a warm prefix cache drops dramatically. The latency benefit of prefix caching - one of the highest-ROI optimisations from Part 3 - is almost entirely negated by cache-unaware routing.
Impact at scale: prefix cache hit rate drops from ~80% (single node) to ~20% (round-robin across 16 replicas). TTFT increases 4–5x for requests that should be cache hits.
What is actually needed: A router that is KV Cache-aware - one that knows which replicas have which prefix hashes cached and routes requests to the replica most likely to have a cache hit. This requires the router to track cache state across the fleet in real time, not just queue depth or connection count. This is one of the core capabilities that Dynamo's smart router provides (Module 5.2).
3 Problem 2 - KV Cache Locality: Cache Lives on the GPU That Built It
▾

Even with a smarter router, KV Cache locality creates a deeper problem. In a disaggregated system (Module 3.6), Prefill workers build KV Cache and transfer it to Decode workers. But in a multi-replica deployment without disaggregation, the KV Cache built by Prefill on one GPU stays on that GPU. If Decode needs to continue on a different GPU - because the first GPU became overloaded, or because a different replica picked up the next turn in a multi-turn conversation - the KV Cache must be rebuilt from scratch or transferred at significant cost.

KV Cache locality - request routing across 4 replicas

Multi-turn conversations make this worse. In a chatbot deployment, a user's conversation history grows with each turn. Turn 1 builds a 200-token KV Cache on replica A. Turn 2 ideally hits replica A and extends that cache. But if the router sends turn 2 to replica B, replica B must rebuild the entire 200-token history from scratch before generating the response - paying full Prefill cost every turn. In a long conversation with 10 turns, this can mean 10 full Prefills instead of 1, for what should have been incremental cache extensions.

The locality-load balance tension: KV Cache locality argues for sticky routing - always send a user's requests to the same replica. Load balancing argues for spreading requests evenly. These objectives directly conflict. A replica that is "sticky" for many active long-context users will accumulate KV Cache pressure while other replicas sit underutilised. Resolving this tension requires a router that simultaneously tracks cache state, load, and memory pressure across all replicas - something a conventional load balancer cannot do.
4 Problem 3 - Disaggregation Is Hard to Operate at Fleet Scale
▾

Module 3.6 covered disaggregation - splitting Prefill and Decode onto dedicated GPU pools. The concept is sound and the performance gains are real. The operational problem is that disaggregation as implemented today (manually configured static P/D ratios, custom KV transfer scripts) does not scale gracefully to large fleets.

Static disaggregation (today)
P:D ratio set manually at deployment
Ratio fixed until operator intervenes
KV transfer via ad-hoc scripts
No awareness of individual request sizes
Prefill queue backs up during traffic spikes
Decode pool idles during light traffic
What fleet scale needs
Dynamic P:D ratio adjustment in real time
Per-request routing to least-loaded Prefill worker
Standardised KV transfer protocol
KV transfer aware of destination cache state
Automatic failover if Prefill worker dies
Unified observability across P and D pools

The coordination problem compounds at scale. With 4 Prefill workers and 12 Decode workers, the routing decision for each request involves: which Prefill worker has capacity, which Decode worker has a matching prefix cache, whether the KV transfer path between those two workers is healthy, and whether the Decode worker has enough free KV Cache blocks to accept the transfer. No single component in the current stack - not vLLM, not TRT-LLM, not Triton - owns this coordination logic. Teams building large disaggregated deployments today write bespoke orchestration code to fill this gap.

The hidden cost of bespoke orchestration: Every large-scale LLM inference team that has implemented disaggregation has written their own version of this coordination layer. It is custom, poorly documented, tightly coupled to their specific hardware, and breaks in non-obvious ways when GPU topology or traffic patterns change. NVIDIA Dynamo and LLM-d are both, in part, attempts to standardise this coordination layer so teams do not have to rebuild it from scratch.
5 Problem 4 - Heterogeneous Workloads: One Configuration Cannot Serve All Request Types
▾

Real production deployments do not receive uniform traffic. A single customer-facing LLM service might simultaneously handle: short interactive chat (100-token prompt, 50-token output), long-context document analysis (8,000-token prompt, 200-token output), code generation (500-token prompt, 2,000-token output), and batch summarisation (2,000-token prompt, 300-token output). Each of these has a completely different optimal configuration.

Request typeOptimal Prefill configOptimal Decode configP:D ratioSingle-server compromise
Interactive chat Small batch, fast dispatch Many concurrent streams, low TPOT 1:4 Over-allocates Prefill capacity; Decode pool too small
Long-context RAG Large batch, Flash Attention critical Short outputs, fast turnaround 3:1 Under-allocates Prefill; long prompts cause head-of-line blocking
Code generation Medium batch size Long outputs, sustained decode throughput 1:3 Decode pool saturates; code requests queue behind chat
Batch summarisation Maximum batch, latency flexible Medium outputs, throughput-first 1:2 Competes with interactive traffic; unpredictable latency

A single vLLM or TRT-LLM instance with one configuration must serve all of these simultaneously. The configuration is necessarily a compromise that under-serves every request type. At small scale this is acceptable. At large scale - where each request type represents thousands of requests per minute - the inefficiency compounds into significant wasted GPU capacity and missed SLAs.

What fleet-scale inference actually needs here: A routing layer that classifies incoming requests by type and directs them to specialised worker pools - a pool optimised for long-context Prefill, a pool optimised for high-concurrency Decode, a pool for batch workloads. This is the concept of heterogeneous worker pools, and it requires a global scheduler with visibility across the entire fleet, not a single per-instance scheduler inside vLLM or TRT-LLM.
6 The Gap: What the Current Stack Is Missing
▾

The four problems above point to a single missing layer in the current stack - a fleet-level coordination and scheduling layer that sits above the individual inference servers and provides:

What exists today (per-instance)
KV Cache management (Paged Attention)
Continuous batching (per server)
Quantisation (per engine)
TP/PP parallelism (per node)
Health checks (per Triton instance)
Metrics (per server, not fleet-wide)
What is missing (fleet-level)
KV Cache-aware routing across replicas
Dynamic P:D ratio adjustment
Standardised KV transfer protocol
Request classification and pool routing
Global scheduler with fleet-wide visibility
Unified observability across all workers

NVIDIA Dynamo addresses this gap with a purpose-built disaggregated inference framework. LLM-d addresses it through Kubernetes-native scheduler extensions and Gateway API. Both are covered in the following modules. The conceptual through-line from this module to those: everything Dynamo and LLM-d do is, at its core, a solution to one of these four specific problems.

Why this matters right now: As of 2025, most enterprise LLM deployments are still in the single-node or small multi-node stage where these problems are manageable. But as models get larger, context windows extend to 128k+, and usage scales, these problems will surface in every significant deployment. Understanding them now positions you to recognise them early in a customer's growth curve - before they are deep in bespoke orchestration code - and recommend the right framework at the right time.
7 4 Key Ideas to Carry into Module 5.2
▾
01
Routing blindness destroys prefix cache hit rates at scale - a KV Cache-aware router is not a nice-to-have, it is a prerequisite for prefix caching to work across a fleet
Prefix caching (Module 3.2) delivers its TTFT savings only when the request reaches the replica that already built the cache. Round-robin routing negates this entirely. At 16+ replicas with diverse system prompts, cache hit rates under blind routing drop to near zero - eliminating the single highest-ROI optimisation from Part 3.
02
KV Cache locality and load balancing are in direct tension - resolving it requires global fleet-state visibility, not local per-instance decisions
Sticky routing maximises cache hits but creates load imbalance. Round-robin routing balances load but destroys cache locality. The only way to serve both objectives simultaneously is a central scheduler that sees cache state, load, and memory pressure across every replica simultaneously - something no component in the current stack provides.
03
Disaggregation at fleet scale requires a coordination layer that today's teams build by hand - every large deployment reinvents the same bespoke orchestration
The P/D split concept works. The operational problem is that dynamic ratio adjustment, KV transfer routing, and Prefill-to-Decode handoff coordination are not provided by any standard component. Teams with large disaggregated deployments have all written custom versions of this logic. Dynamo and LLM-d exist to standardise it.
04
Heterogeneous workloads need specialised worker pools routed by request type - a single configuration serving all request types always under-serves each one
Interactive chat, long-context RAG, code generation, and batch summarisation have fundamentally different optimal P:D ratios, batch sizes, and memory configurations. A global scheduler that classifies requests and routes to specialised pools can serve each type at its optimum simultaneously - something a single vLLM or TRT-LLM instance with one configuration cannot do.
MODULE 5.1 - CHEAT SHEET
Routing blindness
Standard load balancers route by connection count or round-robin, with no knowledge of which replica has which KV Cache blocks. Negates prefix caching benefits at fleet scale. Fix: KV Cache-aware routing.
KV Cache locality
KV Cache built by Prefill on one GPU stays on that GPU. Multi-turn conversations pay full Prefill cost on each turn if routed to a different replica. Sticky routing conflicts with load balancing - needs global state to resolve.
Disaggregation coordination gap
Static P:D ratios, ad-hoc KV transfer scripts, and no standard protocol for Prefill-Decode handoff. At fleet scale, teams build bespoke orchestration. Dynamo and LLM-d standardise this layer.
Heterogeneous workloads
Chat, RAG, code gen, and batch have different optimal P:D ratios. One server config cannot serve all optimally. Fleet-level solution: request classification + specialised worker pools routed by a global scheduler.
What the current stack covers
Per-instance: KV Cache management, continuous batching, quantisation, TP/PP, per-server health + metrics. Everything works well at single-node or small multi-node scale.
What the current stack is missing
Fleet-level: KV Cache-aware routing, dynamic P:D adjustment, standardised KV transfer, request classification, global scheduler, unified cross-fleet observability.
Scale threshold
Problems become significant at ~16+ GPU replicas, high prefix cache reuse, disaggregated deployments, or mixed workload types. Below this threshold, the Part 4 stack is sufficient and adding a coordination layer adds complexity without benefit.
What comes next
NVIDIA Dynamo (5.2): disaggregated inference framework with smart KV-aware router and dynamic worker coordination. LLM-d (5.3): Kubernetes-native scheduling extensions for fleet-scale LLM inference.
QUICK CHECK
A customer runs 16 vLLM replicas behind a standard round-robin load balancer, all serving Llama 3 70B with a shared 3,000-token system prompt. They have prefix caching enabled on each replica. They report that TTFT is averaging 1.6 seconds - nearly as high as before they enabled prefix caching, which should theoretically bring it to under 100ms. What is the most precise diagnosis?
A. Prefix caching is not working correctly - the 70B model is too large for vLLM's Paged Attention block manager to cache efficiently.
B. The round-robin load balancer distributes requests evenly across all 16 replicas regardless of cache state. Each replica only receives approximately 1/16th of total traffic. Most requests arrive at a replica that has not yet built the 3,000-token system prompt cache - or built it recently and has since evicted it under KV pressure from other requests. The prefix cache hit rate across the fleet is near zero even though prefix caching is correctly configured on each individual replica. The fix is a KV Cache-aware router that tracks which replicas have which prefix hashes cached and routes accordingly - not a change to any per-replica configuration.
C. 16 replicas is too many - prefix caching is only effective on a single replica. They should consolidate to fewer, larger instances.
D. The system prompt is too long for prefix caching - 3,000 tokens exceeds the default block size in vLLM and prevents cache reuse.
