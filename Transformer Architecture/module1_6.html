The Full Transformer Block

How Attention and FFN stack together into layers - and what depth means for inference

You now know the two core components of a Transformer: Attention (which gathers context across tokens) and the FFN (which processes each token individually). In this module, you will see how they are assembled into a single transformer block - and how stacking many of these blocks creates the deep, powerful models used in production today.

This is the module where all the previous concepts connect. Understanding how blocks stack also gives you the intuition for why increasing model depth (more layers) increases both quality and inference cost - and how techniques like pipeline parallelism exploit this structure.

1 Anatomy of a Single Transformer Block
▾

A single transformer block is a fixed sequence of four operations. Every block in every model - from the smallest 1B model to the largest 400B model - follows the same structure:

One transformer block - executed once per layer
Input from previous layer
Token embeddings carrying all knowledge from layers below
↓
Layer Normalisation
Stabilises the numbers before attention - keeps training and inference numerically stable
↓
Multi-Head Attention
Every token gathers context from every other token. Q, K, V computed. KV Cache populated.
↓
Residual Connection (Add)
Attention output is added to the original input - preserving what the token knew before attention ran
↓
Layer Normalisation
Stabilises again before the FFN step
↓
Feed Forward Network (FFN)
Each token individually processed - expand to think widely, contract to a refined representation
↓
Residual Connection (Add)
FFN output added to pre-FFN input - again preserving existing information
↓
Output to next layer
Richer token embeddings - carry everything learned so far, passed to the block above
The two new components: You have already learned Attention and FFN in depth. The two additions here are Layer Normalisation (a stability mechanism - think of it as resetting the scale before each major operation) and Residual Connections (which ensure the block adds to what it knows rather than replacing it). Both are covered in the next two sections.
2 Residual Connections - Why Blocks Add, Not Replace
▾

Residual connections are one of the most elegant ideas in deep learning - and they are easy to understand conceptually. The core idea is: instead of a layer transforming the input into something new, it transforms the input into a small update that gets added to the original.

Residual connection - illustrated
Token before attention + What attention learned → Richer token
Token before FFN + What FFN reasoned → Even richer token
Each operation contributes an increment. The token's representation grows richer with each block - it never loses what it already knew.

Without residual connections, information from early layers can get "washed out" as it passes through dozens of layers. With them, the original token information always flows forward, and each layer only needs to learn the delta - what new information to add. This is why models can be made very deep (80, 96, 128 layers) without degrading quality.

Why this matters for inference: Residual connections mean every layer's output depends on the layer below it - which is why transformer inference cannot be trivially parallelised across layers. Layer 10 cannot start until layer 9 finishes. This sequential dependency is what pipeline parallelism is designed to manage when running large models across multiple GPUs. You will learn this in Module 3.4.
3 Layer Normalisation - The Stability Mechanism
▾

Layer normalisation is a small but critical operation that runs before both Attention and the FFN. Its job is simple: keep the numbers in a well-behaved range.

Without normalisation
After 30+ layers of additions and multiplications, some embedding values become very large, others very small. Attention scores become unstable. The model produces garbage outputs or fails to converge during training.
With layer normalisation
Before each major operation, the embedding values are rescaled to have a consistent mean and spread. Operations work in a stable, predictable range at every layer. Deep 80-layer models run reliably.
The analogy: Think of layer normalisation like a mixing board engineer adjusting audio levels before each stage of a recording. If one instrument is unexpectedly loud, it gets turned down before it hits the next processor - otherwise the distortion would compound through every stage. Layer norm does the same for token embeddings: resets the scale before every major operation.
For inference: Layer norm adds a small but non-zero compute cost at every layer. In INT8 or FP8 quantised models, layer norm is often kept in higher precision (FP16) because it is sensitive to numerical errors. This is a detail that comes up when discussing quantised model accuracy with customers.
4 Stacking Blocks - What Depth Actually Means
▾

A single transformer block gives the model one pass at understanding the input. Modern models repeat this block many times - a 7B model has 32 layers, a 70B model has 80. Each pass builds on the previous one.

1–4
Early layers - surface features
The model learns basic syntax, word boundaries, and simple local relationships. "This is a noun." "This token follows a preposition." Pattern recognition at a surface level.
5–16
Middle layers - semantic understanding
The model builds meaning, resolves references, and understands context. "This 'it' refers to 'server'." "This sentence describes a failure." Topic and entity relationships emerge.
17–28
Later layers - reasoning and knowledge
The model applies learned world knowledge, reasons about implications, and makes judgements. "A memory error in a server usually means the process was killed by the OS." Domain expertise emerges.
29–32
Final layers - output preparation
The model consolidates its representation into a form ready for the next-token prediction. The final layer's output is used to select the most likely next token from the vocabulary.
Why layer count matters for inference cost: Every layer must execute sequentially. A 32-layer model runs 32 full Attention + FFN cycles per token. An 80-layer model runs 80. More layers = higher latency per token and more parameters = more GPU memory. This is the direct trade-off between model quality and inference cost - and why model selection (7B vs 70B) is often the single biggest infrastructure decision.

6 How the Layer Structure Enables Pipeline Parallelism
▾

Because a transformer is simply a stack of identical blocks, it is straightforward to split those blocks across multiple GPUs. This is called pipeline parallelism.

Pipeline parallelism - 4 GPUs, 32-layer model
GPU 0
Layers 1–8    (Attention + FFN x8)
GPU 1
Layers 9–16    (Attention + FFN x8)
GPU 2
Layers 17–24    (Attention + FFN x8)
GPU 3
Layers 25–32    (Attention + FFN x8)
Each GPU holds 1/4 of the model's weights. Output from GPU 0 is passed to GPU 1, then GPU 2, then GPU 3. The model is too large for one GPU but fits comfortably across four.
Pipeline parallelism vs tensor parallelism: Pipeline parallelism splits the model vertically by layers - different GPUs handle different layers. Tensor parallelism splits the model horizontally - each GPU handles a slice of every layer simultaneously. Most large-scale deployments use both. You will learn this in detail in Module 3.4.
7 4 Key Ideas to Remember
▾
01
Every transformer block = LayerNorm + Attention + Residual + LayerNorm + FFN + Residual
This six-step sequence is identical in every block of every transformer model. Knowing this structure lets you reason about where time and memory go at each layer - and which optimisations target which step.
02
Residual connections mean each layer adds knowledge, not replaces it
This is what makes deep models (80+ layers) work well. It also creates sequential layer dependencies - layer N cannot start until layer N-1 finishes - which is the constraint pipeline parallelism is designed to manage.
03
Depth (layers) and width (embedding size) both drive quality and cost
More layers = deeper reasoning but higher latency and more parameters. Larger embedding = richer representations but more memory per token. Model architecture is always a quality-vs-cost trade-off across these dimensions.
04
The uniform layer structure is what makes multi-GPU deployment straightforward
Because every block is structurally identical, you can split them evenly across GPUs. Pipeline parallelism assigns groups of layers to different GPUs. Tensor parallelism shards each layer across GPUs. Both exploit the regular, repeating block structure.
MODULE 1.6 - CHEAT SHEET
Transformer Block
One repeating unit: LayerNorm + MHA + Residual + LayerNorm + FFN + Residual. Stacked N times to form the model.
Residual Connection
Adds the layer's output to its input. Ensures each layer learns a delta, not a replacement. Enables very deep models.
Layer Normalisation
Rescales token embeddings before each major operation. Keeps numerical values stable across deep networks.
Model Depth (Layers)
Number of stacked transformer blocks. More layers = deeper reasoning, more parameters, higher cost per token.
Pipeline Parallelism
Splits model layers across GPUs - different GPUs handle different layer ranges. Solves the "model too large for one GPU" problem.
Tensor Parallelism
Splits each layer's weights across GPUs horizontally - every GPU works on every layer but only a slice. High bandwidth needed between GPUs.
Sequential Dependency
Layer N must wait for layer N-1. Inference cannot be parallelised across layers for a single request - only across requests (batching).
Early vs Late Layers
Early layers learn syntax and surface patterns. Middle layers build semantics. Late layers apply reasoning and world knowledge.
QUICK CHECK
A customer has a 70B model (80 layers) they need to deploy across 4 x A100 80GB GPUs. A colleague suggests using pipeline parallelism. What does that mean in practice, and what is the key limitation to be aware of?
A. Pipeline parallelism means each GPU processes a different token simultaneously - the limitation is that tokens must arrive in batches
B. Pipeline parallelism means each GPU holds a different group of layers (e.g. GPU 0 handles layers 1–20, GPU 1 handles 21–40, etc.). The key limitation is pipeline bubbles - GPUs earlier in the pipeline sit idle while waiting for the next batch when the current one finishes
C. Pipeline parallelism means the model weights are duplicated across all 4 GPUs for redundancy - the limitation is 4x the memory cost
D. Pipeline parallelism is not suitable for transformer models because the residual connections prevent layer splitting
