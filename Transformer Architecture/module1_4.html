Multi-Head Attention

Why one attention pass is not enough — and what happens when you run many in parallel

~10 min read Concept level No math
← Previous
1.3 — Attention Mechanism
Next →
1.5 — Feed Forward Network

In Module 1.3 you learned that Attention lets every token look at every other token to decide what is relevant. But a single attention pass can only look for one type of relationship at a time — like only being able to ask one question at a time about a document.

Multi-Head Attention (MHA) solves this by running many attention passes simultaneously, each looking for a different kind of relationship. This module explains what that means, why it matters for model quality, and — critically for your customer conversations — how it directly drives KV Cache size and memory usage during inference.

1 The Problem with a Single Attention Pass
▾

Imagine you are reviewing a technical proposal document. To fully understand it you need to ask many different questions simultaneously:

Questions a reader asks simultaneously
Who is the subject?
Which noun does each pronoun refer to?
What is the action?
What verb connects these parts of the sentence?
What is the tone?
Is this section positive, negative, or neutral?
What is the structure?
How does this paragraph relate to the previous one?

A single attention pass can only focus on one of these questions at a time. If it asks "who is the subject?", it may miss information about tone or structure. Multi-Head Attention lets the model ask all of these questions in parallel — each attention head specialises in a different type of relationship.

The analogy: Think of a panel of expert reviewers reading the same document simultaneously. One reviewer focuses on logical structure. Another focuses on technical accuracy. Another focuses on tone. Each produces independent notes, which are then combined into one comprehensive assessment. Multi-Head Attention works the same way — parallel specialised passes, combined at the end.

3 How It Works — Split, Attend, Combine
▾

Multi-Head Attention follows a simple three-step process. No formulas — just the logic:

1
Split
Each token's embedding is split into h smaller pieces — one for each attention head. If the embedding has 4,096 numbers and there are 32 heads, each head gets a 128-number slice to work with.
2
Attend in Parallel
Each head runs a full, independent attention computation on its slice. All heads run simultaneously on the GPU — this is one of the biggest sources of parallelism in the Transformer. Each head produces its own output, capturing a different aspect of the relationships in the input.
3
Concatenate and Project
The outputs from all heads are joined back together (concatenated) into a single full-size vector. A final linear transformation blends the insights from all heads into one unified representation that flows to the next layer.
Simplified — how 4 heads combine into one output

→ Combined output
Each head contributes its own perspective. The final projection layer learns how to weight and blend each head's contribution for downstream use.
4 MHA vs MQA vs GQA — The Inference Optimisation Variants
▾

This is where Multi-Head Attention connects directly to inference optimisation. You will hear these three terms in customer conversations — here is what they mean and why they matter:

Variant Full Name How it works Impact on inference
MHA Standard Multi-Head Attention Every head has its own separate K and V matrices. Most expressive. Largest KV Cache. Highest memory usage. Best quality.
MQA Optimised Multi-Query Attention All heads share a single K and V. Each head still has its own Q. Dramatically smaller KV Cache. Faster inference. Slight quality trade-off.
GQA Balanced Grouped Query Attention Heads are grouped — each group shares one K and V. Middle ground between MHA and MQA. Significantly reduced KV Cache with minimal quality loss. Used in most modern models (Llama 3, Mistral, Gemma).
Why this matters for your customer conversations: When a customer asks why Llama 3 is more memory-efficient than an older model of similar size, a big part of the answer is GQA. Llama 3 70B uses GQA with 8 key-value heads instead of 64 — meaning the KV Cache is 8x smaller than it would be with full MHA. This is a real, practical inference cost reduction you can point to.
5 How Head Count Drives KV Cache Size
▾

Now you can understand exactly what the KV Cache is made of and why it grows so large. For every token processed, the model stores K and V vectors for every head, at every layer.

KV Cache size — what it depends on
Number of tokens
Longer context = proportionally larger cache. A 32k context is 4x larger than an 8k context.
Number of heads
MHA stores K+V for every head. GQA reduces this by sharing heads across groups.
Number of layers
Each transformer layer has its own attention heads. A 32-layer model stores 32x more KV data than a 1-layer model.
Data precision
FP16 uses 2 bytes per number. INT8 uses 1 byte. KV Cache quantisation halves memory with minimal quality impact.
Practical example: A 70B parameter model serving a 32k-token request with full MHA (64 heads, 80 layers, FP16) can generate a KV Cache of over 80GB — more than the GPU memory on most single A100s. This is exactly why Paged Attention, KV Cache quantisation, and disaggregation exist. Every optimisation technique you will learn in Part 3 is ultimately fighting this memory pressure.
6 4 Key Ideas to Remember
▾
01
Multi-Head Attention runs many attention passes in parallel, each capturing a different relationship
Syntax, coreference, semantics, long-range dependencies — each head specialises independently. This is why LLMs understand language so richly compared to earlier models.
02
Each head has its own K and V in standard MHA — multiplying KV Cache size
More heads = richer understanding but larger KV Cache. A 64-head model stores 64 sets of K and V vectors per token per layer. This is the root cause of GPU memory pressure at inference time.
03
GQA is the modern solution — shared K and V across head groups
Most modern models (Llama 3, Mistral, Gemma) use Grouped Query Attention to cut KV Cache size dramatically with minimal quality impact. When customers ask about memory efficiency improvements, GQA is a key part of the answer.
04
KV Cache size = tokens x heads x layers x precision
These four factors multiply together. Reducing any one of them reduces memory pressure. This is why techniques like KV Cache quantisation (reducing precision), GQA (reducing heads), and context window management (reducing tokens) all matter for inference optimisation.
MODULE 1.4 — CHEAT SHEET
Multi-Head Attention (MHA)
Running many independent attention passes in parallel. Each head captures a different type of relationship.
Attention Head
One independent attention pass. Has its own Q, K, V projections. Specialises in one type of relationship.
MQA (Multi-Query Attention)
All heads share one K and V. Dramatically reduces KV Cache. Slight quality trade-off.
GQA (Grouped Query Attention)
Groups of heads share K and V. Best balance of quality and memory. Used in Llama 3, Mistral, Gemma.
Head Count
Number of parallel attention heads per layer. Larger models have more heads. Directly multiplies KV Cache size.
KV Cache Formula
Size = tokens x KV heads x layers x head dimension x precision. Every factor matters for memory budgeting.
Concatenation
The step where all head outputs are joined back together. Followed by a projection to blend insights from all heads.
Head Specialisation
Heads learn different roles during training automatically — not manually programmed. Different heads focus on syntax, semantics, coreference, etc.
