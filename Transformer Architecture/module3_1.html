Flash Attention

The same mathematical result as standard attention — computed 2–4× faster and using a fraction of the GPU memory. How reordering the same operations changes everything.

Flash Attention does not change what attention computes — it changes where the computation happens inside the GPU. The result is mathematically identical to standard attention. But by carefully rearranging the order of operations to keep data in fast on-chip memory instead of slow off-chip HBM, it achieves 2–4× speedups and reduces memory usage from quadratic to linear with sequence length.

Published by Dao et al. in 2022, Flash Attention has since become the default attention implementation in virtually every major inference framework — vLLM, TensorRT-LLM, and transformer libraries all use it. Understanding why it works makes the rest of Part 3 easier, because the same insight (avoid expensive memory round-trips) underlies many other optimisations.

1 The Root Problem — GPU Memory Is Not Flat
▾

To understand Flash Attention, you first need to understand that a GPU has two very different types of memory — and accessing one of them is vastly slower than the other.

GPU memory hierarchy — speed vs capacity
SRAM (on-chip)
~19 TB/s bandwidth
~20 MB total
HBM (off-chip)
~2–5 TB/s bandwidth
40–141 GB total
SRAM (Static RAM) sits directly on the GPU die. It is tiny but extraordinarily fast — roughly 10× the bandwidth of HBM. HBM (High Bandwidth Memory) is the large off-chip memory where model weights and KV Cache live. Every time data must travel between SRAM and HBM, it passes through a bottleneck. Standard attention does this repeatedly. Flash Attention minimises it.
The analogy: Think of SRAM as a whiteboard right next to you — fast to read and write, but small. HBM is a filing cabinet across the room — much more storage, but every trip takes time. Standard attention repeatedly walks to the filing cabinet and back. Flash Attention reorganises the work so you can do as much as possible at the whiteboard before making a single trip.
2 What Standard Attention Does — and Why It Is Slow
▾

Standard attention (as described in the original Transformer paper) follows a simple but memory-expensive sequence of steps. Each step writes a large intermediate result to HBM, then the next step reads it back. This constant back-and-forth is the bottleneck.

Standard Attention
1
Compute Q×K scores matrix for all token pairs → write full N×N matrix to HBM
2
Read N×N matrix back from HBM → apply Softmax → write result to HBM again
3
Read Softmax result from HBM → multiply by V matrix → write output to HBM
4
Read final output from HBM → done
5 round-trips to HBM. The N×N score matrix alone costs O(N²) memory — 128k tokens = 16 billion entries.
Flash Attention
1
Load a small tile of Q, K, V into SRAM (fast on-chip memory)
2
Compute scores, softmax, and weighted sum entirely within SRAM — no HBM write
3
Accumulate partial result into output. Load next tile. Repeat until all tiles processed.
4
Write final output to HBM once — no intermediate matrices ever stored
1 write to HBM. No N×N matrix ever materialised. Memory usage is O(N) — linear, not quadratic.
The N×N problem in numbers: For a 4,096-token sequence, the standard attention score matrix has 4,096 × 4,096 = 16.7 million entries per attention head, per layer. A 70B model with 64 attention heads across 80 layers would require storing 85 billion intermediate values — tens of gigabytes — just for one forward pass. Flash Attention never stores this matrix at all.
3 Tiling — How Flash Attention Works
▾

The core technique Flash Attention uses is called tiling. Instead of loading the entire Q, K, V matrices and computing attention in one shot, it divides them into small tiles that fit in SRAM and processes them sequentially — accumulating the correct final result tile by tile.

The mathematical trick that makes this work is an online softmax algorithm: you can compute the correct softmax over the full sequence incrementally as tiles are processed, without ever having all the scores in memory at once. This is the key insight of the Flash Attention paper.

What makes this mathematically correct: The challenge with tiling is that softmax normally requires all scores before it can normalise them. Flash Attention's online softmax algorithm keeps running statistics (the current maximum score and sum of exponentials) that are updated incrementally as each new tile is processed. The final output is identical to what standard attention would produce — no approximation, no quality loss.

Why long contexts became practical: Before Flash Attention, running a 32k-token context required storing an attention matrix too large to fit in GPU memory — making it practically impossible at inference time. Flash Attention eliminated this constraint entirely. The explosion of 32k, 64k, 128k, and now 1M+ context window models from 2023 onwards was directly enabled by Flash Attention making long-context computation feasible.
5 Flash Attention 1, 2, and 3 — What Each Version Added
▾

Flash Attention has gone through three major versions since its 2022 introduction, each targeting a different remaining inefficiency:

VersionYearKey additionSpeedup vs prevStatus
Flash Attention 1 2022 Tiling + online softmax. Eliminated the N×N intermediate matrix. Made long contexts practical. 2–4× vs standard Superseded
Flash Attention 2 2023 Better parallelism across attention heads. Reduced non-matmul operations. Improved GPU thread scheduling. Better causal masking. ~2× vs FA1 Current default
Flash Attention 3 2024 H100-specific optimisations — exploits new Tensor Core features (WGMMA, TMA). Overlaps compute and memory transfers. FP8 support. 1.5–2× vs FA2 on H100 H100/H200 deployments
What this means for customers: If a customer is still running standard attention (some older frameworks or custom implementations), switching to Flash Attention 2 is one of the highest-ROI, lowest-risk upgrades available — it is a drop-in replacement that improves Prefill speed and enables longer contexts with zero quality change. If they are on H100s, Flash Attention 3 provides an additional 1.5–2× on top of that.
6 4 Key Ideas to Remember
▾
01
Flash Attention produces the exact same result as standard attention — no approximation
This is the most important thing to communicate to technically cautious customers. It is not a quantisation or an approximation — it is a mathematically equivalent reformulation that uses the GPU's memory hierarchy more efficiently. Quality is completely unchanged.
02
The speedup comes from staying in fast SRAM — not from doing less work
Flash Attention does the same mathematical operations as standard attention. The gain is purely from avoiding slow HBM round-trips for intermediate results. This insight — minimise slow memory access, maximise fast on-chip computation — is the core principle behind many other GPU optimisations.
03
It turns O(N²) memory into O(N) — this is what made long contexts possible
Standard attention needs memory proportional to the square of the sequence length. A 128k-token context with standard attention would require hundreds of gigabytes just for intermediate matrices. Flash Attention only needs memory proportional to the sequence length — enabling 128k, 200k, and 1M+ context windows that are standard today.
04
Flash Attention primarily benefits Prefill — it speeds up TTFT, not TPOT
Flash Attention's biggest gains are during the Prefill phase, where large Q, K, V matrices are being processed for the full input. During Decode, only one new token is computed per step — the attention matrix is tiny — so Flash Attention has less to contribute. If a customer's primary complaint is slow Decode (high TPOT), Flash Attention alone will not fix it.
MODULE 3.1 — CHEAT SHEET
Flash Attention
A memory-efficient attention algorithm that tiles Q, K, V to stay in SRAM, avoiding slow HBM round-trips. Exact same output as standard attention.
SRAM (on-chip)
Tiny (~20MB), extremely fast (~19 TB/s) memory on the GPU die. Flash Attention keeps computation here instead of writing intermediates to HBM.
HBM (off-chip)
Large (40–141GB), slower (2–5 TB/s) memory. Standard attention writes and reads large intermediate matrices to/from HBM. Flash Attention minimises this.
Tiling
Dividing Q, K, V into small blocks that fit in SRAM. Each tile is processed entirely on-chip, and results are accumulated incrementally into the final output.
Online Softmax
The mathematical trick that makes tiling work — allows correct softmax to be computed incrementally as tiles are processed, without storing all scores at once.
O(N²) → O(N)
Standard attention memory scales with sequence length squared. Flash Attention scales linearly. This is what made 128k+ context windows practical.
Primary benefit
Faster Prefill (better TTFT) and support for longer contexts. Does not significantly improve Decode speed (TPOT). Best for long-context and latency-sensitive apps.
Flash Attention 3
H100/H200-specific version using new hardware features (WGMMA, TMA). Adds FP8 support. 1.5–2× faster than FA2 on compatible hardware.
QUICK CHECK
A customer running a document analysis application wants to increase their context window from 8k to 64k tokens to process longer documents. Their current setup uses standard attention. A colleague suggests enabling Flash Attention. Is this the right recommendation, and what should the customer expect?
A. Flash Attention will reduce output quality at 64k tokens — they should use a model specifically fine-tuned for long contexts instead.
B. Flash Attention is exactly the right recommendation. It is a drop-in replacement that makes 64k context computationally feasible by eliminating the O(N²) intermediate matrix — the memory that standard attention would need for 64k tokens would be prohibitive. Output quality is unchanged. TTFT will improve significantly. TPOT improvement will be minimal since Decode attention on one new token is small regardless.
C. Flash Attention is not the right tool — they need more GPUs to handle the longer context window.
D. Flash Attention only helps during training, not inference — they should look at Paged Attention instead.
