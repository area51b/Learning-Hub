How text becomes something a model can actually work with

In Module 1.1, you learned that a Transformer reads all input tokens simultaneously. But what exactly is a token? And how does the model understand the meaning of a word, not just its letters?

This module answers those two questions. They matter for inference because the number of tokens directly determines how much memory and compute your system needs — and it's how all inference costs are measured.

1 What Is a Token?
▾

A token is the basic unit of text that a language model works with. It is not exactly a word — it is a chunk of text that the model has learned to recognize as a meaningful piece.

Think of it like this: a word can be one token, or it can be split into several tokens depending on how common it is. Very common short words like "the" or "is" are usually one token. Longer or rarer words get broken into smaller pieces.

Example — how a sentence is tokenized
"Inference optimization improves throughput."
Inf erence optim ization impr oves through put .
Each colored chip = one token. Notice how "Inference" splits into "Inf" + "erence" and "throughput" splits into "through" + "put". This sentence = 9 tokens.
Rule of thumb: 1 token is roughly 3/4 of a word in English. A page of text is about 500 tokens. A 10-page document is roughly 5,000 tokens. This is why you hear "context window" measured in tokens — e.g. "128k context" means the model can handle 128,000 tokens at once.
2 Why Not Just Use Words?
▾

You might wonder: why not just treat each word as the basic unit? The reason comes down to practicality and efficiency.

Problem with word-level
English has 170,000+ words — impossible to track them all
New words, brand names, typos break the system
Different languages need completely separate systems
Technical terms like "TensorRT-LLM" would be unknown
Why tokens work better
Only ~50,000 tokens needed to cover most of any language
Unknown words get broken into known sub-pieces
Works across languages with the same vocabulary
"TensorRT-LLM" becomes "Tensor" + "RT" + "-" + "LL" + "M"
The vocabulary: The full list of all possible tokens a model knows is called its vocabulary. Most modern LLMs have a vocabulary of around 32,000 to 128,000 tokens. Every prompt you send gets converted into a sequence of these token IDs before the model ever sees it.
3 What Is an Embedding?
▾

Once your text is split into tokens, the model still can't process plain text. It needs numbers. Each token gets converted into a list of numbers called an embedding.

An embedding is not random. It is a carefully learned representation where similar meanings produce similar numbers. This is how the model "understands" language — not through grammar rules, but through numerical relationships.

Simplified illustration — tokens as number lists
"King"
→
0.820.140.910.370.55...
"Queen"
→
0.790.180.880.410.72...
"Car"
→
0.110.930.040.780.22...
King and Queen have similar numbers because they are semantically close. Car has very different numbers. In reality, each embedding has thousands of numbers, not just five.
The analogy: Think of embeddings like GPS coordinates. Two cities near each other have similar coordinates. Two cities on opposite sides of the world have very different coordinates. Embeddings do the same thing for meaning — words with similar meanings end up "close" to each other in numerical space.
4 Embedding Size and Why It Matters for Inference
▾

Each token gets converted into a long list of numbers. The length of this list is called the embedding dimension (also called model width or hidden size). Bigger models have larger embedding dimensions.

Model Size Embedding Dimension What It Means
Small (7B params) 4,096 numbers per token Lighter, faster inference, less GPU memory needed
Medium (70B params) 8,192 numbers per token More expressive, requires more memory per token
Large (400B+ params) 16,384+ numbers per token Highest quality, very expensive to serve at scale
Why this connects to inference costs: Every token in your prompt generates a large vector of numbers that flows through every layer of the model. A longer prompt = more tokens = more vectors = more memory and compute. This is why token count is the fundamental unit of cost in LLM inference, not words or characters.
5 One More Thing — Position Matters
▾

Remember that the Transformer reads all tokens at once — it doesn't process them left to right. This creates a problem: if you read everything simultaneously, how do you know which word came first?

The answer is positional encoding. Before passing tokens into the model, each embedding gets extra information added to it that tells the model its position in the sequence — token 1, token 2, token 3, and so on.

The analogy
Imagine you receive 10 sticky notes from a colleague all at once, mixed together. You can read them all instantly, but you need to know the order they were written in. Positional encoding is like numbering each sticky note before handing them to you — so you know note 1 came before note 2, even though you're reading all 10 at the same time.
Why this matters for inference: Positional encoding is part of why there is a context window limit. The model was trained up to a certain position number. If your prompt + response exceeds that position count, the model's quality degrades. This is why you hear about "128k context" or "1M context" — it's the max position the model was trained to handle.
6 3 Key Ideas to Remember
▾
01
Tokens are not words — they are sub-word chunks
A model never sees raw text. It sees a sequence of token IDs. One word can be one token or several. Token count is the true measure of input size — and of cost.
02
Embeddings turn tokens into meaning-rich numbers
Each token becomes a large vector of numbers. Similar meanings produce similar vectors. This is the foundation of how a model "understands" language — through geometric relationships, not rules.
03
Every token costs memory and compute — at every layer
The embedding of each token flows through all the transformer layers. Longer context = more tokens = more work. This is why context length directly drives inference cost and why KV Cache exists — to avoid recomputing these vectors on every step.
MODULE 1.2 — CHEAT SHEET
Token
Sub-word chunk of text. Basic unit the model processes. ~3/4 of a word on average.
Tokenization
The process of splitting raw text into tokens before the model can use it.
Vocabulary
The complete set of all tokens a model knows. Typically 32k–128k tokens.
Embedding
A list of numbers representing a token's meaning. Similar meanings = similar numbers.
Embedding Dimension
How many numbers per token. Bigger models = higher dimensions = more memory per token.
Positional Encoding
Extra info added to each embedding telling the model where in the sequence that token sits.
Context Window
Max number of tokens (input + output) a model can handle at once. E.g. 128k tokens.
Token Count
The true unit of inference cost and memory usage — not words, not characters.
