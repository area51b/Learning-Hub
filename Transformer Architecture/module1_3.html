How the model decides which tokens matter most — the concept, without the math

Attention is the single most important concept in the entire Transformer architecture. It is what separates modern LLMs from every AI system that came before. Flash Attention — one of the biggest inference optimizations you will talk about with customers — exists entirely because of how Attention works.

This module explains Attention at a conceptual level. No formulas. Just the idea, an interactive visual, and the direct connection to why it creates an inference bottleneck.

1 The Core Idea: Not All Words Are Equally Important
▾

When you read a sentence, you don't treat every word with the same level of focus. Your brain naturally pays more attention to certain words depending on what you are trying to understand.

Consider this sentence:

"The server that the engineer deployed last Tuesday crashed because it ran out of memory."

If you are asked "why did it crash?", your brain immediately links crashed back to memory — even though those words are far apart and separated by several other words. The words in between ("that the engineer deployed last Tuesday") get lower attention for this specific question.

This is exactly what the Attention mechanism does. For every token in the input, it calculates a score for how much it should "pay attention" to every other token. Tokens that are highly relevant to each other get high scores. Irrelevant ones get low scores.

Key insight: Attention is not a fixed rule. It is computed dynamically for every input. The same word gets different attention weights depending on the context it appears in. This is why Transformers are so much more powerful than older models — the "focus" adapts to each unique prompt.

3 The Q, K, V Framework — The Library Analogy
▾

You will hear the terms Query (Q), Key (K), and Value (V) in almost every conversation about LLM inference — especially KV Cache. Here is what they mean without any math.

Imagine a library with a smart search system:

Q
Query
The question being asked. Each token generates a Query that says: "What information do I need from the other tokens?"
Library analogy: your search request — "I am looking for books about network failures."
K
Key
The label on each token that describes what it contains. Other tokens use it to decide whether they are relevant to a Query.
Library analogy: the index card on each book shelf — "This section: networking, failures, systems."
V
Value
The actual content that gets passed forward once a match is found. The information that flows to the token that asked.
Library analogy: the actual content of the books you retrieve after a search match.
How it works together: Every token generates all three — a Q (what am I looking for?), a K (what do I contain?), and a V (what do I send if someone finds me relevant?). The model compares every Q against every K to calculate attention scores, then uses those scores to pull the right V values. This happens for every token, against every other token, at every layer.
4 Why Attention Is Computationally Expensive
▾

Here is the key problem: every token must compare its Query against the Keys of every other token in the sequence. This means the amount of work grows very fast as the sequence gets longer.

Short prompt — 100 tokens
Each token compares against 100 others
100 x 100 = 10,000 comparisons
Fast and manageable
Fits easily in GPU memory
Long prompt — 10,000 tokens
Each token compares against 10,000 others
10,000 x 10,000 = 100,000,000 comparisons
1,000x more work than 100 tokens
Intermediate results overflow GPU memory

This is called quadratic scaling — doubling the input length quadruples the attention work. It is the fundamental reason why long-context inference is so expensive and why every major optimization technique — Flash Attention, Paged Attention, KV Cache management — exists to manage this cost.

The memory problem that Flash Attention solves: When computing attention, the model generates a large intermediate matrix of all those comparison scores. For long sequences, this matrix becomes enormous and does not fit efficiently in fast GPU memory (SRAM). Flash Attention is a technique that avoids materialising this full matrix — it computes attention in tiles, keeping everything in fast memory. Same results, dramatically less memory pressure. You will learn this in detail in Module 3.1.
5 The Direct Connection to KV Cache
▾

Now that you understand Q, K, and V — you can understand exactly what the KV Cache is and why it matters so much.

What is the KV Cache?

During the Prefill phase, the model computes K (Key) and V (Value) vectors for every single input token. These are expensive to compute — and during the Decode phase, when the model generates each new output token, it needs to run attention against all previous tokens.

Without a cache: the model would recompute all K and V vectors from scratch for every single new token generated. For a 5,000-token input, that is 5,000 full K and V computations — repeated hundreds of times during generation.

With the KV Cache: those K and V vectors are computed once during Prefill and saved in GPU memory. Each new token just looks up the cache instead of recomputing. This is what makes generation fast.

The trade-off: The KV Cache makes generation fast, but it is large. For a 70B parameter model with a 32k token context, the KV Cache can be tens of gigabytes. This is the #1 reason GPU memory fills up during inference — and why techniques like Paged Attention, quantisation, and disaggregation exist to manage it.
6 The 4-Step Attention Process (No Math)
▾

For each token, attention happens in four steps. You do not need to know the formulas — just the logic:

STEP 1
Generate Q, K, V
Each token produces its own Query, Key, and Value vectors from its embedding
STEP 2
Score
Compare this token's Query against every other token's Key — producing a relevance score for each pair
STEP 3
Normalise
Convert raw scores into weights that sum to 1 — so they act like percentages of attention
STEP 4
Aggregate
Multiply each token's Value by its weight and sum them all — producing the final output for this token
Why this matters for your customer conversations: When someone asks "what does Flash Attention actually do?", the answer is: it changes how Step 2 and 3 are computed in memory — not what they compute. The mathematical result is identical. The difference is entirely about where the intermediate data lives during computation, which makes it much faster and more memory-efficient.
7 4 Key Ideas to Remember
▾
01
Attention dynamically weights every token against every other token
It is not a fixed rule. It adapts to every input. This is what makes LLMs context-aware rather than just pattern-matching on nearby words.
02
Q, K, V are the three ingredients of every attention calculation
Query = what this token is looking for. Key = what this token contains. Value = what this token sends when found relevant. The KV Cache stores K and V so they are not recomputed during generation.
03
Attention scales quadratically with sequence length
Double the tokens, quadruple the work. This is the core reason long-context inference is expensive — and the reason Flash Attention, KV Cache optimisation, and context-length management matter so much.
04
The KV Cache is the saved output of attention — not a shortcut
Caching K and V vectors avoids recomputation during the Decode phase. It makes generation fast but consumes large amounts of GPU memory — which is the central tension in inference optimisation.
MODULE 1.3 — CHEAT SHEET
Attention
Mechanism that computes relevance scores between every pair of tokens in the sequence.
Query (Q)
What a token is looking for. Used to search other tokens for relevant information.
Key (K)
What a token contains. Compared against Queries to produce attention scores.
Value (V)
The actual information a token sends when it is found relevant by a Query.
Attention Score
A relevance weight between two tokens. High score = strong relationship. Low score = weak relationship.
Quadratic Scaling
Attention work grows as the square of token count. 2x tokens = 4x work. Root cause of long-context cost.
KV Cache
Saved K and V vectors from Prefill. Avoids recomputation during Decode. Large and memory-hungry.
Flash Attention
Optimised attention algorithm that avoids writing the full score matrix to slow memory. Same result, much faster. (Full detail in Module 3.1.)
